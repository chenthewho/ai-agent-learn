# 第 17 章 部署与生产化

> 前面你把 Agent 的能力、安全都打磨好了。但"能在本地跑通"和"能在生产扛住真实流量"之间，还隔着一整套工程问题：部署成什么形态？长任务超时了怎么办？状态存哪？上游模型限流了如何退避？主模型挂了能不能切备用？怎么灰度上线、快速回滚？本章把 Agent 推上生产线，逐个回答这些问题。

> **学习目标**
> - 看懂两种部署形态——Serverless 函数 vs 长驻服务——以及 Agent 的长任务/流式/状态特性如何影响选型。
> - 掌握**长任务**在超时限制下的几种打法：后台任务队列、状态轮询/Webhook、断点续跑、流式保活。
> - 学会状态与会话持久化（存哪、怎么做到无状态扩缩容），以及限流、重试退避、降级、熔断、幂等等可靠性手段。
> - 会做多环境与灰度发布、模型/提示版本管理、快速回滚。
> - 能写出一个**带重试退避 + 超时 + 降级到备用模型**的健壮模型调用封装，并产出一份上线 checklist。

**前置知识**：第 12 章[流式输出与前端集成](./12-流式输出与前端集成.md)（SSE/流式机制）、第 13 章评测（发布门禁的依据）、第 14 章[可观测性与调试](./14-可观测性与调试.md)（监控、告警、回滚的眼睛）。本章会反复呼应错误处理：很多"可靠性"手段你在前端调接口时已经做过（重试、退避、降级），这里只是把它们系统化。

## 17.1 两种部署形态

把 Agent 部署到生产，第一个岔路口是：**跑在哪种运行时上？** 两大类。

**Serverless 函数**（Vercel Functions、Cloudflare Workers、AWS Lambda）：

- 你只写函数，平台负责拉起、扩缩、销毁。请求来了起一个实例，处理完就回收。
- 优点：**零运维、自动扩缩、按用量付费、冷启动外几乎不用管**。对前端工程师极友好——你写 Next.js API route 部署到 Vercel，本质就是 Serverless。
- 致命约束：**有执行时长上限**（几十秒到几分钟不等，平台各异），且默认**无状态**（实例之间不共享内存，请求结束状态就没了）。

**长驻服务**（容器 / Kubernetes / 一直跑着的 Node/Python 进程）：

- 进程常驻，自己管生命周期。
- 优点：**没有硬性的单请求时长限制**，能 hold 住长连接、长任务、内存里的状态。
- 代价：**要自己做运维**——扩缩容、健康检查、连接管理、滚动发布，都得你操心。

```
              Serverless 函数                长驻服务（容器/K8s）
            ┌──────────────────┐          ┌──────────────────┐
请求 ─────▶ │ 临时实例(几十秒~  │  请求 ──▶│ 常驻进程          │
            │  几分钟上限)      │          │ (无硬性时长上限)  │
            │ 处理完即回收      │          │ 可 hold 长连接/状态│
            │ 默认无状态        │          │ 但要自己运维扩缩  │
            └──────────────────┘          └──────────────────┘
            零运维、自动扩缩、按量          没超时、能长任务、
            付费；但有超时、无状态          但运维成本高
```

**Agent 的特性如何撞上这道选择题？** 三个特性让 Agent 比普通 CRUD 接口更难放进 Serverless：

1. **长任务**：Agent 循环可能跑很多轮（思考→调工具→再思考……），一个复杂任务跑几分钟甚至十几分钟很正常。这直接撞上 Serverless 的时长上限。
2. **流式**：Agent 通常要流式输出（边想边吐字），需要 hold 住一条长连接——Serverless 上要看平台是否支持，且仍受总时长约束。
3. **状态**：多轮会话需要历史和记忆，但 API 是无状态的（每次要带全量历史），而 Serverless 实例又不保存内存状态。

所以现实里常见的折中是：**短交互（单轮问答、轻量工具调用）放 Serverless，重交互（长任务、长流式）放长驻服务，或用"Serverless 接入 + 后台任务"的混合架构**（下一节细讲）。

> **前端类比**：Serverless ≈ 你熟悉的 Vercel API routes / 云函数——写完即部署、自动扩缩，但有超时。长驻服务 ≈ 自己用 Node 起一个一直跑的 server，能 hold WebSocket、能放内存缓存，但要自己上 PM2/容器去守护。选型逻辑和你做前端 BFF 时一样：**任务多长、要不要长连接、要不要状态**，三个问题问完答案就出来了。

## 17.2 长任务处理：在超时限制下活下来

这是 Agent 部署最棘手的工程点。一个会跑十几分钟的 Agent 任务，怎么在"单请求几十秒就超时"的环境里跑完？核心思路是**把"长任务"和"HTTP 请求"解耦**——别让用户的请求一直挂着等结果。

几种打法，可组合使用：

**① 后台任务队列（最通用）**

请求进来不直接跑 Agent，而是**创建一个任务、丢进队列、立刻返回一个 `task_id`**。真正的 Agent 执行交给后台 worker（长驻进程）异步跑。

```
用户请求 ──▶ 创建任务，入队，立刻返回 { task_id }   ← HTTP 秒回，不超时
                    │
              任务队列 (Redis/SQS/...)
                    │
            后台 worker (长驻) 慢慢跑 Agent ──▶ 跑完把结果写回 DB
```

**② 状态轮询 / Webhook（拿结果的两种方式）**

任务在后台跑，前端怎么知道好了没？

- **轮询（polling）**：前端拿着 `task_id` 每隔几秒查一次 `/tasks/{id}` 状态（`pending` / `running` / `done` / `failed`），`done` 了就取结果。简单、前端好实现。
- **Webhook / 推送**：任务完成时，后端主动回调一个 URL，或通过 WebSocket / SSE 推给前端。实时性好，但要前端 hold 连接或提供回调端点。

**③ 断点续跑（checkpointing）**

长任务跑到一半，进程被回收 / 崩溃 / 部署重启了怎么办？给任务做**检查点**：把中间状态（已完成的步骤、已有的对话历史、工具结果）持久化下来。任务可以从最近的检查点**续跑**，而不是从头再来。这对 Serverless 尤其重要——它随时可能被回收。

**④ 流式保活（streaming keep-alive）**

如果你确实要 hold 一条流式连接（比如 SSE 实时吐 Agent 的输出），要防连接因"长时间没数据"被中间层（代理、网关、浏览器）掐断。做法是**周期性发心跳**（SSE 注释行 `: heartbeat` 或一个空事件），让连接保持活跃。这也呼应第 12 章的流式内容。

> **一个实用经验**：流式有个隐藏好处——**它能避免 HTTP 超时**。很多 SDK 对"非流式、大 `max_tokens`"的请求会主动报错或设很短的超时，因为一个长时间不返回任何字节的空连接很容易被掐。Claude 的 SDK 就建议：**任何可能输入长、输出长或 `max_tokens` 很大的请求，默认走流式**——哪怕你不需要逐字渲染，也可以用 SDK 的 `.get_final_message()` / `.finalMessage()` 在流式之上拿到完整结果，白赚一层超时保护。

> **前端类比**：这套"请求秒回 task_id + 后台跑 + 轮询/推送拿结果"，就是你做大文件上传、视频转码、报表导出时的异步任务模式，一模一样。Agent 长任务只是又一个"不能让用户傻等同步返回"的场景。

## 17.3 状态与会话持久化

API 是**无状态的**（第 5 章讲过：每次调用都要带全量对话历史，模型不记得上一轮）。但你的 Agent 产品是**有状态的**（多轮会话、长期记忆）。这个落差要靠持久化来填，而且要填得能**无状态扩缩容**。

存什么、存哪：

- **会话历史（短期）**：当前对话的 messages。读写频繁、要快——常用 **Redis**（快、有 TTL 自动过期）或数据库。
- **长期记忆 / 用户画像**：跨会话保留的信息。放**数据库**（结构化）。
- **向量化的知识 / 记忆（RAG）**：放**向量库**（pgvector、Qdrant 等，见第 8 章），按用户/租户分区。

**关键原则：服务进程本身不存状态，状态全在外部存储里。** 这样任意一个实例都能处理任意一个请求——这就是"无状态服务"，是水平扩缩容的前提。

```
✗ 有状态：会话历史存在某个实例的内存里
   → 下次请求被路由到另一个实例，历史就丢了；无法扩缩容

✓ 无状态：实例不存状态，每次从 Redis/DB 取会话历史
   → 任意实例都能处理任意请求 → 随便加机器扩容
   ┌────────┐  ┌────────┐  ┌────────┐
   │实例 A  │  │实例 B  │  │实例 C  │  ← 都不存状态
   └───┬────┘  └───┬────┘  └───┬────┘
       └───────────┼───────────┘
              ┌────▼─────┐
              │ Redis/DB │  ← 状态在这
              └──────────┘
```

> **前端类比**：和你部署多实例 Node 服务时"别把 session 存内存，放 Redis"是同一条铁律。Serverless 实例之间不共享内存这件事，逼着你天然就得无状态——状态全外置，反而省心。

## 17.4 限流与配额：别被上游模型掐脖子

你的 Agent 调的是模型 API，而模型 API 有**速率限制**：每分钟请求数（RPM）、每分钟 token 数（TPM），甚至每天 token 数。流量一大就会撞上 `429 Too Many Requests`。生产化必须主动应对，不能等它报错才反应。

四个手段：

- **客户端限流**：在你这侧主动控制发出去的速率，别一股脑全打过去。用令牌桶 / 信号量控制并发，把请求速率压在配额线以下。
- **重试退避（指数退避）**：撞到 `429` 或 `5xx` 时，不要立刻重试（会火上浇油），而是**等一会儿再试，每次失败等待时间翻倍**（指数退避），并加随机抖动（jitter）避免一群客户端同时重试（惊群）。这正是错误处理里的经典套路。
- **排队**：超出配额的请求先入队，按速率慢慢消化，而不是直接失败。
- **降级**：配额吃紧时，把部分流量降级——切到限流更宽松的备用模型、或返回简化结果（下一节细讲）。

> **好消息**：主流厂商的官方 SDK **内置了重试退避**。Claude 的 SDK 默认会对 `429` 和 `5xx`（以及连接错误、408、409）做指数退避重试，重试次数由 `max_retries` 控制（默认 2 次），并且会读取 `429` 响应里的 `retry-after` 头来决定等多久。**所以多数情况下你不用自己手写退避**——配好 `max_retries` 就行。只有当你需要超出 SDK 默认的行为时（比如更多次数、自定义退避曲线、跨多个模型降级），才自己包一层。下一节的封装就是这种"需要降级到备用模型"的进阶场景。

> **前端类比**：调第三方 API 撞 429 要退避重试、撞限流要排队降级——你在前端对接支付、地图、短信这些有配额的 API 时都干过。区别只在于：Agent 调模型的频率和 token 量更大，限流来得更快更频繁，所以这套要从一开始就做扎实。

## 17.5 可靠性：超时、重试、熔断、降级、幂等

把可靠性手段整理成一组工具箱，按需取用：

- **超时（timeout）**：每个外部调用都要设超时，别让一个卡住的请求拖垮整个任务。注意各 SDK 的超时**单位不同**——有的秒、有的毫秒，配之前查清楚（Claude SDK 默认超时较长，大 `max_tokens` 时建议配合流式）。
- **重试（retry）**：瞬时错误（429、5xx、网络抖动）重试，**带指数退避**；但**业务错误（4xx，如参数错、鉴权失败）不重试**——重试也没用，只会浪费配额。
- **熔断（circuit breaker）**：当某个依赖（模型 API、某个工具）**持续失败**时，暂时"跳闸"——不再发请求，直接快速失败一段时间，给下游喘息，避免雪崩。过一会儿再试探性放行。
- **降级（fallback）**：**主模型挂了 / 限流了，自动切到备用模型。** 比如主用 `claude-opus-4-8`，挂了降到 `claude-sonnet-4-6` 或别的厂商。这是 Agent 可靠性里很关键的一招——保证"还能用"，哪怕质量略降。
- **幂等（idempotency）**：有副作用的操作（发邮件、扣款、写订单）要做幂等——用幂等键确保"重试导致的重复执行"不会真的执行两次。这在有重试的系统里是必须的。

> **降级的两种形态**：一种是你**自己在代码里**捕获错误后切换模型（下一节的封装就是这种，框架无关、通用）。另一种是**厂商提供的能力**——例如 Claude 支持在请求里传 `fallbacks` 参数，当最强模型被安全分类器拒绝（`refusal`）时由服务端自动改投备用模型（这点第 16 章碰过）。两者解决的问题略有不同（前者应对"挂了/限流"，后者应对"被拒绝"），可以叠加用。厂商特定的能力以官方文档为准。

> **前端类比**：超时、重试、熔断、降级，是你做微服务 / BFF 时的标准可靠性套路（很多人用过 axios 拦截器做重试、用熔断库防雪崩）。这里原样适用。**幂等**尤其别忘——有了重试就一定要有幂等，否则一封邮件可能发三遍。

## 17.6 多环境与发布

像发布前端应用一样发布 Agent，但要额外管几样东西。

**多环境**：开发（dev）→ 预发（staging）→ 生产（prod）。代码、配置、模型版本在各环境间隔离，先在预发验证再上生产。

**版本管理（Agent 特有）**：除了代码版本，你还要管两样前端没有的东西：

- **提示版本**：系统提示是 Agent 的"核心逻辑"，改一句话行为就可能大变。把提示当代码一样**纳入版本管理**，每次改动可追溯、可回滚。
- **模型版本**：你用的模型 ID 是配置项。换模型（甚至同系列升级）都可能改变行为和成本，要能受控切换、能回滚。**用确切的模型 ID 字符串**，别用会漂移的别名（具体可用的 ID 以官方文档为准）。

**灰度发布（canary）/ A/B**：新版本（新提示 / 新模型）先放给一小部分流量，观察指标（质量、延迟、成本、错误率）没问题再逐步放量。A/B 则是同时跑两个版本对比效果——这正是第 13 章评测的用武之地：**用评测集和线上指标做发布门禁**，达标才放量。

**快速回滚**：一旦新版本出问题（质量跌、成本飙、报错增），能**一键回到上一个已知良好的版本**。提示和模型版本都要支持秒级回滚——这依赖第 14 章的可观测性来**及时发现**问题。

```
dev ──▶ staging ──(评测门禁:第13章)──▶ canary(5%流量) ──监控指标──▶ 全量
                                          │                  │
                                       出问题 ◀──────────────┘
                                          ▼
                                    一键回滚到上个良好版本(提示+模型)
```

> **前端类比**：灰度、A/B、回滚你都熟——前端发版本本来就这么干。Agent 多出来的功课是：**提示和模型版本也要纳入这套发布流程**。一次"只改了系统提示一句话"的发布，可能比改代码影响还大，所以同样要灰度、要能回滚。

## 17.7 扩缩容与并发

流量涨了怎么扛？

- **水平扩缩容**：靠 17.3 的无状态设计——状态外置后，加机器就能扩容。Serverless 自动扩；长驻服务用 K8s HPA 之类按 CPU/QPS 自动加减实例。
- **并发控制**：单个实例能同时处理多少请求要有上限，避免内存/连接被打爆。Agent 的请求往往"慢且重"（长任务、长连接），并发模型和普通快接口不同——要为长任务预留资源。
- **连接管理**：流式 Agent 要 hold 大量长连接，注意连接数上限、空闲连接回收、复用 HTTP 客户端（别每次新建连接）。
- **成本随量增长**：这是 Agent 和普通服务最不一样的地方——**成本主要是 token 费用，随调用量线性增长，而且增长很快**。要监控成本、设**成本告警**，必要时用第 8 章的提示缓存、用更便宜的模型分流来压成本。把 token 成本当成一等公民来监控，别等月底账单吓一跳。

> **前端类比**：水平扩缩、并发上限、连接复用是后端老问题。唯一的新维度是**成本**——传统服务加机器是固定成本，Agent 每多一次调用都在烧 token，成本和流量强绑定。所以"成本监控 + 告警"在 Agent 里是运维的一等公民，地位等同于错误率和延迟。

## 17.8 上线 Checklist

把生产化要点收敛成一份可勾选的清单。上线前逐项过：

- [ ] **密钥管理**：所有密钥走环境变量 / 密钥管理服务，没有硬编码；密钥不进上下文、不进日志（第 16 章）。
- [ ] **限流**：对上游模型做了客户端限流 / 并发控制，配了 SDK 的 `max_retries`，撞 429 不会直接挂。
- [ ] **超时**：每个外部调用都有超时；大 `max_tokens` 走流式防超时。
- [ ] **重试与退避**：瞬时错误指数退避重试（多数靠 SDK 内置），业务错误（4xx）不重试。
- [ ] **降级**：主模型挂了 / 限流能切备用模型；有副作用操作做了幂等。
- [ ] **可观测性**：接入了日志 / 指标 / 链路追踪（第 14 章），能看到延迟、错误率、token 成本、工具调用；日志已脱敏。
- [ ] **评测门禁**：发布前过评测集（第 13 章），关键指标达标才放量。
- [ ] **成本告警**：监控 token 成本，设了预算告警，异常飙升会通知。
- [ ] **安全审查**：过了第 16 章的清单——提示注入防护、工具白名单、危险操作审批、租户隔离、`refusal` 处理。
- [ ] **回滚预案**：提示和模型版本可一键回滚到上个良好版本；发布走灰度。
- [ ] **状态持久化**：会话/记忆外置（Redis/DB/向量库），服务无状态可扩缩；长任务有断点续跑。

## 17.9 实战：健壮的模型调用封装

把本章几条核心可靠性手段（超时 + 重试退避 + 降级到备用模型）落成一段可运行的封装。这层薄抽象（呼应本书"通过一层 `chat()` 调用模型"的原则）让上层 Agent 代码不用关心底层的重试和降级。

设计要点：

- SDK 已经内置了 `429`/`5xx` 的指数退避重试（配 `max_retries`），所以**单模型的瞬时重试交给 SDK**。
- 我们这层只额外做一件 SDK 不做的事：**当主模型彻底失败时，降级到备用模型**——依次尝试模型列表，前一个挂了就试下一个。
- 配上超时；并区分"该降级的错误"（限流、过载、5xx、超时）和"不该降级的错误"（如参数错 400、鉴权 401——换模型也没用，直接抛）。

#### TypeScript

```typescript
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic(); // 从 ANTHROPIC_API_KEY 读取，SDK 默认 max_retries=2

// 主模型在前，备用模型在后——前一个失败就降级到下一个
const MODEL_CHAIN = ["claude-opus-4-8", "claude-sonnet-4-6"];

interface ChatOptions {
  system?: string;
  maxTokens?: number;
  timeoutMs?: number;
}

/** 健壮的模型调用：SDK 管单模型重试退避，这层管超时 + 降级到备用模型 */
async function robustChat(
  messages: Anthropic.MessageParam[],
  opts: ChatOptions = {},
): Promise<string> {
  const { system, maxTokens = 4096, timeoutMs = 60_000 } = opts;
  let lastErr: unknown;

  for (const model of MODEL_CHAIN) {
    try {
      // 流式 + finalMessage：大输出防超时，又能拿到完整结果（第 12 章）
      const stream = client.messages.stream(
        { model, max_tokens: maxTokens, system, messages },
        { timeout: timeoutMs }, // 注意 TS SDK 超时单位是毫秒
      );
      const msg = await stream.finalMessage();

      // 读 content 前先判停止原因（第 16 章）：拒绝时降级到下一个模型试试
      if (msg.stop_reason === "refusal") {
        lastErr = new Error(`模型 ${model} 拒绝了请求`);
        continue;
      }
      return msg.content.find((b) => b.type === "text")?.text ?? "";
    } catch (err) {
      lastErr = err;
      // 不该降级的错误（参数错、鉴权失败）直接抛——换模型也没用
      if (
        err instanceof Anthropic.BadRequestError ||
        err instanceof Anthropic.AuthenticationError
      ) {
        throw err;
      }
      // 限流 / 过载 / 5xx / 超时 / 网络错误：SDK 已在单模型内退避重试过仍失败，
      // 这里降级到下一个模型继续试
      console.warn(`模型 ${model} 调用失败，降级到下一个：`, err);
    }
  }

  // 所有模型都失败了
  throw new Error(`所有模型均不可用，最后错误：${String(lastErr)}`);
}
```

#### Python

```python
import anthropic

client = anthropic.Anthropic()  # 从 ANTHROPIC_API_KEY 读取，SDK 默认 max_retries=2

# 主模型在前，备用模型在后——前一个失败就降级到下一个
MODEL_CHAIN = ["claude-opus-4-8", "claude-sonnet-4-6"]

def robust_chat(messages, system=None, max_tokens=4096, timeout_s=60.0) -> str:
    """健壮的模型调用：SDK 管单模型重试退避，这层管超时 + 降级到备用模型。"""
    last_err = None

    for model in MODEL_CHAIN:
        try:
            # 流式 + get_final_message：大输出防超时，又能拿到完整结果（第 12 章）
            # Python SDK 超时单位是秒
            with client.with_options(timeout=timeout_s).messages.stream(
                model=model,
                max_tokens=max_tokens,
                system=system or anthropic.NOT_GIVEN,
                messages=messages,
            ) as stream:
                msg = stream.get_final_message()

            # 读 content 前先判停止原因（第 16 章）：拒绝时降级到下一个模型试试
            if msg.stop_reason == "refusal":
                last_err = RuntimeError(f"模型 {model} 拒绝了请求")
                continue
            return next((b.text for b in msg.content if b.type == "text"), "")

        except (anthropic.BadRequestError, anthropic.AuthenticationError):
            # 不该降级的错误（参数错、鉴权失败）直接抛——换模型也没用
            raise
        except (anthropic.RateLimitError, anthropic.APIStatusError,
                anthropic.APITimeoutError, anthropic.APIConnectionError) as err:
            # 限流 / 5xx / 超时 / 网络错误：SDK 已在单模型内退避重试过仍失败，
            # 这里降级到下一个模型继续试
            last_err = err
            print(f"模型 {model} 调用失败，降级到下一个：{err}")

    # 所有模型都失败了
    raise RuntimeError(f"所有模型均不可用，最后错误：{last_err}")
```

几个值得注意的设计：

- **重试退避不用自己写**——SDK 内置（`max_retries`）。这层封装只补 SDK 不管的"跨模型降级"。
- **错误要分类**：4xx 业务错误直接抛（重试/降级都没用），瞬时/可用性错误才降级。这呼应第 16 章错误处理里"该重试的才重试"。
- **流式 + `finalMessage()` / `get_final_message()`**：既防大输出超时，又能在不逐字渲染的场景下拿到完整结果。
- **降级也覆盖 `refusal`**：主模型拒绝时，试试备用模型（注意拒绝处理细节见第 16 章）。

上层 Agent 循环只管调 `robustChat(messages)`，超时、重试、降级全被这层吃掉了——这就是"薄抽象"的价值。

## 17.10 前端视角：像发版前端一样，但多管三样

本章你会发现：**Agent 的部署运维，骨架就是你熟悉的现代前端 / 全栈那一套。**

| 前端 / 全栈你熟悉的 | Agent 里基本照搬 | Agent 额外要管的 |
| --- | --- | --- |
| Vercel / 云函数部署、自动扩缩 | Serverless 形态完全适用 | 长任务撞超时 → 后台队列 + 轮询 |
| 多实例无状态、session 放 Redis | 会话/记忆外置、无状态扩缩 | 多了向量库这层记忆存储 |
| 调第三方 API 重试退避、降级 | 限流退避、熔断、降级照搬 | 限流来得更频繁、要能切**模型** |
| dev/staging/prod、灰度、回滚 | 发布流程一模一样 | 多管**提示版本 + 模型版本** |
| 监控延迟、错误率、告警 | 可观测性照搬（第 14 章） | 多管 **token 成本** 监控告警 |

把它说白：发布 Agent 和发布前端应用没本质区别——一样要多环境、灰度、回滚、监控告警。你只需要在脑子里加三个"额外要盯的东西"：**模型版本、token 成本、上游限流**。其余的工程直觉，原样能用。

## 17.11 常见坑 / 最佳实践

- **把长任务塞进 Serverless 同步等结果。** 几十秒一到就超时，任务白跑。长任务一律走"秒回 task_id + 后台执行 + 轮询/推送"。
- **状态存在实例内存里。** 多实例 / Serverless 一扩缩就丢。状态全外置（Redis/DB/向量库），服务保持无状态。
- **自己手写重试退避，却忘了 SDK 已经内置。** 多数情况配 `max_retries` 就够；自己写容易写错（漏 jitter、对 4xx 也重试）。
- **业务错误也重试。** 400/401 重试纯属浪费配额，还可能放大问题。只对瞬时/可用性错误退避重试。
- **没有降级预案。** 主模型一抖动，整个产品挂。备好备用模型链，挂了能切。
- **有重试没幂等。** 重试 + 有副作用操作 = 邮件发三遍、扣款扣两次。有重试就必须有幂等键。
- **改提示直接全量上线。** 系统提示改一句话行为可能大变。提示要纳入版本管理、走灰度、能回滚。
- **不监控 token 成本。** 成本随量线性涨，不设告警可能某天账单暴雷。把成本当一等指标盯。
- **超时单位搞错。** 各 SDK 秒 / 毫秒不一致，配错要么形同虚设要么过短。配前查文档。

## 17.12 本章小结

- **两种部署形态**：Serverless（零运维、自动扩缩，但有超时、无状态）vs 长驻服务（无时长限制、能 hold 状态/长连接，但要自己运维）。Agent 的长任务/流式/状态特性常逼你走混合架构。
- **长任务**靠"请求秒回 task_id + 后台队列执行 + 轮询/Webhook 拿结果 + 断点续跑 + 流式保活"在超时限制下活下来。
- **状态持久化**：会话历史/记忆/向量全外置（Redis/DB/向量库），服务保持无状态才能水平扩缩。
- **限流与可靠性**：客户端限流 + 指数退避重试（多靠 SDK 内置 `max_retries`）+ 排队 + 熔断 + 降级到备用模型 + 幂等。
- **发布**：多环境 + 评测门禁（第 13 章）+ 灰度/A/B + 快速回滚；提示和模型版本要像代码一样纳入版本管理。
- **扩缩容**靠无状态设计；并发、连接要为长任务/长连接预留；**token 成本随量线性增长，必须监控告警**。
- **健壮的模型调用封装**：超时 + 降级到备用模型，单模型重试退避交给 SDK；错误分类（4xx 抛、瞬时错降级）。
- **前端视角**：部署运维骨架照搬现代前端那一套，额外多盯三样——模型版本、token 成本、上游限流。

# 第 12 章 流式输出与前端集成

> 前面几篇你已经把 Agent 的"大脑"搭起来了：会循环、会调工具、会查 RAG。但用户面对的不是你的循环，而是一块屏幕。如果模型憋了 8 秒才一次性吐出 500 字，体验是灾难；如果它像 ChatGPT 那样逐字蹦出来、还能看到"正在调用搜索工具"，体验立刻不一样。**这一章是前端工程师的主场**——把后端那个会循环的 runtime，接到一个流畅、能打断、能展示中间步骤的前端 UI 上。这正是你相比算法/后端同学的天然优势。

> **学习目标**
> - 说清**为什么 Agent 应用几乎都要流式**：LLM 逐 token 生成，流式能把"首字延迟（TTFT）"从几秒降到几百毫秒。
> - 对比三种传输方式——**SSE、WebSocket、HTTP chunked**——讲清为什么 Agent 应用首选 SSE。
> - **后端**：用 Node/TS（Web Streams / Vercel AI SDK）和 Python（FastAPI `StreamingResponse`）各暴露一个流式接口，把 Claude / OpenAI 的流转成 SSE。
> - **前端**：用 `fetch` + `ReadableStream`、`EventSource`、以及 `@ai-sdk/react` 的 `useChat`，做一个带打字机效果、loading、错误、可中断的流式聊天 UI。
> - 设计一套**事件协议**（`text` / `tool_call` / `tool_result` / `thinking` / `done`），把 Agent 的"中间步骤"也流给前端——这是 Agent 产品体验的关键。
> - 处理**中断（AbortController）、重连、背压**这些工程细节。

**前置知识**：第 5 章[Agent 核心循环与推理范式](../02-核心能力篇/05-agent核心循环与推理范式.md)（你需要知道循环每一步会产出什么）。HTTP/fetch、Promise、React 组件、SSE/WebSocket 的基础概念（这些你本来就会）。

## 12.1 为什么要流式：把"首字延迟"打下来

先看一个数字感受。一个模型回答 600 个 token、生成速度约 50 token/s，那么：

- **不流式**：服务端等模型把 600 个 token 全生成完（约 12 秒），再一次性返回。用户盯着空白屏幕 12 秒，然后"啪"地出现一整段。
- **流式**：模型生成出第一个 token（约 0.3 秒）就立刻推给前端开始渲染。用户 0.3 秒就看到字开始往外蹦，剩下的逐字补齐。

总耗时几乎一样（模型生成速度没变），但**体感天差地别**。关键指标叫 **首字延迟（TTFT, Time To First Token）**——从发请求到屏幕上出现第一个字的时间。流式把 TTFT 从"整段生成时间"压到"第一个 token 的时间"。

> **前端类比：流式就是渐进式渲染。**
> 你早就懂这个道理了——
> - **图片渐进式加载**（progressive JPEG）：先出一张模糊的整图，再逐步变清晰，而不是从上到下一行行刷。
> - **SSR 流式 / React Suspense 流式渲染**：HTML 一块块吐给浏览器，先渲染骨架，数据回来再补内容，而不是等全部数据齐了才返回整页。
> - **骨架屏（skeleton）**：先给个占位，让用户知道"在加载了"，而不是白屏。
>
> 流式 LLM 输出和这些是**同一种思想**：**别让用户等"全部完成"，能展示多少先展示多少。** 你做前端时为了体验做的所有"渐进式"优化，在 Agent 应用里换个皮就能复用。

还有两个非流式扛不住的现实原因：

1. **长输出会撞 HTTP 超时。** 模型开大 `max_tokens`（比如 64K）做长篇生成时，一个非流式请求可能跑好几分钟，中间连接很容易被网关/代理判定为空闲而掐断。流式因为持续有数据流动，连接一直"活着"。各家 SDK 对大 `max_tokens` 甚至**强制要求流式**正是这个原因。
2. **Agent 要展示"过程"。** 一个多步 Agent 跑十几秒很正常。如果这十几秒里前端什么都不显示，用户会以为卡死了。流式让你能实时推送"正在思考""正在调用 XX 工具""查到了 3 条资料"——这对 Agent 产品几乎是刚需（12.5 节详谈）。

## 12.2 三种传输方式：SSE vs WebSocket vs HTTP chunked

"把数据一点点推给浏览器"有三条路。先看一张对比表，再讲为什么 Agent 应用多用 SSE。

| 维度 | **SSE**（Server-Sent Events） | **WebSocket** | **HTTP chunked**（裸分块） |
| --- | --- | --- | --- |
| 方向 | 单向：服务端 → 客户端 | 双向：全双工 | 单向：服务端 → 客户端 |
| 底层 | 普通 HTTP（`Content-Type: text/event-stream`） | 独立协议（`ws://`/`wss://`，需升级握手） | 普通 HTTP（`Transfer-Encoding: chunked`） |
| 浏览器 API | `EventSource`（或 `fetch` + 流） | `WebSocket` | `fetch` + `ReadableStream` |
| 自动重连 | ✅ `EventSource` 内置 | ❌ 自己实现 | ❌ 自己实现 |
| 消息分帧 | ✅ 内置（`data:` / `event:` 行格式） | 需自己定消息格式 | ❌ 纯字节流，自己切 |
| 经过普通 HTTP 基建 | ✅ 走 80/443，CDN/网关友好 | ⚠️ 需基建支持 ws 升级 | ✅ |
| 鉴权 | 跟普通请求一样（Header/Cookie） | 握手阶段稍麻烦 | 跟普通请求一样 |

**为什么 Agent 应用首选 SSE？**

LLM 聊天的数据流向有个鲜明特点：**绝大多数时候是"服务端单向往客户端推"**——模型生成的 token、工具调用事件、思考过程，全是从服务端流向前端。客户端要做的"发消息"是一次性的普通 POST 请求，并不需要在同一条连接上持续地双向通信。

这正好命中 SSE 的设计：

- **单向推送**正是 SSE 的本职，不需要 WebSocket 的全双工开销。
- **基于普通 HTTP**：复用你已有的鉴权（Cookie/Bearer）、CDN、反向代理、Serverless 平台——不用为 WebSocket 单独搞一套基建。这点对部署在 Vercel/Cloudflare 这类平台的全栈应用尤其香。
- **内置分帧和重连**：SSE 的 `data:` 行格式天然就是"一条条事件"，`EventSource` 还自带断线重连。chunked 是裸字节流，这些都得自己造。

什么时候才需要 WebSocket？**真正需要低延迟双向通信**时——比如语音实时对话（音频流双向）、协同编辑、需要服务端和客户端高频互相打断的实时游戏。普通的"发一句、流式收一段"的 Agent 聊天，SSE 足够且更省心。

> **一句话决策**：先用 SSE。等你确实遇到"需要在同一条长连接上双向高频通信"（典型是实时语音）再上 WebSocket。本章后面的代码都以 SSE 为主。

### SSE 的报文长什么样

SSE 就是一段 `text/event-stream` 响应，body 是一行行文本，事件之间用空行分隔：

```
event: text
data: {"delta":"你"}

event: text
data: {"delta":"好"}

event: tool_call
data: {"name":"search","input":{"query":"杭州天气"}}

event: done
data: {"stopReason":"end_turn"}

```

规则就三条：
- `event:` 行声明事件类型（可省略，省略则前端按默认 `message` 事件处理）。
- `data:` 行是数据（通常塞一段 JSON 字符串）。
- **空行**表示"这条事件结束了"。

记住这个格式，下面后端要做的事就是：**把模型 SDK 吐出来的流，翻译成这种一行行的文本。**

## 12.3 后端：暴露一个流式接口

后端的核心任务：**调模型的流式接口 → 拿到一个个增量事件 → 转写成 SSE 推给前端**。我们先用最朴素的方式手写一遍（讲透原理），再展示框架（Vercel AI SDK）怎么替你省掉样板。

> 下面用 Anthropic Claude 的流式 API 演示，因为流式事件的结构跟具体厂商相关，讲透更有价值。换 OpenAI 时把"取 `content_block_delta` 里的 `delta.text`"换成"取 chunk 的 `choices[0].delta.content`"，骨架不变。模型 ID（`claude-opus-4-8`）属易变信息，以[资源清单](../06-附录/03-资源与工具清单.md)为准。

### 12.3.1 先认识模型 SDK 的流

Claude 的流式响应是一串**事件**，关键的几种：

| 事件 | 含义 |
| --- | --- |
| `message_start` | 整条消息开始，带元信息 |
| `content_block_start` | 一个内容块开始（可能是 `text`、`thinking`、`tool_use`） |
| `content_block_delta` | 内容块的增量：`delta.text`（正文）/ `delta.thinking`（思考）/ `delta.partial_json`（工具入参的 JSON 片段） |
| `content_block_stop` | 一个内容块结束 |
| `message_delta` | 消息级更新，含 `stop_reason`、`usage`（token 数） |
| `message_stop` | 整条消息结束 |

我们要做的就是：**遍历这些事件，把我们关心的（正文增量、思考增量、工具调用）转成自定义的 SSE 事件。**

### 12.3.2 Node/TS：用 Web Streams 手写 SSE

现代 Node（18+）和所有边缘运行时都原生支持 **Web Streams**（`ReadableStream`）和 Web 标准的 `Response`。我们用一个 `ReadableStream` 当 SSE 的 body，把模型流写进去。

#### TypeScript

```typescript
// route.ts —— 以 Web 标准 Request/Response 写一个流式接口
// 适用于 Next.js Route Handler、Hono、或任何 Web 标准的运行时
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic(); // 从 ANTHROPIC_API_KEY 读取密钥，绝不硬编码

// 小工具：把一条事件编码成 SSE 的文本格式（event: ... / data: ... / 空行）
function sse(event: string, data: unknown): string {
  return `event: ${event}\ndata: ${JSON.stringify(data)}\n\n`;
}

export async function POST(req: Request): Promise<Response> {
  const { messages } = await req.json();
  const encoder = new TextEncoder();

  const stream = new ReadableStream({
    async start(controller) {
      // 把一条 SSE 事件推给前端
      const send = (event: string, data: unknown) =>
        controller.enqueue(encoder.encode(sse(event, data)));

      try {
        // 1. 开 Claude 的流式请求。大 max_tokens 时流式还能避免 HTTP 超时
        const modelStream = client.messages.stream({
          model: "claude-opus-4-8",
          max_tokens: 4096,
          messages,
        });

        // 2. 遍历模型事件，挑我们关心的转成 SSE
        for await (const ev of modelStream) {
          if (
            ev.type === "content_block_delta" &&
            ev.delta.type === "text_delta"
          ) {
            // 正文增量：逐 token 推给前端，这是打字机效果的来源
            send("text", { delta: ev.delta.text });
          } else if (ev.type === "message_delta" && ev.delta.stop_reason) {
            // 消息结束：把 stop_reason 告诉前端
            send("done", { stopReason: ev.delta.stop_reason });
          }
        }
      } catch (err) {
        // 3. 出错也要走流：推一条 error 事件，前端才能优雅提示
        send("error", { message: (err as Error).message });
      } finally {
        controller.close(); // 关闭流，前端的 reader 会收到结束信号
      }
    },
  });

  return new Response(stream, {
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache", // 别让中间层缓存流
      Connection: "keep-alive",
    },
  });
}
```

几个关键点：
- **`Content-Type: text/event-stream`** 是 SSE 的身份证，少了它前端的 `EventSource` 不认。
- **`Cache-Control: no-cache`** 防止 CDN/代理把流缓存起来或缓冲后一次性吐出（缓冲会让流式失效）。
- **错误也走流**：模型调用半路失败时，不要让连接默默断掉，而是推一条 `error` 事件——前端才能区分"正常结束"和"出错了"。

### 12.3.3 Python：用 FastAPI 的 StreamingResponse

Python 侧用 FastAPI 的 `StreamingResponse`——给它一个**异步生成器**，`yield` 出的每段文本都会即时刷给客户端。

#### Python

```python
# main.py —— FastAPI 暴露一个 SSE 流式接口
import json
import os
from anthropic import AsyncAnthropic
from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse

app = FastAPI()
client = AsyncAnthropic()  # 从 ANTHROPIC_API_KEY 读取密钥，绝不硬编码


def sse(event: str, data: dict) -> str:
    """把一条事件编码成 SSE 文本格式：event: ... / data: ... / 空行"""
    return f"event: {event}\ndata: {json.dumps(data, ensure_ascii=False)}\n\n"


@app.post("/api/chat")
async def chat(req: Request):
    body = await req.json()
    messages = body["messages"]

    async def event_stream():
        try:
            # 1. 开 Claude 的流式请求
            async with client.messages.stream(
                model="claude-opus-4-8",
                max_tokens=4096,
                messages=messages,
            ) as stream:
                # 2. 遍历事件，挑正文增量转成 SSE
                async for event in stream:
                    if (
                        event.type == "content_block_delta"
                        and event.delta.type == "text_delta"
                    ):
                        yield sse("text", {"delta": event.delta.text})
                    elif event.type == "message_delta" and event.delta.stop_reason:
                        yield sse("done", {"stopReason": event.delta.stop_reason})
        except Exception as e:
            # 3. 出错也走流
            yield sse("error", {"message": str(e)})

    return StreamingResponse(
        event_stream(),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"},
        # X-Accel-Buffering: no —— 告诉 Nginx 别缓冲这个响应，否则流式会变成"攒一波再发"
    )
```

> **部署坑预警**：Nginx 默认会缓冲响应（`proxy_buffering on`），这会把你的流"攒成一坨"再发，流式直接失效。加 `X-Accel-Buffering: no` 响应头（上面 Python 已加），或在 Nginx 配置里对该路由关掉 `proxy_buffering`。Node 侧同理，确认中间层没有缓冲。

### 12.3.4 用框架：Vercel AI SDK 替你省掉样板

手写一遍是为了讲透原理。真到项目里，**Vercel AI SDK** 能把上面这堆样板压成几行——它替你做了"调模型流 → 转成标准协议 → 设置正确的响应头"这一整套，而且**前后端用同一套协议**，前端那边能直接用配套的 React hook 接住（12.4.3 节）。

#### TypeScript

```typescript
// route.ts —— 用 Vercel AI SDK，几行搞定流式接口
import { anthropic } from "@ai-sdk/anthropic";
import { streamText, convertToModelMessages } from "ai";

export async function POST(req: Request): Promise<Response> {
  const { messages } = await req.json();

  // streamText 开流式请求；toUIMessageStreamResponse() 直接返回一个
  // 带正确 SSE 头、且前端 useChat 能识别的 Response
  const result = streamText({
    model: anthropic("claude-opus-4-8"),
    messages: convertToModelMessages(messages),
  });

  return result.toUIMessageStreamResponse();
}
```

> 框架替你做了什么，心里要清楚：① 调用模型的流式接口；② 把模型事件转成它自己定义的一套流式协议（含文本、工具调用、结束等）；③ 设好 SSE 响应头。代价是你绑定了它的协议格式——好在它前端有配套 hook，端到端打通。**先理解 12.3.2 的手写版，再用框架，你才知道框架在替你做什么、出问题往哪查。** Vercel AI SDK 的 API 名（如 `toUIMessageStreamResponse`、`convertToModelMessages`）迭代较快，以其官方文档为准。

## 12.4 前端：消费流式接口

后端的流出来了，前端怎么接？三条路，由低到高：① `fetch` + `ReadableStream`（最底层、最可控）；② `EventSource`（最省事，但有限制）；③ `useChat` hook（最快出活）。

### 12.4.1 用 fetch + ReadableStream(最通用)

为什么不直接用 `EventSource`？因为 **`EventSource` 只能发 GET 请求、不能自定义请求体和大部分 Header**。而聊天要 POST 一大段 `messages`、要带鉴权 Header、要能 `AbortController` 中断——这些 `EventSource` 都做不好。所以**生产里更常用 `fetch` + 手动读流**。

`fetch` 的 `response.body` 就是一个 `ReadableStream`，我们手动读它、按 SSE 格式切分。

#### TypeScript

```typescript
// streamChat.ts —— 用 fetch 消费 SSE 流，逐事件回调
// onEvent 在每解析出一条 SSE 事件时被调用
export async function streamChat(
  messages: unknown[],
  onEvent: (event: string, data: any) => void,
  signal?: AbortSignal, // 传入 AbortController 的 signal 以支持中断
): Promise<void> {
  const res = await fetch("/api/chat", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ messages }),
    signal,
  });
  if (!res.ok || !res.body) throw new Error(`HTTP ${res.status}`);

  // response.body 是 ReadableStream<Uint8Array>；用 TextDecoderStream 转成文本流
  const reader = res.body.pipeThrough(new TextDecoderStream()).getReader();
  let buffer = "";

  while (true) {
    const { value, done } = await reader.read();
    if (done) break;
    buffer += value;

    // SSE 以空行分隔事件；按 \n\n 切出一条条完整事件
    const parts = buffer.split("\n\n");
    buffer = parts.pop() ?? ""; // 最后一段可能不完整，留到下次

    for (const part of parts) {
      if (!part.trim()) continue;
      // 解析 event: 和 data: 两行
      let event = "message";
      let data = "";
      for (const line of part.split("\n")) {
        if (line.startsWith("event:")) event = line.slice(6).trim();
        else if (line.startsWith("data:")) data += line.slice(5).trim();
      }
      if (data) onEvent(event, JSON.parse(data));
    }
  }
}
```

注意那个 **`buffer`**——网络分块和 SSE 事件边界**不对齐**：一次 `read()` 拿到的字节可能切在某条事件中间。所以我们攒进 buffer，只处理 `\n\n` 之前的完整事件，剩下的留到下一块。这是手动解析 SSE 最容易翻车的地方。

### 12.4.2 接进 React：打字机效果 + loading + 错误 + 中断

把上面的 `streamChat` 接进一个 React 组件。我们要的体验：发消息后 loading、模型回复逐字蹦（打字机）、出错给提示、能点"停止"中断。

#### TypeScript

```tsx
// Chat.tsx —— 一个带打字机、loading、错误、可中断的最小流式聊天 UI
import { useState, useRef } from "react";
import { streamChat } from "./streamChat";

type Msg = { role: "user" | "assistant"; content: string };

export function Chat() {
  const [messages, setMessages] = useState<Msg[]>([]);
  const [input, setInput] = useState("");
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);
  const abortRef = useRef<AbortController | null>(null); // 持有当前请求的中断器

  async function send() {
    if (!input.trim() || loading) return;
    setError(null);
    const userMsg: Msg = { role: "user", content: input };
    // 先把用户消息和一条"空的助手消息"放进去——助手那条会被流式逐字填充
    const history = [...messages, userMsg];
    setMessages([...history, { role: "assistant", content: "" }]);
    setInput("");
    setLoading(true);

    const controller = new AbortController();
    abortRef.current = controller;

    try {
      await streamChat(
        history,
        (event, data) => {
          if (event === "text") {
            // 收到正文增量：追加到最后一条（助手）消息的末尾 → 打字机效果
            setMessages((prev) => {
              const next = [...prev];
              next[next.length - 1] = {
                ...next[next.length - 1],
                content: next[next.length - 1].content + data.delta,
              };
              return next;
            });
          } else if (event === "error") {
            setError(data.message);
          }
          // "done" 事件：循环结束后自然走到 finally，这里不必特殊处理
        },
        controller.signal,
      );
    } catch (e) {
      // AbortError 是用户主动中断，不算错误
      if ((e as Error).name !== "AbortError") setError((e as Error).message);
    } finally {
      setLoading(false);
      abortRef.current = null;
    }
  }

  function stop() {
    abortRef.current?.abort(); // 触发 fetch 的 signal，请求被取消
  }

  return (
    <div>
      <div className="messages">
        {messages.map((m, i) => (
          <div key={i} className={m.role}>
            {m.content}
            {/* 最后一条助手消息且仍在 loading 时，显示一个闪烁光标 */}
            {loading && i === messages.length - 1 && m.role === "assistant" && (
              <span className="cursor">▍</span>
            )}
          </div>
        ))}
      </div>
      {error && <div className="error">出错了：{error}</div>}
      <div className="composer">
        <input
          value={input}
          onChange={(e) => setInput(e.target.value)}
          onKeyDown={(e) => e.key === "Enter" && send()}
          disabled={loading}
          placeholder="说点什么…"
        />
        {loading ? (
          <button onClick={stop}>停止</button>
        ) : (
          <button onClick={send}>发送</button>
        )}
      </div>
    </div>
  );
}
```

这就是流式聊天 UI 的全部骨架，每一块都是你熟悉的 React 套路：
- **打字机效果** = 每收到一个 `text` 增量，就把它追加到最后那条助手消息的 `content`，React 重渲染就是逐字长出来。
- **loading / 光标** = 一个 `loading` 状态，控制按钮文案和那个闪烁的 `▍`。
- **错误** = 收到 `error` 事件或 `catch` 到异常就 `setError`，渲染一条错误提示。
- **中断** = `AbortController`，"停止"按钮调 `abort()`，`fetch` 立刻取消（12.6 节详谈）。

### 12.4.3 用 useChat：连样板都省了

如果后端用了 Vercel AI SDK（12.3.4），前端用配套的 `@ai-sdk/react` 的 `useChat`，上面那一整个组件的状态管理（messages、loading、流式拼接、中断）几乎都不用自己写：

#### TypeScript

```tsx
// Chat.tsx —— 用 useChat，状态管理全包了
import { useChat } from "@ai-sdk/react";

export function Chat() {
  // messages：消息列表（含流式中的助手消息，自动逐字更新）
  // sendMessage：发消息；status：状态；stop：中断；error：错误
  const { messages, sendMessage, status, stop, error } = useChat();
  const isStreaming = status === "streaming" || status === "submitted";

  return (
    <div>
      {messages.map((m) => (
        <div key={m.id} className={m.role}>
          {/* 消息内容由若干 part 组成，文本 part 直接渲染 */}
          {m.parts.map((p, i) => (p.type === "text" ? <span key={i}>{p.text}</span> : null))}
        </div>
      ))}
      {error && <div className="error">出错了：{error.message}</div>}
      {isStreaming && <button onClick={stop}>停止</button>}
      <ChatInput onSubmit={(text) => sendMessage({ text })} disabled={isStreaming} />
    </div>
  );
}
```

> `useChat` 把"维护 messages、调流式接口、逐字拼接、loading/error 状态、中断"全包了——本质上就是把 12.4.1 + 12.4.2 那一坨封装成了一个 hook。**理解了手写版，你才知道这个 hook 在替你做什么、它的每个返回值对应自己手写时的哪段逻辑。** `useChat` 的具体 API（返回字段名、`messages` 的结构）随版本演进，以 Vercel AI SDK 官方文档为准。

## 12.5 流式不只是流文本：把 Agent 的"中间步骤"也流出去

到这里你做的还只是"流式聊天"。但**真正的 Agent 产品，流式的价值远不止逐字蹦文本**——它要把 Agent 在那十几秒里**干了什么**实时展示出来：现在在思考、调用了搜索工具、查到了 3 条资料、又调用了计算工具……

回想第 5 章的 Agent 循环：每一轮的产出有好几种——模型的**思考**、发起的**工具调用**、工具返回的**结果**、最终的**文本**。这些都应该流给前端。这正是 ChatGPT、Claude、各种 Agent 产品里你看到的"正在搜索网页…""正在分析…"那种体验的来源。

> **前端类比**：这就像你给一个耗时操作做的**进度反馈**——文件上传的进度条、CI 流水线里一个个步骤的实时状态、构建工具滚动的日志。用户不想盯着转圈,想知道"现在到哪一步了"。Agent 的中间步骤就是它的"进度条"。

### 12.5.1 设计一套事件协议

要流多种东西，就得给 SSE 事件分类型。一套朴素但够用的协议：

| 事件类型（`event:`） | `data` 里放什么 | 前端怎么渲染 |
| --- | --- | --- |
| `text` | `{ delta }` 正文增量 | 追加到当前助手气泡（打字机） |
| `thinking` | `{ delta }` 思考增量 | 折叠区里展示"思考过程"（可选，灰色小字） |
| `tool_call` | `{ id, name, input }` 工具名+入参 | 展示一张"正在调用 XX 工具"的卡片 |
| `tool_result` | `{ id, output }` 工具结果 | 把对应卡片标记为"完成"，可展开看结果/引用 |
| `done` | `{ stopReason, usage }` 结束信息 | 收尾：停光标、显示 token 数 |
| `error` | `{ message }` 错误 | 错误提示 |

**为什么自定义协议、而不是直接把模型 SDK 的原始事件透传给前端？**
- **解耦**：前端不该知道你后端用的是 Claude 还是 OpenAI。你定一套自己的协议，换厂商时只改后端的"翻译层"，前端不动。
- **精简**：模型 SDK 的原始事件又多又碎（`message_start`、`content_block_start`…），前端只关心那几种业务语义。后端做一次翻译，前端清爽很多。
- **可扩展**：你可以塞自己的事件，比如 `citation`（引用来源，呼应第 8 章 RAG）、`progress`（自定义进度），前端按需渲染。

### 12.5.2 后端：把循环里每一步翻译成事件

把第 5 章的 Agent 循环和 12.3 的 SSE 接上——循环里每发生一件事，就推一条对应的 SSE 事件。下面用 Python 演示一个带工具的 Agent 循环的流式版（TS 同理，把 `yield` 换成 `controller.enqueue`）。

#### Python

```python
# agent_stream.py —— 把带工具的 Agent 循环流式化，推出多种事件
import json
from anthropic import AsyncAnthropic

client = AsyncAnthropic()


def sse(event: str, data: dict) -> str:
    return f"event: {event}\ndata: {json.dumps(data, ensure_ascii=False)}\n\n"


async def agent_event_stream(messages: list, tools: list, tool_impls: dict):
    """一个会调用工具的 Agent 循环，把每一步以 SSE 事件 yield 出去。
    messages: 对话历史；tools: 工具定义；tool_impls: {工具名: 执行函数}
    """
    max_steps = 10  # 循环必须有刹车（见第 5 章）
    for _ in range(max_steps):
        # 流式开一轮模型调用，用 thinking 开启自适应思考（新模型用 adaptive，不要用 budget_tokens）
        async with client.messages.stream(
            model="claude-opus-4-8",
            max_tokens=4096,
            thinking={"type": "adaptive", "display": "summarized"},
            tools=tools,
            messages=messages,
        ) as stream:
            async for event in stream:
                if event.type == "content_block_delta":
                    d = event.delta
                    if d.type == "text_delta":
                        yield sse("text", {"delta": d.text})       # 正文增量
                    elif d.type == "thinking_delta":
                        yield sse("thinking", {"delta": d.thinking})  # 思考增量

            # 一轮流式结束，取完整消息，看模型这轮的决策
            final = await stream.get_final_message()

        # 把这一轮的 assistant 回复追加进历史（第 5 章强调过：漏了模型会"失忆"）
        messages.append({"role": "assistant", "content": final.content})

        # 模型不再调工具 → 结束
        if final.stop_reason != "tool_use":
            yield sse("done", {"stopReason": final.stop_reason})
            return

        # 模型要调工具：逐个执行，并把"调用"和"结果"都推给前端
        tool_results = []
        for block in final.content:
            if block.type == "tool_use":
                # 先推一条 tool_call：前端立刻显示"正在调用 XX 工具"
                yield sse("tool_call", {"id": block.id, "name": block.name, "input": block.input})
                # 执行工具（你的业务代码）
                output = await tool_impls[block.name](block.input)
                # 再推一条 tool_result：前端把卡片标记为完成
                yield sse("tool_result", {"id": block.id, "output": output})
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": str(output),
                })

        # 工具结果作为一条 user 消息回填，进入下一轮循环
        messages.append({"role": "user", "content": tool_results})

    # 兜底：撞到最大步数
    yield sse("error", {"message": "达到最大步数上限"})
```

注意这段代码就是**第 5 章的 Agent 循环 + 在每个关键节点插一条 SSE 推送**。循环骨架没变，只是把"原本默默发生的事"变成了"实时播报给前端"。`thinking` 用 `adaptive` 自适应思考、并设 `display: "summarized"` 才能拿到可读的思考摘要（不设的话新模型默认不返回思考文本）。

### 12.5.3 前端：分类型渲染中间步骤

前端的 `onEvent` 回调按事件类型分发，渲染成不同的 UI。核心是：**用一个数组维护"消息流里的条目"，每条可以是文本、工具卡片、思考块**。

#### TypeScript

```tsx
// 前端：把多类型事件渲染成一条条 UI 条目
type Item =
  | { kind: "text"; text: string }
  | { kind: "thinking"; text: string }
  | { kind: "tool"; id: string; name: string; input: unknown; status: "running" | "done"; output?: unknown };

// 在 streamChat 的 onEvent 回调里更新 items
function handleEvent(setItems: React.Dispatch<React.SetStateAction<Item[]>>) {
  return (event: string, data: any) => {
    setItems((prev) => {
      const next = [...prev];
      const last = next[next.length - 1];
      if (event === "text") {
        // 追加到当前文本条目，没有就新建一条
        if (last?.kind === "text") last.text += data.delta;
        else next.push({ kind: "text", text: data.delta });
      } else if (event === "thinking") {
        if (last?.kind === "thinking") last.text += data.delta;
        else next.push({ kind: "thinking", text: data.delta });
      } else if (event === "tool_call") {
        // 新建一张工具卡片，状态"运行中"
        next.push({ kind: "tool", id: data.id, name: data.name, input: data.input, status: "running" });
      } else if (event === "tool_result") {
        // 找到对应 id 的卡片，标记完成、填入结果
        const card = next.find((it) => it.kind === "tool" && it.id === data.id) as Extract<Item, { kind: "tool" }>;
        if (card) { card.status = "done"; card.output = data.output; }
      }
      return next;
    });
  };
}

// 渲染：按 kind 分发到不同组件
function ItemView({ item }: { item: Item }) {
  switch (item.kind) {
    case "text":
      return <div className="bubble">{item.text}</div>;
    case "thinking":
      return <details className="thinking"><summary>思考过程</summary>{item.text}</details>;
    case "tool":
      return (
        <div className={`tool-card ${item.status}`}>
          {item.status === "running" ? "🔧 正在调用" : "✅ 已调用"} <b>{item.name}</b>
          {item.status === "done" && <pre>{JSON.stringify(item.output, null, 2)}</pre>}
        </div>
      );
  }
}
```

这下你的 Agent UI 就"活"了：用户能看到模型在思考、在调用工具、查到了什么，而不是干等一团黑盒。**这种"展示过程"的流式 Agent UI，正是前端工程师能做出差异化体验的地方**——后端把事件流出来，怎么把它渲染得清晰、好看、有反馈，是你的看家本领。

## 12.6 中断、重连与背压

流式接口上线后会碰到三类工程问题：用户中途想停、连接断了要不要重连、前端处理不过来。

### 12.6.1 中断（取消）：AbortController

用户点"停止"、或在模型还没说完就发了新消息——都需要**取消正在进行的流式请求**。前端用的就是标准的 `AbortController`（12.4.2 已用过）：

```typescript
const controller = new AbortController();
fetch("/api/chat", { signal: controller.signal, /* ... */ });
// 想中断时：
controller.abort(); // fetch 立刻 reject，抛出 AbortError
```

要点：
- **区分"中断"和"出错"**：`abort()` 会让 `fetch` 抛 `AbortError`，这是用户主动行为，不该当错误弹提示。判断 `err.name === "AbortError"` 把它放过（12.4.2 的 `catch` 里就这么做的）。
- **服务端要响应取消**：前端 `abort` 后连接断开，服务端应该**停止继续调模型**——否则模型还在后台生成、白烧 token。在 Node Web 标准里，`Request` 带有 `req.signal`，连接断开时它会 abort，可以把它透传给模型 SDK 的 `signal` 参数；FastAPI 里可以用 `await request.is_disconnected()` 检测。这一步常被漏掉，导致"前端停了、后端还在烧钱"。

### 12.6.2 重连

连接中途断了（网络抖动、服务重启）怎么办？

- **`EventSource` 自带重连**：断线后会自动重连,还能用 `Last-Event-ID` 告诉服务端"我上次收到哪条了"。但前面说过 `EventSource` 不能 POST，聊天场景用得少。
- **`fetch` + 流要自己处理**：捕获到流中断的异常，决定要不要重试。但**聊天的"重连"很微妙**——模型生成是有状态的、不可幂等重放的,断在半句话上,简单重连会重复生成或丢上下文。
- **实务上的折中**：大多数聊天 App 对"流式中断"的处理是**优雅降级**——把已经收到的部分留在屏幕上,提示用户"回复未完成,点击重试",由用户决定是否重新发起。真正做到"无缝断点续传"需要服务端给每条事件编号、缓存会话状态、重连时按 ID 续发,工程量大,只有少数产品做。**先做优雅降级,够用了再上续传。**

### 12.6.3 背压（backpressure）

**背压**：生产数据的速度 > 消费数据的速度，中间得有机制让生产方"慢下来"，否则数据在内存里越堆越多。

- **好消息**：你大概率不用操心。LLM 生成 token 的速度（几十 token/s）远慢于前端渲染和网络传输的速度——**消费方比生产方快得多，几乎不会堆积**。Web Streams（`ReadableStream`）本身也内置了背压机制：当下游来不及读，`controller.enqueue` 会受到节流。
- **真要注意的场景**：前端**每收到一个 token 就触发一次 React 重渲染**，高频更新可能让 UI 卡顿。优化办法是**攒一小批再渲染**——比如用 `requestAnimationFrame` 节流、或攒够几个字符/几十毫秒再 `setState` 一次。这不是传统意义的背压，而是"渲染节流"，但解决的是同类问题：消费侧（渲染）成了瓶颈。

> **前端类比**：背压就像你做过的**防抖/节流（debounce/throttle）**——输入框联想、滚动监听里你都干过"别每次都触发，攒一下再处理"。流式 UI 里对高频 token 做渲染节流，是同一个套路。

## 12.7 前端视角：流式 Agent UI 是前端的核心竞争力

这一章基本是你的舒适区，最后把这件事说透：**在 Agent 产品里,前端的价值正在被重新放大,而流式 Agent UI 是其中最硬的一块。**

为什么这么说：

- **模型能力在拉平,体验在分化。** 大家都能调到同样的模型。一个 Agent 产品好不好用,越来越取决于**交互体验**——流式跟手不跟手、中间步骤展示得清不清楚、能不能优雅地中断和重试、工具调用的可视化够不够直观。这些**全是前端的活**,而且是算法/后端同学普遍不擅长的。

- **你已有的技能直接迁移。** 这一章你应该已经发现了:流式 = 渐进式渲染、事件协议 = 你定过的接口契约、打字机 = 状态追加重渲染、中断 = AbortController、背压 = 节流防抖。**没有一个是全新概念**,全是你前端肌肉记忆的换皮应用。

- **"展示过程"是 Agent 体验的胜负手。** 一个 Agent 跑十几秒,期间是黑盒还是有清晰的"思考 → 调工具 → 出结果"的过程展示,用户的信任感和体验差距巨大。把 12.5 的事件协议渲染成清晰好看的 UI,这种能力很值钱。

- **全栈打通是你的护城河。** 后端流式接口 + 前端流式 UI + 中间的协议设计,端到端打通——这种"一个人能把流式 Agent 从模型接到屏幕"的全栈能力,正是很多团队稀缺的。

一句话:**别把自己框在"切图仔"里。在 Agent 时代,能把一个会思考、会调工具的 runtime,流畅地、有反馈地呈现给用户——这是前端工程师最有价值的新战场。** 第 8 章的 RAG 答案要流式吐、第 5 章的 Agent 循环每步要流式播报,而你,正是把这些"流"接到用户眼前的人。

## 常见坑 / 最佳实践

- **忘了设 `Content-Type: text/event-stream`。** SSE 的身份证,缺了前端 `EventSource` 不认、`fetch` 解析也会乱。
- **被中间层缓冲掉了。** Nginx 默认 `proxy_buffering on`、CDN 缓存、Serverless 平台的响应缓冲,都会把你的流"攒成一坨"再发,流式直接失效。加 `X-Accel-Buffering: no`、`Cache-Control: no-cache`,并确认网关没缓冲。**这是流式上线最高频的"为什么没效果"。**
- **手动解析 SSE 没处理半条事件。** 网络分块边界和 SSE 事件边界不对齐,必须用 buffer 攒到 `\n\n` 再切,否则会把半条 JSON 喂给 `JSON.parse` 报错。
- **错误不走流、直接断连接。** 模型调用半路失败时,要推一条 `error` 事件再关流,前端才能区分"正常结束"和"出错",而不是莫名其妙断掉。
- **前端把"中断(AbortError)"当成错误弹窗。** 用户主动停止不是 bug,判断 `err.name === "AbortError"` 放过它。
- **前端 abort 了,后端还在烧钱。** 连接断了要让服务端停止继续调模型(透传 `req.signal` / 检测 `is_disconnected`),否则模型在后台继续生成、白烧 token。
- **每个 token 都 setState,UI 卡顿。** 高频重渲染是性能坑,对 token 做渲染节流(攒一批 / `requestAnimationFrame`)。
- **直接把模型 SDK 原始事件透传给前端。** 前端被绑死在某个厂商的事件格式上,换厂商就得改前端。定一套自己的事件协议,后端做翻译层。
- **新模型还手写 ReAct/思考解析。** 流式里拿思考用 `thinking: {type: "adaptive", display: "summarized"}`,别自己去 parse 模型吐的 "Thought:" 文本(呼应第 5 章)。

## 本章小结

1. **流式的本质是渐进式渲染**:别等"全部完成",能展示多少先展示多少。核心收益是把首字延迟(TTFT)从"整段生成时间"压到"第一个 token 的时间",外加避免长输出撞 HTTP 超时。
2. **Agent 应用首选 SSE**:数据流向是服务端单向推送、走普通 HTTP、内置分帧和重连——正好命中 SSE。需要双向高频通信(如实时语音)才上 WebSocket。
3. **后端的活是"翻译"**:把模型 SDK 的流式事件转成一行行 SSE 文本。手写用 Web Streams(TS)/ `StreamingResponse`(Python);框架(Vercel AI SDK)能省掉样板,但要先懂手写版。
4. **前端三条路**:`fetch` + `ReadableStream`(最通用、最可控)、`EventSource`(省事但不能 POST)、`useChat`(最快出活)。打字机 = 增量追加重渲染。
5. **流式不止流文本**:把 Agent 的思考、工具调用、工具结果都流出去,设计一套事件协议(`text`/`thinking`/`tool_call`/`tool_result`/`done`),前端分类型渲染——这是 Agent 产品体验的关键。
6. **工程细节**:中断用 `AbortController`(并让服务端响应取消)、重连优先做优雅降级、背压基本不用愁(消费快于生产),真要管的是前端渲染节流。
7. **流式 Agent UI 是前端的核心竞争力**:模型能力在拉平、体验在分化,而"把会思考会调工具的 runtime 流畅地呈现给用户"全是前端的活,且能力直接从你已有技能迁移。

## 练习题

1. **(基础)** 把 12.3 的后端(TS 或 Python 任选)和 12.4.2 的 React 组件跑通,实现一个能逐字蹦、能点"停止"中断的最小流式聊天。故意把后端的 `Content-Type` 改错,观察前端表现,理解这个头的作用。

2. **(基础)** 在 12.4.2 的组件里加"渲染节流":不要每个 `text` 增量都 `setState`,而是攒够 ~30ms 或几个字符再更新一次。对比节流前后,在快速生成时 UI 是否更顺。

3. **(进阶)** 把 12.5 的事件协议落地:后端按 12.5.2 流出 `tool_call`/`tool_result`,前端按 12.5.3 渲染成工具卡片(运行中 → 完成)。给 Agent 配一个简单工具(如查时间/算数),看 UI 上工具卡片如何实时变化。

4. **(进阶)** 实现"中断后服务端停止调模型":前端 `abort` 时,让后端检测到连接断开(Node 透传 `req.signal` 给模型 SDK 的 `signal`,或 FastAPI 轮询 `await request.is_disconnected()`)并停止流式生成。用日志验证 abort 后模型确实停了。

5. **(挑战)** 实现一个最小的"优雅降级 + 重试":流式中途断开时(可手动 kill 后端模拟),前端把已收到的部分保留在屏幕上,显示"回复未完成,点击重试"按钮,点击后用当前历史重新发起请求。思考:为什么聊天场景做"无缝断点续传"比普通文件下载难得多?(提示:模型生成有状态、不可幂等重放。)

# 第 13 章 评测与测试

> 你写前端时,改完代码靠测试兜底:单元测试断言 `add(1, 2) === 3`,E2E 测试断言"点击按钮后出现弹窗"。但 Agent 给你出了个难题——同一个问题问两遍,模型可能给出两段措辞不同但都对的答案。`assert ===` 直接傻眼。这章讲怎么给这种**输出不确定、没有唯一正确答案**的系统建立信心:从单元测试,到提示评测,到端到端的 Agent 轨迹评测,再到回归测试和上线监控。核心心法一句话:**断言要从"相等"变成"够好"。**

> **学习目标**
> - 说清**为什么 Agent 难测**:非确定性、无唯一正确答案——为什么不能只靠 `assert ===`。
> - 区分评测的**三个层次**:单元(工具函数)→ 单步输出(提示/单次调用)→ 端到端轨迹(trajectory)。
> - 会做**离线评测**:构建评测集(输入 + 期望/参考)、跑批、打分、出报告。
> - 掌握打分方法:精确匹配/包含、规则/正则、**LLM-as-Judge**——讲原理、给双语代码、讲它的局限和如何降偏差。
> - 做 **RAG 专项评测**(召回/精度、忠实度、相关性,呼应第 8 章)和 **Agent 轨迹评测**(调对工具没、步数、是否完成目标)。
> - 把评测接进**回归测试 / CI**,防止提示或模型升级后悄悄退化;了解**在线评测**(A/B、用户反馈、线上指标)。

**前置知识**:第 5 章[Agent 核心循环与推理范式](../02-核心能力篇/05-agent核心循环与推理范式.md)(知道"轨迹"是什么)、第 8 章[RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)(RAG 评测会呼应它)。你已有的单元测试 / E2E 测试经验是最好的底子。

## 13.1 为什么 Agent 这么难测

先把痛点摆清楚。传统软件测试的根基是**确定性**:同样的输入 → 同样的输出。`add(1, 2)` 永远返回 `3`,所以你能写 `expect(add(1, 2)).toBe(3)`。

LLM 和 Agent 把这个根基抽掉了:

1. **输出非确定(non-deterministic)。** 同一个 prompt 跑两遍,生成的文本可能逐字不同。即便把随机性调到最低,长输出也几乎不可能完全一致。
2. **没有唯一正确答案。** "把这段话总结一下"——有无数种都正确的总结。"杭州今天天气怎么样"——"晴,25 度"和"今天杭州是晴天,气温 25 摄氏度"都对。你没法预先写死一个期望字符串去比相等。
3. **失败是渐变的,不是黑白的。** 传统测试要么过要么挂。LLM 的输出可能"基本对但漏了一点""对但啰嗦""语气不合适"——质量是个**光谱**,不是布尔值。
4. **Agent 还多一层:过程也可能错。** 一个多步 Agent 即使最终答案对了,过程可能很糟(绕了大弯、调错了工具、白烧 token);答案错了也可能是"工具调对了但模型理解错结果"。**得同时评结果和过程。**

> **前端类比**:这就像你测一个**视觉/动画**或**推荐结果**的差别。
> - 测 `add()` 像测一个纯函数——`assert ===` 够了。
> - 测 LLM 输出像测**UI 截图**——你不会逐像素比对(那太脆),而是用**快照测试**:存一张"基准",新结果跟基准比,差太多才报警,而且允许人工确认"这个变化是预期的"。
> - 测 Agent 轨迹像测一条 **E2E 用户流程**——不只看最终页面,还看中间走了哪几步、有没有点错按钮。
>
> 所以本章的评测,你可以理解成**"给不确定系统写的快照测试 + E2E 测试"——只是断言更软**(从"逐字相等"放宽到"语义够接近""调对了工具""完成了目标")。

结论:**你不能放弃测试,只能换一套断言方式。** 下面就是这套方式。

## 13.2 评测的三个层次

别一上来就想"端到端测整个 Agent"。和前端测试金字塔(单元多、集成中、E2E 少)一样,Agent 评测也分层,从确定性高、好测的往不确定、难测的走:

```
        ┌─────────────────────────────────────┐
        │  3. 端到端轨迹 (trajectory)          │  最难测、最贵
        │     整个 Agent 跑完:调对工具没?     │  非确定、要评过程
        │     几步完成?目标达成没?           │
        ├─────────────────────────────────────┤
        │  2. 单步输出 (prompt / 单次调用)     │  较难测
        │     给定输入,单次模型输出够不够好?  │  非确定、软断言
        │     (分类对不对、总结忠不忠实)      │
        ├─────────────────────────────────────┤
        │  1. 单元 (工具函数 / 纯逻辑)         │  好测、确定
        │     工具的实现、解析、格式化         │  传统 assert ===
        └─────────────────────────────────────┘
```

**第 1 层:单元测试——和你现在写的一模一样。**
Agent 里有大量**确定性的普通代码**:工具的实现(`get_weather` 真去调了 API 并正确解析)、第 8 章的切块函数、把模型输出解析成结构化数据的逻辑、SSE 事件的编码……这些**就是纯函数 / 普通逻辑,用你熟悉的 Jest/Vitest/pytest 正常写 `assert ===`**。这一层别用 LLM,确定性的东西就该确定性地测。**很多人忽略这层,其实它最该先做扎实——Agent 出问题,经常是工具实现的普通 bug,跟模型没关系。**

**第 2 层:单步输出评测——非确定性从这里开始。**
给定一个输入,**单次**模型调用的输出够不够好?例如:情感分类对不对、抽取的字段全不全、总结有没有忠实于原文。这一层输出非确定,得用软断言(13.4 的各种打分法)。

**第 3 层:端到端轨迹评测——最难也最重要。**
让整个 Agent 跑完一个真实任务,评它的**轨迹(trajectory)**:调对工具了吗、调了几步、有没有绕路、最终目标达成没。这层最贵(跑一次端到端可能十几秒、好几次模型调用)、最不稳定,但最接近"用户真实体验"。

策略和前端测试金字塔一致:**底层多写、跑得勤(每次提交都跑);顶层精选少量关键场景、跑得稀(合并/发版前跑)。**

## 13.3 离线评测:评测集 + 跑批 + 打分

**离线评测(offline evaluation)**是质量保障的主力:在不接触线上用户的情况下,拿一批预先准备好的测试用例,把你的系统跑一遍,逐条打分,汇总成报告。

> **前端类比**:这就是你的**测试套件 + 测试报告**。评测集 = 一组 test case;跑批 = `npm test`;打分 = 每个 case 的断言;报告 = 那个"X passed, Y failed"的汇总。只不过断言从 `===` 换成了"够好吗"。

### 13.3.1 构建评测集

评测集就是一个列表,每条至少包含**输入**和**参考(期望)**:

```jsonc
// eval_set.jsonl —— 每行一条评测用例
{"id": "1", "input": "这家餐厅太难吃了,再也不来了", "expected": "负面"}
{"id": "2", "input": "服务一般,菜还行", "expected": "中性"}
{"id": "3", "input": "环境很好,菜品惊艳!", "expected": "正面"}
```

构建评测集的几条经验:

- **从真实数据来。** 最好的评测集来自线上真实的用户输入(脱敏后),以及线上出过的 bad case——把每个线上事故固化成一条评测用例,它就再也不会悄悄复发(这正是回归测试,13.6)。
- **覆盖边界和坑。** 不止覆盖"正常情况",更要放那些容易出错的:模棱两可的、含特殊字符的、超长的、恶意注入的、空输入。
- **"参考"要因评测方法而异。** 精确匹配类用例,`expected` 是确定答案;开放生成类(总结、问答),`expected` 可以是"参考答案"(给 LLM-as-Judge 当参照)或"评分标准(rubric)",而不是一个必须逐字相等的字符串。
- **从小做起。** 不必一上来攒几千条。20~50 条精心挑选、覆盖关键场景的,就足够开始发挥价值,后面持续往里加。
- **存成版本化的文件。** JSONL/CSV/YAML 都行,放进 git 跟代码一起版本化——评测集本身也是资产。

### 13.3.2 跑批 + 打分 + 出报告

跑批就是个循环:遍历评测集,对每条跑你的系统,拿输出和 `expected` 比,记录分数,最后汇总。骨架很简单(伪代码):

```
对评测集里的每条 case:
    实际输出 = 跑你的系统(case.input)
    分数 = 打分(实际输出, case.expected)   # 打分方法见 13.4
    记录(case.id, 分数, 实际输出)
汇总:通过率、平均分、失败的 case 列表 → 报告
```

完整可运行的脚本放在 13.5(含 LLM-as-Judge 打分 + 报告输出),这里先把打分方法讲清楚。

## 13.4 打分方法:从"硬比"到"用模型评模型"

打分(scoring / grading)是评测的核心。从最硬最便宜、到最软最贵,有这么几档。**优先用便宜确定的,搞不定再上 LLM-as-Judge。**

### 13.4.1 精确匹配 / 包含

最简单:实际输出和期望**完全相等**,或实际输出**包含**期望关键词。

- **适用**:分类(标签就那几个)、抽取出的确定字段、有标准答案的选择题。
- **不适用**:开放生成(总结、问答、写作)——逐字相等几乎不可能成立。

```typescript
// 精确匹配 / 包含 —— 适合分类、确定字段
const exactMatch = (output: string, expected: string) =>
  output.trim() === expected.trim();
const contains = (output: string, expected: string) =>
  output.includes(expected);
```

### 13.4.2 规则 / 正则

用规则或正则判定输出是否满足某些**结构性**要求:格式对不对、含不含某模式、长度是否合理。

- **适用**:校验 JSON 是否合法、是否含必填字段、有没有泄露敏感词、是否符合某种格式(日期、邮箱)。
- 本质上是把"质量"拆成一组可机判的规则。能用规则判的部分,就别麻烦模型。

```python
import json

def is_valid_json_with_fields(output: str, required: list[str]) -> bool:
    """判定输出是合法 JSON 且包含所有必填字段"""
    try:
        data = json.loads(output)
    except json.JSONDecodeError:
        return False
    return all(k in data for k in required)
```

### 13.4.3 LLM-as-Judge:用模型给模型打分

开放生成(总结好不好、回答切不切题、语气合不合适)用上面那些硬方法没法判——这时用 **LLM-as-Judge(让另一个模型当裁判)**:把"输入 + 模型的输出 +(可选)参考答案 + 评分标准"丢给一个模型,让它按标准打分并说明理由。

**原理**:模型有判断"这段文字好不好"的能力。你给它一份明确的**评分标准(rubric)**和待评内容,它就能像人类评审一样给出分数和理由。这是评测开放生成质量最实用的方法。

**关键:给清晰的 rubric,并让它结构化输出。** 模糊的"打个分"会得到噪声很大的结果;明确的标准(比如"是否完全基于给定资料、有没有编造""是否回答了问题""有无事实错误")才能让裁判稳定。让裁判用结构化输出(第 4 章)吐出 `{score, reasoning}`,既好解析又能留痕看它为什么这么判。

#### TypeScript

```typescript
// llm_judge.ts —— 用一个模型给另一个模型的输出打分
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic(); // 从 ANTHROPIC_API_KEY 读取,绝不硬编码

// 裁判返回的结构:1-5 分 + 理由(结构化输出,便于解析和留痕)
const JUDGE_SCHEMA = {
  type: "object" as const,
  properties: {
    score: { type: "integer", enum: [1, 2, 3, 4, 5], description: "1=很差, 5=很好" },
    reasoning: { type: "string", description: "打这个分的理由" },
  },
  required: ["score", "reasoning"],
  additionalProperties: false,
};

export async function judge(
  input: string,
  output: string,
  reference: string,
): Promise<{ score: number; reasoning: string }> {
  // rubric 写在 system 里:明确的评分标准是降低裁判噪声的关键
  const system = `你是一个严格的评审。根据以下标准给"模型回答"打分(1-5):
- 是否切题、正确回答了用户的问题
- 内容是否与"参考答案"一致,有没有事实错误或编造
- 表达是否清晰
只依据这些标准打分,不要被回答的长度或措辞华丽与否影响。`;

  const res = await client.messages.create({
    model: "claude-opus-4-8", // 裁判建议用更强的模型,见下文
    max_tokens: 512,
    system,
    messages: [
      {
        role: "user",
        content: `【用户问题】\n${input}\n\n【参考答案】\n${reference}\n\n【待评的模型回答】\n${output}`,
      },
    ],
    // 结构化输出:强制裁判按 schema 返回 {score, reasoning}
    output_config: { format: { type: "json_schema", schema: JUDGE_SCHEMA } },
  });

  const text = res.content.find((b) => b.type === "text");
  return JSON.parse((text as Anthropic.TextBlock).text);
}
```

#### Python

```python
# llm_judge.py —— 用一个模型给另一个模型的输出打分
import json
from anthropic import Anthropic

client = Anthropic()  # 从 ANTHROPIC_API_KEY 读取,绝不硬编码

JUDGE_SCHEMA = {
    "type": "object",
    "properties": {
        "score": {"type": "integer", "enum": [1, 2, 3, 4, 5], "description": "1=很差, 5=很好"},
        "reasoning": {"type": "string", "description": "打这个分的理由"},
    },
    "required": ["score", "reasoning"],
    "additionalProperties": False,
}

SYSTEM = """你是一个严格的评审。根据以下标准给"模型回答"打分(1-5):
- 是否切题、正确回答了用户的问题
- 内容是否与"参考答案"一致,有没有事实错误或编造
- 表达是否清晰
只依据这些标准打分,不要被回答的长度或措辞华丽与否影响。"""


def judge(input_text: str, output: str, reference: str) -> dict:
    res = client.messages.create(
        model="claude-opus-4-8",  # 裁判建议用更强的模型,见下文
        max_tokens=512,
        system=SYSTEM,
        messages=[{
            "role": "user",
            "content": f"【用户问题】\n{input_text}\n\n【参考答案】\n{reference}\n\n【待评的模型回答】\n{output}",
        }],
        # 结构化输出:强制裁判按 schema 返回 {score, reasoning}
        output_config={"format": {"type": "json_schema", "schema": JUDGE_SCHEMA}},
    )
    text = next(b.text for b in res.content if b.type == "text")
    return json.loads(text)
```

**LLM-as-Judge 的局限,以及怎么降偏差:**

LLM 裁判不是完美的标尺,有这些已知偏差,必须警惕:

- **位置偏差**:成对比较两个回答时,裁判会偏向放在前面(或后面)的那个。**对策**:做成对比较时,把两个回答的顺序交换各跑一次,取一致的结论;不一致就算平手。
- **冗长偏差**:裁判容易偏向更长、看起来更详尽的回答,哪怕没更对。**对策**:在 rubric 里明确写"不要被长度影响"(上面代码已写),并尽量给客观标准。
- **自我偏好**:模型可能偏好和自己风格相近的输出。**对策**:让**裁判用和被评模型不同(且更强)的模型**——这也是为什么裁判常选更强的模型。
- **标准模糊则噪声大**:rubric 越模糊,分数越不稳。**对策**:把评分标准写得越具体越好,最好拆成几个可独立判断的维度。

降偏差的几条通用手段,记住这几招:
1. **给明确的 rubric**(最重要)——把"好"拆成具体、可判断的维度。
2. **用更强的模型当裁判**——裁判能力要 ≥ 被评对象。
3. **成对比较代替绝对打分**——很多时候"A 和 B 哪个更好"比"给 A 打几分"更稳、更便宜(裁判更容易做相对判断),做 A/B 对比时尤其好用。
4. **同一条多评几次取平均/多数**——抵消单次随机性(代价是更贵)。
5. **拿人工标注校准裁判**——抽一批让人也打分,看裁判和人的一致性(如相关系数),确认裁判靠谱再大规模用。

> **一句话**:LLM-as-Judge 是评开放生成的利器,但它是**有偏的标尺**——用明确 rubric、更强的裁判、成对比较来压偏差,并定期拿人工标注校准。别把它的分当成绝对真理,当成"可大规模、低成本、比人快"的近似信号。

## 13.5 双语代码:一个最小评测脚本

把 13.3 的跑批和 13.4.3 的 LLM-as-Judge 串起来,做一个**最小可运行的评测脚本**:读评测集 → 跑系统 → LLM-as-Judge 打分 → 输出报告。这里把"被测系统"简化成一个直接的模型问答(实际项目里换成你的 Agent / RAG 调用即可)。

#### TypeScript

```typescript
// run_eval.ts —— 最小评测:读评测集 + 跑系统 + LLM-as-Judge 打分 + 报告
import Anthropic from "@anthropic-ai/sdk";
import { readFileSync } from "node:fs";
import { judge } from "./llm_judge"; // 复用 13.4.3 的裁判

const client = new Anthropic();

// 被测系统:这里简化成一次问答;实际换成你的 Agent / RAG 调用
async function systemUnderTest(input: string): Promise<string> {
  const res = await client.messages.create({
    model: "claude-opus-4-8",
    max_tokens: 1024,
    messages: [{ role: "user", content: input }],
  });
  const t = res.content.find((b) => b.type === "text");
  return (t as Anthropic.TextBlock).text;
}

type Case = { id: string; input: string; expected: string };

async function main() {
  // 读 JSONL 评测集(每行一条)
  const cases: Case[] = readFileSync("eval_set.jsonl", "utf-8")
    .trim()
    .split("\n")
    .map((line) => JSON.parse(line));

  const results: { id: string; score: number; reasoning: string }[] = [];

  for (const c of cases) {
    const output = await systemUnderTest(c.input); // 跑系统
    const { score, reasoning } = await judge(c.input, output, c.expected); // 打分
    results.push({ id: c.id, score, reasoning });
    console.log(`  case ${c.id}: ${score}/5  ${reasoning.slice(0, 50)}…`);
  }

  // 汇总报告
  const avg = results.reduce((s, r) => s + r.score, 0) / results.length;
  const passed = results.filter((r) => r.score >= 4).length; // 4 分及以上算通过
  console.log("\n===== 评测报告 =====");
  console.log(`总用例: ${results.length}`);
  console.log(`通过(≥4分): ${passed}  通过率: ${((passed / results.length) * 100).toFixed(1)}%`);
  console.log(`平均分: ${avg.toFixed(2)}/5`);
  const fails = results.filter((r) => r.score < 4);
  if (fails.length) {
    console.log("\n失败用例:");
    fails.forEach((r) => console.log(`  - ${r.id}: ${r.score}/5  ${r.reasoning}`));
  }

  // CI 里可据此决定红绿:通过率低于阈值就让流水线失败
  if (passed / results.length < 0.8) process.exit(1);
}

main();
```

#### Python

```python
# run_eval.py —— 最小评测:读评测集 + 跑系统 + LLM-as-Judge 打分 + 报告
import json
import sys
from anthropic import Anthropic
from llm_judge import judge  # 复用 13.4.3 的裁判

client = Anthropic()


def system_under_test(input_text: str) -> str:
    """被测系统:这里简化成一次问答;实际换成你的 Agent / RAG 调用"""
    res = client.messages.create(
        model="claude-opus-4-8",
        max_tokens=1024,
        messages=[{"role": "user", "content": input_text}],
    )
    return next(b.text for b in res.content if b.type == "text")


def main():
    # 读 JSONL 评测集(每行一条)
    with open("eval_set.jsonl", encoding="utf-8") as f:
        cases = [json.loads(line) for line in f if line.strip()]

    results = []
    for c in cases:
        output = system_under_test(c["input"])         # 跑系统
        verdict = judge(c["input"], output, c["expected"])  # 打分
        results.append({"id": c["id"], **verdict})
        print(f"  case {c['id']}: {verdict['score']}/5  {verdict['reasoning'][:50]}…")

    # 汇总报告
    avg = sum(r["score"] for r in results) / len(results)
    passed = sum(1 for r in results if r["score"] >= 4)  # 4 分及以上算通过
    print("\n===== 评测报告 =====")
    print(f"总用例: {len(results)}")
    print(f"通过(≥4分): {passed}  通过率: {passed / len(results) * 100:.1f}%")
    print(f"平均分: {avg:.2f}/5")
    fails = [r for r in results if r["score"] < 4]
    if fails:
        print("\n失败用例:")
        for r in fails:
            print(f"  - {r['id']}: {r['score']}/5  {r['reasoning']}")

    # CI 里据此决定红绿:通过率低于阈值就让流水线失败
    if passed / len(results) < 0.8:
        sys.exit(1)


if __name__ == "__main__":
    main()
```

> 这个脚本就是评测的"骨架":换 `system_under_test` 的实现、换 `judge` 的 rubric、换评测集,它就能评你的任何系统。**`process.exit(1)` / `sys.exit(1)` 这一步是它能接进 CI 的关键**——通过率低于阈值就让进程非零退出,流水线变红(13.6)。

## 13.6 回归测试与 CI 集成

光评一次没用,关键是**防退化**。Agent 系统里有两样东西经常变,一变就可能让效果悄悄变差:

1. **提示(prompt)。** 你改了一句系统提示想优化 A 场景,结果悄悄搞坏了 B 场景——这在 LLM 应用里极其常见,因为提示的影响是全局且难预测的。
2. **模型。** 厂商发了新模型、或你想从贵模型换到便宜模型——新模型在你的场景上未必更好,可能某些 case 反而退化(第 2 篇讲模型迁移时也强调过)。

**回归测试(regression testing)** 的思路和前端完全一样:**每次改提示 / 换模型,都把评测集重新跑一遍,和上一版的分数对比,掉了就拦住。**

> **前端类比**:这就是**快照测试 + CI 卡口**。改了组件重跑快照,对不上就报"快照变了,确认是不是预期";改了提示重跑评测,通过率/平均分掉了就报"效果退化了"。区别只是断言软——不是"快照字节相等",而是"通过率没有显著下降"。

**接进 CI 的做法**:把 13.5 的评测脚本作为一个 CI 步骤,在合并 PR / 发版前跑。脚本根据通过率决定退出码,流水线据此判红绿:

```yaml
# .github/workflows/eval.yml —— 在 CI 里跑评测(示意)
name: eval
on: [pull_request]
jobs:
  run-eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - name: 跑评测集
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}  # 密钥走 secrets,绝不硬编码
        run: npx tsx run_eval.ts   # 通过率低于阈值时非零退出 → 流水线变红
```

实务上的注意点:
- **评测要花钱花时间(每条都调模型),别在每次 `git push` 都全量跑。** 分层:轻量的单元测试(第 1 层)每次提交跑;贵的 LLM 评测(2、3 层)在 PR 合并 / 发版前跑,或定时(nightly)跑。
- **设合理阈值,容忍噪声。** 因为非确定性,通过率会有小幅波动。别把阈值卡到 100%,留点余量(比如"通过率 ≥ 80%"或"平均分不低于上一版 0.2"),否则会被随机波动频繁误报。
- **失败时给足信息。** 报告里带上失败 case 的输入、实际输出、裁判理由,让人能一眼看出哪退化了、为什么。
- **把线上 bad case 持续加进评测集。** 每修一个线上问题,就把它固化成一条用例——评测集会越来越强,这正是回归测试的复利。

## 13.7 RAG 专项评测

第 8 章讲过 RAG 答得不对要**先分清是检索坏了还是生成坏了**。评测也照这个分层来——RAG 有它专属的指标:

**1. 检索质量(检索环节)**——呼应第 8 章 8.10:
- **召回率(recall)**:该检索到的相关片段,有多少被检索到了?(漏不漏)
- **精确率(precision)**:检索回来的片段里,有多少真相关?(准不准)
- 这两个是经典信息检索指标,做法是准备"问题 → 应命中哪些片段"的标注集,跑检索看命中情况。这层是**确定性的**(命中与否可机判),不需要 LLM。

**2. 答案质量(生成环节)**:
- **忠实度 / 有据性(faithfulness / groundedness)**:答案是不是**基于检索到的资料**,有没有模型自己脑补、编造?这是 RAG 最该盯的指标——RAG 的全部意义就是让答案有出处。判法:把"答案 + 检索到的上下文"丢给 LLM-as-Judge,问"答案里每个论断是否都能在上下文里找到依据"。
- **答案相关性(answer relevance)**:答案切不切题,有没有回答到用户真正问的。
- **上下文相关性(context relevance)**:检索到的上下文和问题相不相关(检索准不准的生成侧视角)。

**忠实度的 LLM-as-Judge 示例**(rubric 是关键——明确要求"逐条对照上下文"):

```python
# 忠实度评测:答案是否完全基于检索到的上下文(伪代码,复用 13.4.3 的 judge 结构)
FAITHFULNESS_SYSTEM = """你是 RAG 答案的事实核查员。给定【检索到的上下文】和【模型答案】,
判断答案中的每个事实性论断是否都能在上下文里找到依据。
打分(1-5):5=完全有据,无任何编造;1=大量内容在上下文里找不到依据(脑补/编造)。
逐条对照,只看"有没有依据",不看答案本身对不对。"""
# 把 上下文 + 答案 拼进 user 消息,让裁判按上面标准输出 {score, reasoning}
```

> **关键**:RAG 评测一定要**分检索和生成两层各自评**。只看最终答案分数,你不知道是检索没捞到资料、还是捞到了但模型没用好——这正是第 8 章 8.10 那条"排错口诀"在评测里的体现。专门的 RAG 评测框架(如 Ragas)就是把上面这些指标(忠实度、答案相关性、上下文相关性、召回率等)封装好了,直接用。

## 13.8 Agent 轨迹评测

到最顶层(13.2 的第 3 层):评整个 Agent 跑一个任务的**轨迹(trajectory)**。回想第 5 章,一条轨迹就是 Agent 循环跑出来的那串"思考 → 调工具 → 看结果 → 再思考……→ 最终答案"。轨迹评测同时看**结果**和**过程**:

| 评什么 | 怎么评 | 例子 |
| --- | --- | --- |
| **是否完成目标(结果)** | 检查最终产出是否达成任务目标 | 用户要"订一张明天去上海的票",最后真的订上了吗 |
| **是否调对工具(过程)** | 检查轨迹里实际调的工具序列,对不对 | 该调 `search` 的地方调了 `search` 没;有没有调不该调的 |
| **工具入参对不对(过程)** | 检查传给工具的参数 | 查天气传的城市/日期对不对 |
| **步数 / 效率(过程)** | 看用了几步、有没有绕路/重复 | 3 步能完成的绕了 8 步,或反复调同一个工具 |
| **是否走了死路 / 卡住(过程)** | 看有没有重复无效调用、有没有触发兜底 | 第 5 章提的"重复检测"场景 |

怎么实现?**第 5 章已经埋好了伏笔——循环里每一步都留痕**(调了哪个工具、什么参数、返回什么、`stop_reason`)。轨迹评测就是拿这份留痕来判:

- **过程类(调对工具没、步数)大多可机判**:把实际工具调用序列和"期望工具序列"比,或直接断言"轨迹里必须出现过对 `search` 的调用"。这是确定性的,不用 LLM。
- **结果类(目标达成没)常需 LLM-as-Judge 或规则**:对于有明确成功标志的(订到票了→数据库有记录),用规则判;对于开放的(写出的报告好不好),用 LLM-as-Judge。

```python
# Agent 轨迹评测:既看结果,也看过程(伪代码)
def eval_trajectory(trajectory: list[dict], task: dict) -> dict:
    """trajectory: 第 5 章循环留痕的步骤列表,每步含 {tool_name, tool_input, result, stop_reason}
    task: {goal, expected_tools, max_steps}"""
    tool_calls = [s["tool_name"] for s in trajectory if s.get("tool_name")]

    return {
        # 过程:期望用到的工具是否都调了(可机判)
        "used_expected_tools": all(t in tool_calls for t in task["expected_tools"]),
        # 过程:步数是否在预算内(可机判)
        "within_step_budget": len(trajectory) <= task["max_steps"],
        # 过程:有没有连续重复调用同一工具(可机判,呼应第 5 章重复检测)
        "no_repeated_calls": len(tool_calls) == len(set(map(str, tool_calls)))
                             or not has_consecutive_dup(tool_calls),
        # 结果:目标是否达成(开放任务用 LLM-as-Judge,有明确标志的用规则)
        "goal_achieved": judge_goal(trajectory[-1].get("final_text", ""), task["goal"]),
    }
```

> **关键**:Agent 轨迹评测的价值在于**抓住"答案对了但过程烂"和"过程对了但答案错"这两类问题**——只看最终答案会漏掉前者(绕了大弯白烧钱),只看过程会漏掉后者(工具调对了但模型理解错结果)。两者都评,才知道 Agent 到底好不好、坏在哪。

## 13.9 在线评测:A/B、用户反馈、线上指标

离线评测(13.3~13.8)是上线前的保障;**在线评测(online evaluation)** 是上线后看真实表现。两者互补:离线评测集再全也是"模拟",真实用户的输入永远更野。

主要手段:

- **A/B 测试。** 把流量分一部分给新版本(新提示 / 新模型 / 新 RAG 策略),对比两版的线上指标。这是判断"改动到底有没有让真实体验变好"的金标准——离线评测说"通过率涨了",还得 A/B 验证用户那边是不是真的更满意。
- **用户反馈。** 最直接的信号:👍/👎、星级、"重新生成"按钮的点击率、用户有没有追问/纠正。这些是免费的、真实的质量标签——**还可以反哺评测集**:被踩的回答固化成评测用例。
- **线上指标。** 量化的健康度信号:
  - 任务完成率 / 放弃率(用户有没有半路离开)
  - 平均轮数、平均工具调用数(Agent 是不是绕路了)
  - 首字延迟(TTFT,呼应第 12 章)、总响应时间
  - 成本(每次会话的 token 花费,呼应第 15 章)、`refusal` 率、报错率
- **线上抽样 + LLM-as-Judge。** 从线上真实对话里抽一批,用 13.4.3 的裁判离线打分,持续监控质量趋势——把 LLM-as-Judge 从"离线评测"延伸到"线上质量监控"。

> 这些线上信号从哪来?靠第 14 章[可观测性与调试](./14-可观测性与调试.md)的埋点和 tracing——每次会话、每步工具调用、每次模型调用的耗时/token/结果都记下来,在线指标和线上抽样评测才有数据可算。成本相关的指标(token 花费、模型选型对成本的影响)则呼应第 15 章[成本与性能优化](./15-成本与性能优化.md)。**离线评测保下限、在线评测追真实——两条腿都要有。**

## 13.10 工具:别自己造轮子

上面的方法都能手写(13.5 就是),但生产里通常用现成的 evals 框架,它们把跑批、打分、报告、可视化、和 tracing 的集成都封装好了。点名几个方向(具体 API / 能力以各自官方文档为准,迭代很快):

| 工具 | 偏向 | 一句话 |
| --- | --- | --- |
| **promptfoo** | 提示 / 单步评测、本地 CI 友好 | 用 YAML 声明评测集和断言(含 LLM-as-Judge),命令行跑批出报告,极易接进 CI |
| **Langfuse / LangSmith** | 全链路 tracing + 评测一体 | 既是第 14 章的可观测平台,也内置评测/数据集/LLM-as-Judge,线上线下打通 |
| **Ragas** | RAG 专项 | 把 13.7 的 RAG 指标(忠实度、答案/上下文相关性、召回等)封装好,专评 RAG |
| **OpenAI Evals** | 通用评测框架 | 开源评测框架,定义评测任务和打分逻辑,跑批对比 |

选型建议:**只评提示/单步、想快速接 CI → promptfoo;要 tracing + 评测一体、线上线下统一 → Langfuse/LangSmith;专做 RAG → Ragas。** 但无论用哪个,**核心概念都是本章这套**(评测集、打分方法、LLM-as-Judge、回归)——框架只是把它工程化,理解了原理,换哪个框架都能上手。

## 常见坑 / 最佳实践

- **想用 `assert ===` 测开放生成。** 总结、问答这类没有唯一答案,逐字相等几乎必挂。该用 LLM-as-Judge / 语义相似度等软断言。
- **忽略第 1 层单元测试。** Agent 出问题经常是工具实现的普通 bug(API 没调对、解析错了),跟模型无关。确定性的代码就该用传统单元测试测扎实,别全推给 LLM 评测。
- **LLM-as-Judge 不给 rubric。** 模糊的"打个分"噪声极大。把"好"拆成具体可判的维度,裁判才稳。
- **裁判用了和被评对象一样弱(或一样)的模型。** 裁判能力要 ≥ 被评对象,且最好不同模型,以减自我偏好;评高难任务时尤其要用更强的裁判。
- **没意识到裁判有位置/冗长偏差。** 成对比较时交换顺序各跑一次;rubric 里明确"别被长度影响";定期拿人工标注校准裁判。
- **评测集没有版本化、没纳入线上 bad case。** 评测集是资产,要进 git;每个线上事故都固化成一条用例,回归测试才有复利。
- **CI 阈值卡死到 100%。** 非确定性会让通过率小幅波动,卡满会被随机波动频繁误报。留余量、容忍噪声。
- **每次 push 都全量跑贵评测。** LLM 评测花钱花时间。分层:单元测试每次跑,LLM 评测 PR/发版前或 nightly 跑。
- **RAG 只看最终答案分数。** 不分检索/生成两层,就不知道是没捞到资料还是没用好资料——调优全靠瞎猜。
- **Agent 只评最终答案、不评轨迹。** 会漏掉"答案对但绕大弯白烧钱"和"过程对但理解错结果"。结果和过程都要评。
- **只有离线评测、不做在线。** 离线评测集再全也是模拟。上线后靠 A/B、用户反馈、线上指标看真实表现。

## 本章小结

1. **Agent 难测的根因是非确定 + 无唯一答案**,失败是渐变而非黑白——所以断言要从"相等"放宽到"够好"。可以理解成给不确定系统写的"快照测试 + E2E 测试"。
2. **评测分三层**:单元(工具函数,传统 `assert ===`,最该先做扎实)→ 单步输出(单次模型调用,软断言)→ 端到端轨迹(整个 Agent,最难最贵)。像测试金字塔一样底层多跑、顶层精选少跑。
3. **离线评测 = 评测集 + 跑批 + 打分 + 报告**。评测集从真实数据和线上 bad case 来,覆盖边界,版本化,从小做起。
4. **打分从硬到软**:精确匹配/包含 → 规则/正则 → **LLM-as-Judge**。能用便宜确定的就别上 LLM。LLM-as-Judge 是评开放生成的利器,但有位置/冗长/自我偏好等偏差——用明确 rubric、更强的裁判、成对比较降偏差,拿人工标注校准。
5. **RAG 分检索(召回/精度,可机判)和生成(忠实度/相关性,LLM 评)两层各自评**;**Agent 轨迹评测同时看结果(目标达成没)和过程(调对工具没、步数、有没有绕路)**,靠第 5 章的循环留痕来判。
6. **回归测试防退化**:改提示/换模型就重跑评测集,掉了就拦——接进 CI(脚本按通过率定退出码),阈值留余量容忍噪声,贵评测分层跑。
7. **在线评测追真实**:A/B、用户反馈(可反哺评测集)、线上指标(完成率、轮数、TTFT、成本、报错率),数据靠第 14 章的可观测性。**离线保下限、在线追真实,两条腿都要。**

## 练习题

1. **(基础)** 给第 8 章你写的 RAG(或任意一个问答系统)建一个 20 条左右的评测集(JSONL,含 `input` 和参考答案),跑通 13.5 的脚本,看通过率和平均分。再故意把系统提示改差一点,重跑,观察分数是否下降——体会回归测试在干什么。

2. **(基础)** 给 13.4.3 的 LLM-as-Judge 换两套不同详细程度的 rubric(一套模糊"打个分",一套明确分维度),对同一批输出各评一遍,对比分数的稳定性(比如同一条评 3 次的方差),亲眼看看 rubric 对噪声的影响。

3. **(进阶)** 实现"成对比较"版的 LLM-as-Judge:给裁判两个回答 A/B,让它选哪个更好;然后**交换 A/B 顺序再跑一次**,统计有多少比例两次结论一致(一致率越低说明位置偏差越大)。

4. **(进阶)** 实现 13.7 的 RAG 分层评测:对一批 RAG 问答,分别算检索层的命中情况(需要"问题→应命中片段"的标注)和生成层的忠实度(LLM-as-Judge)。找一条"答案错"的用例,用这两层分数判断它是检索坏了还是生成坏了。

5. **(挑战)** 实现 13.8 的 Agent 轨迹评测:给第 5 章你写的带工具的 Agent 加上"每步留痕",然后写一个 `eval_trajectory`,对一个任务断言"是否调用了期望的工具、步数是否在预算内、目标是否达成"。再构造一个"答案对但绕了大弯"的 case,验证你的轨迹评测能把它标出来(而只看最终答案不能)。

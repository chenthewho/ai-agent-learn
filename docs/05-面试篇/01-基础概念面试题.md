# 基础概念面试题

> 这一组题考的是"你是否真懂大模型怎么工作"。它们看似简单，却最能区分"调过 API"和"理解原理"两类候选人。面试官常用它们暖场，再顺着你的回答往深处挖。
>
> 对照阅读：[第 2 章 大语言模型基础](../01-基础篇/02-大语言模型基础.md)、[第 3 章 提示工程](../01-基础篇/03-提示工程.md)、[第 4 章 结构化输出与函数调用](../01-基础篇/04-结构化输出与函数调用.md)。
>
> 每题给出：**考点 → 参考答案要点 → 加分项 / 陷阱**。建议先盖住答案自己说一遍，再对照。

---

## 1. 一句话解释大语言模型在做什么？为什么说它本质是"预测下一个 token"？

**考点**：对 LLM 工作机制的最底层理解；能不能把"生成"还原成"概率预测"。

**参考答案要点**：

- LLM 本质是一个**条件概率模型**：给定前面的所有 token（上下文），预测**下一个 token 的概率分布**，然后从分布里挑一个，把它接到序列末尾，再预测下一个——**自回归（autoregressive）地一个一个往外吐**。
- 所谓"理解""推理""写代码"，在机制层面都是这同一件事的涌现结果：模型在海量文本上学到了语言和世界的统计规律，于是"最可能的下一个 token"往往就是正确、连贯、有逻辑的内容。
- 它**没有**显式的数据库、没有"想好整句再说"的规划，是边算边说。这解释了它很多"脾气"：比如生成是流式的（所以能做打字机效果）、比如它会"一本正经地胡说"（见幻觉那题）。

**加分项 / 陷阱**：

- 加分：能点出"token 不是词，是子词单元（subword）"，并联系到计费和上下文窗口都按 token 算。
- 加分：能区分**训练**（学习这个概率分布，一次性、昂贵）和**推理**（用这个分布逐 token 生成，每次调用都在做）。
- 陷阱：别把它说成"它在数据库里查答案"或"它真的懂",面试官会立刻追问幻觉、追问它为什么会算错简单算术——你要能用"统计预测，没有符号计算"来圆。

## 2. token 是什么？它和"字 / 词"是什么关系？为什么所有计费和限制都按 token 算？

**考点**：token 的准确定义，以及它和成本、上下文窗口的关系。

**参考答案要点**：

- **token 是模型处理文本的最小单位**，由**分词器（tokenizer）**把原始文本切出来。它通常是"子词"——常见词可能是 1 个 token，生僻词或长词会被切成几个，标点、空格也占 token。
- token 数 ≠ 字数 ≠ 词数。粗略经验：英文约 1 token ≈ 0.75 个词；中文一个汉字往往是 1～2 个 token（取决于分词器），**别用字数估算中文成本，会偏**。
- 为什么按 token 计费 / 限制：因为模型的**输入、输出、上下文窗口、注意力计算**全都是以 token 为单位进行的——它是模型唯一"看得见"的颗粒。所以**输入 token + 输出 token 分别计价**（通常输出比输入贵几倍）。

**加分项 / 陷阱**：

- 加分：知道**不同模型用不同分词器**，同一段文本在不同模型上 token 数不同；要估算就用官方 tokenizer / `count_tokens` 接口，别拍脑袋。
- 加分：能联系工程——"对话越长，每轮重发的历史越长，输入 token 单调增长，成本和延迟都涨"（接第 4 题的无状态）。
- 陷阱：把 token 等同于"单词"是常见误区；中文场景尤其要小心。

## 3. 什么是上下文窗口（context window）？它满了会怎样？用前端的什么概念类比？

**考点**：上下文窗口的本质、它的硬性约束、超限后的工程后果。

**参考答案要点**：

- **上下文窗口是模型单次推理能"看到"的 token 总量上限**，且通常是 **输入 + 输出共享**这一份预算（你留给输出的越多，能塞的输入就越少）。
- 前端类比：它很像**一次 HTTP 请求能携带的 payload 上限**——你这一趟能带多少信息，是有天花板的，超了就得裁剪或分批。
- 超限的后果：要么 API 直接报错，要么你（或框架）必须**截断/压缩历史**才能塞进去——这就引出了记忆与上下文管理（见[第 7 章](../02-核心能力篇/07-记忆与上下文管理.md)）。
- 重要细节：**窗口大 ≠ 越长越好**。上下文越长，单次越贵越慢，而且存在"**中间迷失（lost in the middle）**"——模型对长上下文中段的信息利用率会下降。所以工程上追求"**只放相关的，别一股脑全塞**"。

**加分项 / 陷阱**：

- 加分：区分"上下文窗口"（单次推理上限）和"模型记忆"（模型不会跨请求自动记住，见下题）这两个常被混为一谈的概念。
- 加分：提到 1M token 这种大窗口模型的出现，但强调"能塞不代表该塞"，RAG/压缩依然必要。
- 陷阱：以为"窗口够大就不用 RAG 了"——成本、延迟、注意力衰减三座大山还在。

## 4. 为什么 LLM 的 `/messages` 类接口是无状态的？这对应用架构意味着什么？

**考点**：这是前端最容易"想当然"的点——能不能讲清"模型不记得上一轮"。

**参考答案要点**：

- **模型推理本身是无状态的**：每次调用，服务器不替你保存任何历史。你这一轮想让它"记得"前面说过什么，**必须把从头到现在的完整对话历史（messages 数组）整个重新发过去**。
- 它就像一个**纯函数**：`output = f(完整历史)`。同样的历史进去，逻辑上得到同样分布的输出；上一次调用的"记忆"不会自动留在服务端。
- 架构含义：
  1. **"记忆"是你应用层的责任**——你得自己存对话、自己拼历史、自己决定截断/压缩策略（短期记忆/长期记忆，见[第 7 章](../02-核心能力篇/07-记忆与上下文管理.md)）。
  2. **输入 token 随对话变长单调增加** → 成本、延迟随之上涨 → 需要上下文压缩、提示缓存（见下文）。
  3. 多轮会话要在自己的存储（Redis / DB）里维护 `session_id → messages`。

**加分项 / 陷阱**：

- 加分：能区分"**模型无状态**"和"**某些封装层（如带 conversation/thread 的高层 API）替你存历史**"——上层可能帮你存，但底层模型推理仍是无状态的，别被封装迷惑。
- 加分：联系提示缓存——历史的稳定前缀可以被缓存，降低重发的成本（见第 11 题）。
- 陷阱：误以为"模型会自己记住上一轮"，于是只发最新一句——这会导致模型完全失忆，是新手最常见的 bug。

## 5. `temperature`（温度）是干什么的？`top_p` 呢？为什么有些新模型把这些采样参数移除了？

**考点**：采样控制的原理，以及对"模型 API 在演进"的敏感度（一个高区分度的点）。

**参考答案要点**：

- 模型每步输出的是**下一个 token 的概率分布**，从分布里挑哪个 token 就叫**采样（sampling）**。
- **`temperature`（温度）**：调节分布的"陡峭/平缓"。温度低（接近 0）→ 分布更尖，倾向选最高概率的 token，输出更确定、更保守；温度高 → 分布更平，低概率 token 也有机会被选中，输出更随机、更发散。
- **`top_p`（核采样）**：只在"累积概率达到 p 的那一小撮候选 token"里采样，是另一种控制随机性的旋钮。**经验上一般只调一个，不要 temperature 和 top_p 同时调**，否则行为难预测。
- **新模型移除采样参数的趋势**：一部分新一代模型（例如 Claude 4.6 及以上）**已移除 `temperature` / `top_p` / `top_k`，给这些模型传采样参数会直接报错（400）**。它们把行为控制的重心从"调采样随机性"转移到了"**控制思考多深 / 努力程度（effort）**"，并推荐**用提示词引导风格**而不是靠调温度。

**加分项 / 陷阱**：

- 加分：点出"**即使 `temperature=0` 也不保证两次输出字字相同**"——浮点运算、并行调度等因素仍会带来微小不确定性。所以"确定性"是相对的。
- 加分：把这题升华成一条工程原则——"**别凭记忆写参数，先确认你用的那个具体模型支持哪些参数**"，因为模型 API 变化极快（具体以官方文档为准）。
- 陷阱：信誓旦旦说"所有模型都能调 temperature"——这在新模型上是错的，会暴露你没跟上变化。

## 6. 什么是"幻觉（hallucination）"？为什么会发生？工程上怎么缓解？

**考点**：对 LLM 最核心缺陷的理解深度，以及有没有落地的缓解手段。这是必考题。

**参考答案要点**：

- **定义**：模型生成**看起来通顺、自信，但事实上错误或凭空捏造**的内容（编造的引用、不存在的 API、错误的数字/日期）。
- **为什么会发生**（要打到机制层）：
  1. 模型本质是"预测最可能的下一个 token"，它优化的是**语言上的合理性，不是事实正确性**——通顺和正确是两回事。
  2. 它没有"我不知道"的天然机制；在缺乏信息时，它倾向于**给出一个统计上像样的答案**，而不是承认不确定。
  3. 知识被压在参数里，**有截止日期、可能过时、可能记混**。
- **缓解手段**（分层说，体现工程思维）：
  1. **接入外部事实**：RAG——先检索权威资料，让模型**基于给定材料作答**而非凭记忆（见[第 8 章](../02-核心能力篇/08-rag检索增强生成.md)）。
  2. **给工具**：让它查数据库、调计算器、查实时 API，而不是自己"想"。
  3. **提示约束**：明确要求"只根据提供的资料回答，资料没有就说不知道""给出引用来源"。
  4. **结构化 + 校验**：用 JSON Schema 约束输出，下游再做规则/事实校验。
  5. **降低生成自由度**：对事实性任务用更低的随机性 / 更强的模型 / 更高 effort。
  6. **评测兜底**：用 LLM-as-Judge / 事实一致性指标在上线前和线上抽检（见[第 13 章](../03-工程篇/13-评测与测试.md)）。

**加分项 / 陷阱**：

- 加分：明确"**幻觉无法被根除，只能被抑制和管控**"——这是诚实且专业的态度，也是开放题里的高频追问（见[第 4 篇开放题](./04-项目经验与开放题.md)）。
- 加分：区分"**事实性幻觉**"和"**指令性幻觉**（不遵守格式/约束）"，缓解手段不同。
- 陷阱：只会说"用更好的模型"或"调低温度"就完事——面试官想听到 RAG / 工具 / 引用 / 校验这套组合拳。

## 7. 什么是提示工程（Prompt Engineering）？写一个好提示有哪些基本功？

**考点**：提示工程是不是"玄学碰运气"，还是有章法可循。

**参考答案要点**：

- **定义**：通过设计输入（指令、上下文、示例、格式约束）来稳定地引导模型产出期望输出的工程实践。它不是玄学，是**可迭代、可版本化、可评测**的工程活。
- 基本功（挑几条讲透，别背清单）：
  1. **角色 + 任务明确**：用 system 设定身份和目标，别让模型猜你要什么。
  2. **给约束和格式**：要什么结构、什么长度、什么语气，明确写出来；要机器解析就上结构化输出。
  3. **提供上下文**：把它需要的资料喂进去（这就是 RAG 的雏形），而不是指望它"知道"。
  4. **拆解复杂任务**：复杂任务分步骤说清，或显式要求它分步思考（CoT）。
  5. **给示例（Few-shot）**：用 1～n 个示范输入→输出对，把"我想要的样子"演示给它看。
  6. **迭代 + 评测**：建小测试集，改一版跑一版，别靠感觉。

**加分项 / 陷阱**：

- 加分：提到**提示要版本管理**（像管代码一样），并配套小评测集做回归——这是工程化的标志。
- 加分：针对新模型（已移除采样参数的那类），**风格/确定性更多靠提示词来控制**，所以提示工程的重要性反而上升了。
- 陷阱：把提示工程说成"凑咒语"——专业回答要强调它的可测量、可迭代。

## 8. Few-shot 和 Chain-of-Thought（CoT）分别是什么？什么时候用？

**考点**：两个最常用的提示技巧，能不能说清原理、适用场景和代价。

**参考答案要点**：

- **Few-shot（少样本提示）**：在提示里放几条"输入→期望输出"的示例，让模型**照着示范来**。
  - 用途：稳定输出格式、教它一个不好用语言描述的隐式规则、提升分类/抽取的一致性。
  - 对比 **Zero-shot**（不给例子，只给指令）：Few-shot 更稳但更费 token。
- **Chain-of-Thought（思维链，CoT）**：引导模型**先写出推理过程，再给结论**（如"让我们一步步想"）。
  - 用途：数学、逻辑、多步推理类任务，显式推理能显著提升正确率。
  - 原理直觉：让模型把"中间计算"也生成出来，相当于给了它"草稿纸"，每一步都基于前一步，减少跳步出错。
- **何时用**：格式/风格不稳 → Few-shot；推理出错、需要分步 → CoT；两者可叠加。

**加分项 / 陷阱**：

- 加分：知道**新一代"会思考"的模型已内置推理能力**，你往往不需要手写"让我们一步步想"，模型自己会在内部思考；这时显式 CoT 的边际收益变小，过度堆叠反而费 token。
- 加分：提到 Few-shot 的示例**要有代表性、覆盖边界 case**，烂示例会把模型带偏。
- 陷阱：以为 CoT 一定提升效果——对简单任务可能纯属浪费 token 和延迟；要看任务。

## 9. 什么是结构化输出？它和"在提示里写'请输出 JSON'"有什么本质区别？

**考点**：理解"约束"和"请求"的差别——工程可靠性的关键。

**参考答案要点**：

- **结构化输出**：在 **API 层面**用 schema（如 JSON Schema）**强约束**模型，使其**只能**吐出符合该结构的合法 JSON。
- 和"提示里写'请输出 JSON'"的本质区别：
  - 提示里写"请输出 JSON"只是**请求**，模型**大概率**照做，但仍可能加前缀寒暄、套 markdown 代码块、漏字段、给非法 JSON——**"大概率"对需要 `JSON.parse()` 的下游程序是不够的**。
  - 结构化输出是**保证**：合法性由 API 兜底，你不用写一堆正则去"抠" JSON、打补丁。
- **核心矛盾**：模型天生输出自然语言（面向人），程序需要结构化数据（面向机器）。结构化输出就是在 API 层架的那座桥。

**加分项 / 陷阱**：

- 加分：知道不同厂商的实现方式（Claude 用 `output_config.format` 传 schema；OpenAI 用 `response_format: {type: "json_schema"}` 或 SDK 的 parse 助手；具体以官方文档为准），但不必背确切字段名。
- 加分：提到"**schema 字段加 description 也是在提示模型**"——好的字段命名和描述能提升填充质量。
- 陷阱：把"结构化输出"和"函数调用"混为一谈——它们机制相关（都靠 schema），但目的不同：前者是**拿一份结构化数据**，后者是**让模型决定调哪个工具**（见下题）。

## 10.（高频核心题）函数调用 / 工具使用（Function Calling / Tool Use）的本质是什么？模型到底执行不执行你的函数？

**考点**：**这是初学者几乎都会答错的题，也是面试官最爱用来筛人的题。**

**参考答案要点**：

- **核心事实：模型不执行任何函数。** 你在请求里用 schema 声明了一组可用工具（`{name, description, input_schema}`），模型做的只是——**判断该不该调某个工具，以及调它该传什么参数**，然后把这个"调用意图 + 参数"**作为结构化输出**返回给你。
- **真正执行工具的是你的代码。** 完整往返是：
  1. 你带着 `tools` 定义发起请求；
  2. 模型若决定用工具，返回一个表示"我想调 `get_weather`，参数 `{city: "北京"}`"的结构化结果（Claude 里体现为 `stop_reason: "tool_use"`）；
  3. **你的程序解析这个意图，真正去执行那个函数**（查 API、读 DB……）；
  4. 你把执行结果作为 `tool_result` 放进下一轮消息发回去；
  5. 模型基于结果生成最终回答（或再要求调下一个工具）。
- 前端类比：**工具就像你给模型注册的一组"回调函数"**——模型只负责说"现在该触发哪个回调、传什么参数"，**触发和执行都在你这边**。模型是大脑，工具是手脚，但手脚归你控制。

**加分项 / 陷阱**：

- 加分：能点出安全含义——**模型只是"请求"调用，要不要真执行、怎么鉴权、要不要人工确认，主动权完全在你的代码**。这是防越权和防提示注入的关键抓手（见[第 16 章](../03-工程篇/16-安全与防护.md)）。
- 加分：能把它和[第 5 章 Agent 循环](../02-核心能力篇/05-agent核心循环与推理范式.md)连起来——把这个往返放进 `while` 循环，模型连续调用工具，就成了 Agent。
- 陷阱（致命）：说成"模型会自动帮我调用 API / 查数据库"——这是事实性错误，几乎一票否决。**模型只输出意图，执行永远在你的代码里。**

## 11. 什么是提示缓存（prompt caching）？它为什么能省钱省时间？

**考点**：一个直接关系到成本的实用机制，能不能讲清原理。

**参考答案要点**：

- **提示缓存**：把请求中**稳定不变的前缀**（如长系统提示、固定的工具定义、大段背景资料）缓存起来；下次请求只要前缀**完全一致**，就**命中缓存**，这部分不必重新完整计算，**按更低的价格计费、且更快**。
- 为什么有效：LLM 是自回归的，处理前缀的计算（注意力的 KV）可以被复用。**前缀匹配**是关键——缓存命中要求被缓存部分逐字节一致，所以要把**不变的内容放前面、变化的内容（如用户最新输入）放后面**。
- 典型收益场景：多轮对话（历史前缀稳定）、RAG（同一批资料反复用）、Agent（系统提示 + 工具定义每轮都重发）。这些场景输入 token 大且重复，缓存收益最明显。

**加分项 / 陷阱**：

- 加分：知道实现细节因厂商而异（Claude 用 `cache_control: {type: "ephemeral"}` 标注可缓存断点；缓存有 TTL/最短长度等限制；具体以官方文档为准），但能讲清"前缀匹配 + 降价"这个原理内核。
- 加分：联系架构——**把 system / 工具定义 / 共享资料放在 messages 前部并保持稳定**，是为缓存命中而做的工程设计。
- 陷阱：以为缓存能跨不同前缀复用，或把动态内容放在前面导致永远不命中。

## 12. 什么是 Embedding（嵌入）？它和对话模型是一回事吗？它解决什么问题？

**考点**：RAG 的地基概念，向量检索的入口。

**参考答案要点**：

- **Embedding 就是把一段文本转成一串数字（向量）**，通常几百到几千维。
- 关键性质：**语义相近的文本，向量也相近；语义无关的，向量离得远**。这是用海量语料训练出来的能力。于是"找意思相近的内容"就变成了"在向量空间里找距离近的点"（常用**余弦相似度**衡量）。
- 它**和对话模型是两回事**：Embedding 由**专门的 Embedding 模型**产出（只负责"文本→向量"），对话模型负责"基于资料生成答案"。两者是不同的模型、不同的接口。
- 解决什么问题：让机器能做**语义检索**（而非只能关键词精确匹配），这是 RAG、语义记忆、去重、聚类、推荐等的基础（见[第 8 章](../02-核心能力篇/08-rag检索增强生成.md)）。

**加分项 / 陷阱**：

- 加分：知道两条铁律——**(1) 建库和查询必须用同一个 Embedding 模型**（不同模型的向量空间不通用，混用结果全乱）；**(2) 维度要匹配**。换 Embedding 模型 = 整库重建。
- 加分：能区分**语义检索（向量）**和**关键词检索（BM25/全文）**各自的长短，并提到混合检索（见[核心能力面试题](./02-核心能力面试题.md)）。
- 陷阱：把"Embedding 模型"和"对话模型"混为一谈，或以为"向量越多维越好"。

## 13. 同样的输入，为什么 LLM 两次回答可能不一样？怎么尽量让它稳定？

**考点**：对"不确定性来源"的理解，以及工程上控制它的手段。

**参考答案要点**：

- 不一样的根源是**采样**：模型输出的是概率分布，每次从中采样可能选到不同 token；温度越高越发散。
- 让它更稳定的手段：
  - 老模型：把 `temperature` 调到 0（贪心选最高概率），但**注意：即便温度为 0 也不保证字字相同**（浮点/并行等因素）。
  - 新模型（已移除采样参数）：**用提示词把任务和格式约束死**、用结构化输出锁定结构、必要时调高 effort 让它想清楚。
  - 通用：**结构化输出 + 下游校验**，把"自然语言的随机性"挡在结构之外；给固定的 system / Few-shot 减少漂移。
- 但要诚实：**完全的逐字确定性几乎拿不到**，工程上追求的是"**结构稳定、语义稳定、可校验**"，而不是字节级复现。

**加分项 / 陷阱**：

- 加分：区分"我要它**格式稳定**"（结构化输出能保证）和"我要它**字字一样**"（基本做不到，也通常没必要）。
- 陷阱：承诺"temperature=0 就完全确定"——不准确，会被追问打脸。

## 14. 系统提示（system prompt）、用户消息（user）、助手消息（assistant）分别是什么角色？为什么要分开？

**考点**：消息数组的基本结构，多轮对话怎么组织。

**参考答案要点**：

- 一次请求由一个**消息数组**构成，每条消息有 `role`：
  - **system**：设定全局身份、规则、约束，优先级高，贯穿整个对话（"你是谁、要遵守什么"）。
  - **user**：用户的输入。
  - **assistant**：模型之前的回复（多轮对话里把历史 assistant 回复也带上）。
  - （工具往返里还会有 `tool_use` / `tool_result` 形式的内容。）
- 为什么分开：让模型清楚**哪些是规则、哪些是对话、哪些是它自己说过的**，从而正确地延续对话、遵守约束。多轮对话就是把这串 user/assistant 交替的历史**整个重发**（呼应无状态）。

**加分项 / 陷阱**：

- 加分：知道**把不变的 system / 工具定义放前面有利于提示缓存**；知道 system 是放约束和防注入提示的重要位置。
- 加分：提到部分新模型**不支持 assistant 预填充（prefill）**，强行预填会报错（具体以官方文档为准）——又一个"别凭记忆写"的例子。
- 陷阱：把所有内容都塞进一条 user 消息、不区分角色，会削弱约束效果。

## 15. 输入 token 和输出 token 为什么分开计价？这对应用设计有什么影响？

**考点**：成本结构的理解，是否有成本意识。

**参考答案要点**：

- 分开计价是因为两者的计算开销不同：**输出 token 通常比输入 token 贵几倍**（生成是逐 token 自回归，开销更大）。
- 对应用设计的影响：
  1. **输入侧**：对话变长会让输入 token 单调增长（无状态、每轮重发历史）→ 需要上下文压缩、提示缓存、RAG 只取相关片段。
  2. **输出侧**：让模型**别啰嗦**（提示约束长度）、用结构化输出避免冗余寒暄，能直接省钱、降延迟。
  3. **模型分级**：简单任务用便宜小模型，难任务才上大模型（见[第 15 章](../03-工程篇/15-成本与性能优化.md)）。

**加分项 / 陷阱**：

- 加分：能给出一个粗略的成本估算思路——"单次成本 ≈ 输入 token × 输入单价 + 输出 token × 输出单价"，并指出对话类应用输入 token 会随轮次累积。
- 加分：提到"思考/推理 token"在某些模型里也计入输出计费，effort 越高越贵——是个容易被忽略的成本点（具体以官方文档为准）。
- 陷阱：只盯着模型单价比价，忽略了"输入累积""输出冗长""思考开销"这些真正吃成本的地方。

## 16. 上下文窗口很大（比如 1M token）了，是不是就不需要 RAG 和上下文压缩了？

**考点**：能不能跳出"窗口大就万事大吉"的误区——一个很好的拔高题。

**参考答案要点**：**不是。** 即便窗口足够大，RAG 和压缩依然必要，原因有三：

1. **成本**：上下文越长，每次调用输入 token 越多、越贵。把无关内容全塞进去是在烧钱。
2. **延迟**：长上下文推理更慢，对话/Agent 体验变差。
3. **效果（注意力衰减）**："**中间迷失**"——模型对超长上下文中段信息的利用率会下降，塞太多反而稀释了关键信息，准确率可能不升反降。

所以正确做法仍是"**只放相关的**"：用 RAG 检索出最相关的片段、用压缩裁掉无用历史，**把宝贵的上下文预算花在刀刃上**。

**加分项 / 陷阱**：

- 加分：一句话总结——"**能塞 ≠ 该塞**；上下文是稀缺、昂贵且有边际递减的资源。"
- 加分：能联系到记忆设计（短期放窗口、长期放向量库按需取）和成本优化。
- 陷阱：被"大窗口"带节奏，回答"以后不用 RAG 了"——这是典型的认知误区。

## 17. 怎么估算和控制一次模型调用的成本？

**考点**：把前面 token / 计费的知识落到可操作的成本控制上。

**参考答案要点**：

- **估算**：成本 ≈ 输入 token × 输入单价 + 输出 token × 输出单价（+ 思考 token，若计入输出）。先用官方 tokenizer / `count_tokens` 量出典型输入输出规模，乘单价得单次成本，再乘 QPS/日活估总量。
- **控制手段**（分层）：
  1. **降输入**：上下文压缩、RAG 精检索、提示缓存命中（稳定前缀降价）。
  2. **降输出**：约束输出长度、结构化输出去冗余、按需开 effort（别无脑 max）。
  3. **选模型**：模型分级，简单任务走便宜模型/缓存命中，难任务才上大模型。
  4. **架构层**：并行/批处理、对可缓存结果做应用层缓存、对高频问题走预生成。

**加分项 / 陷阱**：

- 加分：强调"**先量化再优化**"——没有 trace 和 token 统计就谈省钱是空谈（接[可观测第 14 章](../03-工程篇/14-可观测性与调试.md)）。
- 加分：提到价格和模型 ID 是易变信息，**以官方文档/资源清单为准**。
- 陷阱：只会说"换便宜模型"——忽略缓存、压缩、输出长度这些往往收益更大的手段。

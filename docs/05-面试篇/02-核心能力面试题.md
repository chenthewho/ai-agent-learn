# 核心能力面试题

> 这一组进入 Agent 的"五大支柱"：循环与推理、工具、记忆、RAG、多 Agent。它最能区分"会调 API"和"会搭系统"。面试官常给一个具体场景（"做个能查订单的客服 Agent"），看你能不能把这些能力组合起来、并讲清取舍。

## 1. 什么是 Agent？它和 Chatbot、和固定 Workflow 有什么区别？

**考点**：能不能给 Agent 一个有边界的定义，而不是把什么都叫 Agent。

**参考答案要点**：

- **Agent 的本质**：一个把"调用 LLM"放进**循环**里、并允许模型在每轮**自主决定调用工具**、根据结果决定下一步、直到任务完成的程序。关键词是"**自主决定控制流**"。
- 三者对比：
  | | Chatbot | 固定 Workflow | Agent |
  |---|---|---|---|
  | 控制流 | 一问一答，无工具或单次工具 | **由代码写死**的步骤编排 | **由模型动态决定**走哪步、调几次工具 |
  | 步数 | 单轮 | 预定且固定 | 不定，循环到完成 |
  | 适合 | 闲聊/单点问答 | 流程清晰、可枚举的任务 | 步骤不确定、需要随机应变的任务 |
- 一句话：**Workflow 是"你把路线写死"，Agent 是"你给地图和工具，让模型自己找路"。**

**加分项 / 陷阱**：

- 加分：点出"**不是所有任务都该用 Agent**"。流程固定、可靠性要求高的，用 Workflow 反而更稳更便宜；Agent 的自主性是把双刃剑（不可控、贵、慢）。能说"先能用 Workflow 解决就别上 Agent"显示成熟度。
- 加分：把循环类比成前端的**事件循环 / 状态机**——驱动式地"观察→思考→行动"。
- 陷阱：把任何"调了一次工具的对话"都叫 Agent；或把 Agent 神化成"全自动万能"。

## 2. 描述一下 Agent 的核心循环。它靠什么决定"继续"还是"停止"？

**考点**：能不能手写/讲清那个 `while` 循环——Agent 的心脏。

**参考答案要点**：

- 循环四步：**观察（当前 messages）→ 思考（发给模型，模型决定说话还是调工具）→ 行动（你的代码执行工具）→ 观察（把 `tool_result` 追加回 messages）**，回到第一步。
- **停止条件**：当模型这一轮**不再要求调工具、直接给出最终回答**时退出（Claude 里体现为 `stop_reason === "end_turn"`；要调工具则是 `tool_use`）。
- **必须装"刹车"**：除了自然停止，工程上还要加**最大步数上限、超时、token 预算上限**，防止模型陷入死循环或反复调同一个失败工具把钱烧光。

**加分项 / 陷阱**：

- 加分：能说清各种 `stop_reason` 的处理，包括需要"续跑"的情形（如长任务被中断时把状态接回去继续；具体机制以官方文档为准）。
- 加分：强调"**执行工具的是你的代码，不是模型**"（呼应函数调用本质），循环的主动权在你手里。
- 陷阱：忘了加最大步数/超时，是线上 Agent 烧钱和卡死的头号原因。

## 3. ReAct、Plan-and-Execute、Reflection 三种推理范式分别是什么？怎么选？

**考点**：对主流 Agent 推理范式的理解与取舍判断。

**参考答案要点**：

- **ReAct（Reason + Act）**：交替"推理一步、行动一步、看结果、再推理"。最常见、最灵活，适合**步骤不确定、要边走边看**的任务。缺点：可能走弯路、步数多、token 多。
- **Plan-and-Execute（先规划后执行）**：先让模型把整个任务**拆成计划**，再逐步执行（可配一个执行器）。适合**步骤较多、结构清晰**的任务，计划让过程更可控、可并行、可观测。缺点：计划一旦错了，后面照着错的执行；对动态变化适应性差，常需"执行中重规划"。
- **Reflection（反思 / 自我批判）**：模型产出后，**自己（或另一个评审角色）审查并修正**，多轮迭代提质量。适合**对质量要求高、能定义好坏**的任务（写代码、写报告）。缺点：多轮调用，贵且慢。
- **怎么选**：任务发散不可预测 → ReAct；步骤多且清晰 → Plan-and-Execute；质量优先且可评判 → 叠加 Reflection。三者**可组合**（如先 Plan，每步内 ReAct，最后 Reflection 把关）。

**加分项 / 陷阱**：

- 加分：能讲"组合使用"和"用最简单够用的范式"——不是越复杂越好，每加一层都加成本和延迟。
- 加分：把 Reflection 和"评测/自愈"联系起来。
- 陷阱：把三者说成互斥的"三选一"，或只会背名字说不出适用场景和代价。

## 4. 设计一个好工具（tool）有哪些原则？工具的 description 和参数 schema 为什么重要？

**考点**：工具是 Agent 的手脚，工具设计的好坏直接决定 Agent 能不能正确行动。

**参考答案要点**：

- 核心认知：**工具定义（name + description + input_schema）本身就是提示词**。模型完全靠它来判断"该不该调、怎么调"。写烂了，模型就调错、传错参。
- 设计原则：
  1. **职责单一、语义清晰**：一个工具干一件事；名字和描述让模型一眼懂"什么时候该用我"。
  2. **描述写清边界**：不仅说"做什么"，还要说"**什么时候用 / 不用**""参数含义""返回什么"。
  3. **参数用 schema 强约束**：类型、枚举、必填、范围都定义好；用 description 解释每个参数。
  4. **粒度适中**：太细→模型要编排一堆调用，易乱；太粗→一个工具干太多、难复用。
  5. **返回结果对模型友好**：返回结构化、信息密度高的内容，别把一大坨原始 JSON 甩给它（既费 token 又干扰判断）；必要时在工具里做裁剪/摘要。
  6. **错误要可读**：失败时返回模型能理解、能据此重试的错误信息，而不是抛裸异常。

**加分项 / 陷阱**：

- 加分：从模型视角想问题——"**模型只能看到 schema 和描述，看不到你的实现**"，所以描述就是接口契约。
- 加分：提到**工具数量过多会稀释模型注意力**、增加误选概率；可按场景动态裁剪可用工具集，或用分层/路由。
- 陷阱：只从"后端 API 怎么写"角度答，忘了"这是给模型读的"。

## 5. 并行工具调用有什么好处？有哪些坑？

**考点**：性能优化意识 + 对并发副作用的警觉。

**参考答案要点**：

- **好处**：当模型一轮里请求**多个相互独立**的工具时，并行执行能显著**降低延迟**（如同时查天气、查汇率、查库存，而不是串行等）。
- **坑**：
  1. **依赖关系**：只有**彼此无依赖**的调用才能并行；有先后依赖（B 需要 A 的结果）的不能并行，否则逻辑错。
  2. **副作用与顺序**：写操作（下单、扣款、删文件）并行或乱序可能造成竞态、重复执行、数据不一致——**写类操作要谨慎并行，必要时串行 + 幂等**。
  3. **错误处理**：并行中部分失败怎么办？要能把"哪个成功哪个失败"清楚地回传给模型，而不是一个失败拖垮全部。
  4. **资源/限流**：并行放大了对下游 API 的瞬时压力，可能触发限流。
  5. **结果回传**：多个 `tool_result` 要正确对应到各自的调用 ID 再发回，别串了。

**加分项 / 陷阱**：

- 加分：提"**读多写少时并行收益最大；写操作配合幂等键 + 人工确认**"。
- 加分：联系前端经验——很像 `Promise.all` 的好处与陷阱（一个 reject 怎么处理、副作用顺序）。
- 陷阱：无脑"全并行加速"，忽略副作用和依赖。

## 6. Agent 执行中工具报错、模型乱调、陷入循环，怎么做错误处理与自愈？

**考点**：可靠性工程——这是 demo 和生产的分水岭。

**参考答案要点**：

- **工具报错**：不要直接抛异常中断；把错误**结构化地作为 `tool_result` 回传给模型**（"调用失败：参数 X 不合法 / 服务超时"），让模型据此**重试、换参数、或换路径**。这是"自愈"的基础。
- **重试策略**：对瞬时错误（超时、限流）做**指数退避重试**；对确定性错误（参数非法）不要瞎重试，应反馈让模型改。
- **防死循环**：最大步数、超时、token 预算；检测"反复调同一工具传同样参数还失败"的模式，及时熔断并向上报告。
- **降级**：工具长期不可用时走兜底（返回缓存/默认值/转人工），别让整个 Agent 卡死。
- **可观测**：每一步都记 trace，失败时能回放定位（接[第 14 章](../03-工程篇/14-可观测性与调试.md)）。

**加分项 / 陷阱**：

- 加分：区分"**该让模型自愈的错**（语义/参数问题，反馈给模型）"和"**该由代码兜底的错**（基础设施故障，重试/降级）"——不是所有错都丢给模型。
- 加分：提到**幂等**——重试写操作要带幂等键，避免重复下单。
- 陷阱：要么"一报错就抛异常终止"（太脆），要么"无限让模型重试"（烧钱卡死）；两个极端都不行。

## 7. 短期记忆和长期记忆有什么区别？分别怎么实现？

**考点**：记忆设计——Agent 跨轮、跨会话"记住事情"的工程方案。

**参考答案要点**：

- **短期记忆（工作记忆）**：当前会话的上下文，就是那串 **messages**，活在**上下文窗口**里。受窗口大小限制，会话变长就要截断/压缩。实现：在自己的存储里维护 `session_id → messages`，每轮重发（呼应无状态）。
- **长期记忆**：要**跨会话、长期保留**的信息（用户偏好、历史事实、知识）。它**放不进也不该放进窗口**，所以存到外部：
  - **结构化记忆**：用户档案、偏好 → 存 DB，按需查出来拼进提示。
  - **语义记忆**：大量非结构化知识 → 做成 **Embedding 存向量库**，**按相关性检索**后再拼进上下文（这正是 RAG）。
- 二者关系：**短期记忆是"这次对话窗口里的"，长期记忆是"存在外面、用时再取回窗口"**。

**加分项 / 陷阱**：

- 加分：能讲"**记忆的写入策略**"——什么该沉淀为长期记忆、何时写、怎么去重/更新（不是什么都存）。
- 加分：把长期记忆和 RAG 打通——长期语义记忆的检索机制就是 RAG。
- 陷阱：以为模型自己会记住跨会话的事（它不会，全靠你存取）。

## 8. 对话越来越长、快撑爆上下文窗口了，有哪些上下文压缩 / 管理策略？

**考点**：长会话的实际工程难题，直接关系成本和可用性。

**参考答案要点**：

- **滑动窗口 / 截断**：只保留最近 N 轮，丢弃过早的。简单，但会"忘事"。
- **摘要压缩**：把较早的历史用模型**总结成一段摘要**，用摘要替代原文，保留要点、压缩 token。可分层（近的保留原文、远的压成摘要）。
- **结构化抽取**：从历史里抽出关键事实（决定、待办、用户偏好）存成结构化状态，比原文紧凑得多。
- **外置 + 按需取回（RAG 式）**：把历史/知识存外部，**当前只放相关片段**，需要时再检索回来——避免"全量带在身上"。
- **保护关键信息**：压缩时务必保住 system、关键约束、近期上下文；别把当前任务必需的信息压没了。

**加分项 / 陷阱**：

- 加分：强调"**压缩有信息损失，要按重要性分层**"，并配合提示缓存（稳定前缀缓存降本）。
- 加分：联系"中间迷失"——压缩不仅省钱，**裁掉噪声反而能提升模型对关键信息的利用**。
- 陷阱：只会说"截断最早的"——粗暴截断会丢掉关键上下文导致 Agent 失忆出错。

## 9.（核心题）完整讲一遍 RAG 的流程。离线和在线分别做什么？

**考点**：RAG 是 AI 应用最高频的考点，必须能完整、有序、准确地讲全链路。

**参考答案要点**：

- **一句话**：RAG（检索增强生成）= **先从知识库检索相关资料，把资料拼进提示，再让模型基于资料作答**，用外部知识抑制幻觉、补充模型不知道的私有/实时信息。前端类比：**先查数据库再渲染**。
- **离线（建库 / 索引）**：
  1. **加载**：读入文档（PDF / Markdown / 网页 / DB）。
  2. **切块（chunking）**：切成大小合适、语义自洽的片段。
  3. **Embedding**：每块转成向量。
  4. **入库**：向量 + 原文 + 元数据存进向量库。
- **在线（检索 / 生成）**：
  1. **问题 Embedding**：用**同一个** Embedding 模型把问题转成向量。
  2. **检索 top-k**：在向量库里找最相似的若干片段。
  3. **（可选）重排（rerank）**：用更精的模型对候选精排，提相关性。
  4. **拼上下文**：把命中片段拼进提示（带来源）。
  5. **生成**：模型基于这些资料作答，并可附引用。

**加分项 / 陷阱**：

- 加分：点出"**建库和查询必须用同一个 Embedding 模型，维度要匹配**"，换模型 = 整库重建。
- 加分：能讲"**RAG 效果差，根因常在切块**"（切太大太笼统/硬切割断语义），调 RAG 先看切块。
- 陷阱：把"检索"和"生成"混着说、漏掉切块或 Embedding 环节、不提"基于给定资料作答"这个抑制幻觉的关键点。

## 10. 切块（chunking）、检索、重排（rerank）、混合检索分别解决什么问题？

**考点**：RAG 各环节的细节，区分"会用"和"懂得调"。

**参考答案要点**：

- **切块**：决定"知识的颗粒度"。块太大→向量语义笼统、检索不准、片段长占上下文；太小→丢上下文。要调**块大小**和**重叠（overlap，让边界平滑不割裂语义）**；优先**按语义边界切**（段落/标题）而非死按字数。
- **检索**：给查询向量，从库里取**最相似的 top-k**（向量库用 ANN 索引加速）。k 太小可能漏、太大引入噪声。
- **重排（rerank）**：检索是"粗召回"（快但糙），重排用一个更强的 **cross-encoder/rerank 模型**对候选**精排**，把真正最相关的顶上去。两阶段"**粗召回 + 精排**"兼顾速度和精度。
- **混合检索（hybrid）**：把**关键词检索（BM25/全文，擅长精确词、专有名词、ID）**和**向量检索（擅长语义、近义改写）**结合，取长补短。很多专有名词/型号靠向量反而召不回，混合检索更稳。

**加分项 / 陷阱**：

- 加分：能说"**召回率优先用混合检索 + 大 k，精度优先靠重排收口**"的两段式思路。
- 加分：提到**元数据过滤**（按用户/权限/时间先过滤再检索），尤其多租户场景。
- 陷阱：只知道"向量检索"，不知道关键词检索在专有名词上更靠谱，也不知道重排的存在。

## 11. 什么是 Agentic RAG？它比"一次性检索"强在哪？

**考点**：RAG 的进阶形态，体现把 RAG 和 Agent 循环结合的理解。

**参考答案要点**：

- **传统（一次性）RAG**：query 来了，检索一次，拼上下文，生成。**检索是固定的一步**。
- **Agentic RAG**：把**检索本身变成一个工具**，交给 Agent 在循环里**自主决定**：要不要检索、检索什么、检索几次、要不要换个查询再检索、要不要先拆解问题分别检索再综合。
- 强在哪：
  1. **多跳问题**：需要先查 A 才知道该查 B 的问题，一次性检索搞不定，Agentic 能多轮检索。
  2. **查询改写**：第一次没检到好结果，能换措辞/拆子问题重检。
  3. **按需检索**：能判断"这个问题不需要查库，我直接答"，省一次检索。
  4. **多源**：能根据问题选不同知识库/工具。

**加分项 / 陷阱**：

- 加分：点出代价——**多轮检索更贵更慢**，要加步数/检索次数上限，别让它无限检索。
- 加分：把它和 ReAct 联系（检索就是 Act 的一种）。
- 陷阱：把 Agentic RAG 说成"只是检索质量更好"——核心是"**检索时机和方式由模型动态决定**"。

## 12. 什么时候该上多 Agent？什么时候坚决别上？

**考点**：这是高级题。能不能克制——多 Agent 是把双刃剑，乱上是灾难。

**参考答案要点**：

- **该考虑多 Agent 的信号**：
  1. 任务能**清晰拆分成职责不同的角色**（如研究员/写作/审校），各角色 prompt、工具、上下文差异大。
  2. 单 Agent 的**上下文/工具过多、相互干扰**，拆开后各自上下文更聚焦。
  3. 子任务**可并行**，多 Agent 能提速（如多个研究子 Agent 并行查不同主题）。
- **坚决别上的情况**：
  1. 任务简单、线性——单 Agent 甚至 Workflow 就够，多 Agent 纯属过度设计。
  2. 子任务**强耦合、需要频繁共享中间状态**——多 Agent 间通信开销和上下文同步成本巨大。
  3. 你还没把单 Agent 调可靠——多 Agent 会**放大**单 Agent 的不可靠（错误在 Agent 间传播、调试更难）。
- 原则：**先用单 Agent 把问题解决，确实遇到瓶颈再拆**。多 Agent 带来更高的成本（多次调用）、延迟、复杂度和调试难度。

**加分项 / 陷阱**：

- 加分：一句话——"**多 Agent 解决的是'分工和上下文隔离'，不是'让 Agent 更聪明'**"；它不会凭空提升单点能力。
- 加分：提到典型适用场景"**深度研究/报告生成**"（可并行检索多主题再综合）。
- 陷阱：一上来就堆一堆 Agent 显得"高级"——面试官更欣赏"能不用就不用"的克制。

## 13. 多 Agent 有哪些常见架构 / 编排模式？

**考点**：对多 Agent 组织方式的体系化认识。

**参考答案要点**：

- **Supervisor / Orchestrator（主管编排）**：一个主管 Agent 负责拆任务、分派给子 Agent、汇总结果。子 Agent 之间不直接通信，全经主管。清晰、可控，最常用。
- **Handoff（交接 / 路由）**：把控制权在 Agent 间**移交**（如客服总线路由到"退款 Agent""技术支持 Agent"）。适合按意图分流。
- **Pipeline（流水线）**：Agent 串成固定链（研究→写作→审校），前一个的输出是后一个的输入。更接近 Workflow，可控但不灵活。
- **并行 + 聚合（Map-Reduce 式）**：主管把子任务并行分给多个同类子 Agent，再聚合（深度研究系统的典型形态）。
- **协作 / 辩论**：多个 Agent 互相评审、辩论后收敛（提质量，但贵且不一定稳定）。

**加分项 / 陷阱**：

- 加分：能讲"**Agent 间怎么通信**"——通过结构化消息/共享状态/Handoff 协议传递，以及**每个子 Agent 的上下文应当隔离**（见下题）。
- 加分：把 Supervisor 类比成前端的"**状态提升 + 中心调度**"。
- 陷阱：只会说"多个 Agent 一起干"，说不出编排结构和通信方式。

## 14. 多 Agent 里为什么要做"上下文隔离"？怎么做？

**考点**：多 Agent 工程里一个关键且常被忽视的点。

**参考答案要点**：

- **为什么隔离**：
  1. **聚焦**：每个子 Agent 只看到自己任务相关的上下文，不被别人的对话历史/工具噪声干扰，判断更准。
  2. **省 token / 控成本**：不必把全局历史复制给每个子 Agent。
  3. **防污染/防泄漏**：一个 Agent 的错误或敏感信息不无差别扩散到其他 Agent；权限可按 Agent 隔离。
- **怎么做**：
  1. 每个子 Agent 维护**独立的 messages / 记忆**，只接收主管下发的**精炼任务说明 + 必要输入**，而不是整个对话历史。
  2. 子 Agent 完成后**只把结构化结果**回传主管，主管决定怎么用，而非把整段过程灌给别人。
  3. 工具/权限**按角色最小化授予**（研究 Agent 只有检索工具，下单 Agent 才有写工具）。

**加分项 / 陷阱**：

- 加分：把它和"上下文窗口是稀缺资源""中间迷失""最小权限"三点串起来。
- 加分：提到"**主管负责跨 Agent 的信息编排（谁该知道什么）**"是隔离的关键。
- 陷阱：把全局历史无脑共享给所有子 Agent——既贵又乱还不安全。

## 15. 工具返回了一大坨数据（比如一个超长 JSON / 整个网页），直接塞回模型有什么问题？怎么办？

**考点**：上下文经济意识 + 工具结果处理的工程细节。

**参考答案要点**：

- **问题**：
  1. **吃 token、涨成本、增延迟**，可能直接撑爆上下文。
  2. **稀释关键信息**：噪声多，模型抓不住重点（中间迷失）。
  3. 后续每一轮都重发这坨东西（无状态），代价叠加。
- **怎么办**：
  1. 在**工具实现里**就做裁剪/过滤/摘要，只返回**模型决策真正需要的字段**。
  2. 大结果**落外部存储**，工具只回一个**引用/ID + 摘要**，模型需要细节时再用另一个工具按需取（"二次取回"）。
  3. 对超长文本先**抽取/总结**再回传。
  4. 用结构化输出规范工具返回格式，信息密度高。

**加分项 / 陷阱**：

- 加分：一句话——"**工具的返回值也是上下文，要像管 payload 一样精打细算**"。
- 加分：联系 Agentic RAG 的"按需检索/取回"思想。
- 陷阱：把"原始响应整个 return"当成默认做法。

## 16. 怎么让 Agent 在执行高风险动作（删数据、付款、发邮件）前停下来确认？

**考点**：人机协同（human-in-the-loop）+ 安全意识。

**参考答案要点**：

- 核心抓手回到函数调用本质——**模型只是"请求"调用工具，真正执行在你的代码里**，所以你完全可以在执行前插入确认环节。
- 做法：
  1. 给工具**标注风险级别**；对高风险工具，模型请求调用后**不立即执行**，而是**暂停、把"将要做什么 + 参数"呈现给用户确认**，用户批准后才执行（human-in-the-loop）。
  2. 对中风险可走"**自动执行但可撤销/可审计**"。
  3. 配合**最小权限**：Agent 默认只有读权限，写/危险操作需显式授权。
  4. **幂等 + 审计日志**：执行的危险操作要可追溯、可回滚。

**加分项 / 陷阱**：

- 加分：把它和**提示注入防御**联系——确认环节能挡住"被注入指令诱导去删库"的攻击（见[第 16 章](../03-工程篇/16-安全与防护.md)）。
- 加分：前端视角——这就像危险操作弹"二次确认对话框"，只是对象换成了 Agent 的行动。
- 陷阱：以为"靠提示词告诉模型别乱删"就安全——提示约束可被绕过，**硬控制（代码层确认/权限）才可靠**。

## 17. Agent 在多步任务中"跑偏了"（偏离目标、漏掉要求），怎么纠偏？

**考点**：长任务的方向保持，体现对 Agent 可靠性的把控。

**参考答案要点**：

- **目标常驻**：把任务目标/约束放在 system 或每轮上下文的稳定位置，别被长历史淹没；必要时周期性"重述目标"。
- **计划 + 勾选**：用 Plan-and-Execute 把任务拆成可核对的步骤，执行中对照计划检查"还差什么"。
- **Reflection / 自检**：在关键节点让模型（或一个评审角色）核对"是否满足了全部要求"，发现偏离就修正。
- **结构化中间状态**：用结构化的"待办/已完成"状态跟踪进度，比让模型从长文本里回忆更可靠。
- **可观测**：trace 每步的"思考+行动"，跑偏了能定位是哪一步开始歪的。

**加分项 / 陷阱**：

- 加分：联系"中间迷失"——长上下文里目标容易被稀释，所以要**主动把目标顶到显眼处**。
- 陷阱：只靠"加长 prompt 强调要求"——长了反而更容易被忽略；结构化跟踪 + 节点自检更有效。

## 18. RAG 检索回来的内容和模型自身知识冲突时，怎么让它"信资料"而不是"信自己"？

**考点**：RAG 落地中的真实难题——知识冲突。

**参考答案要点**：

- **提示明确优先级**：在 system/指令里写死"**以提供的资料为准；资料与你已有知识冲突时，以资料为准；资料没有就说不知道，不要编造**"。
- **给来源、要引用**：要求模型**标注每条结论来自哪段资料**，没有出处的结论视为不可信——既约束它，又便于核查。
- **检索质量是前提**：如果检索本身召回了错的/不相关的片段，模型"信资料"反而错。所以**先把检索/重排做准**，再谈让它信资料。
- **冲突可显式处理**：让模型在资料相互矛盾时**指出冲突**而非强行编一个答案。

**加分项 / 陷阱**：

- 加分：点出"**RAG 抑制幻觉的前提是检索到对的资料**；检索错了，RAG 会一本正经地照错资料答"——这是 RAG 的真实局限。
- 加分：联系评测——要专门评"**忠实度 / 是否基于资料（faithfulness）**"和"**引用是否正确**"。
- 陷阱：以为"上了 RAG 就没幻觉了"——错。检索质量差、提示不约束，照样幻觉。

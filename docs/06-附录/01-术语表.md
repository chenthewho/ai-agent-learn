## 一、基础与模型

| 术语 | 英文 | 一句话解释（带前端类比） |
|---|---|---|
| 大语言模型 | LLM, Large Language Model | 一个"超大号的文本补全函数"：给它一段文字，它预测最可能的下一段文字。类比：一个被海量代码和文档训练过的、概率版的 `autocomplete`。详见[基础篇 · 大语言模型基础](../01-基础篇/02-大语言模型基础.md)。 |
| Token | Token | 模型处理文本的最小计费/计数单位，约等于"半个英文单词"或"一个汉字多一点"。类比：HTTP 请求里按字节计费，模型按 token 计费。**所有限额和价格都按 token 算。** |
| 上下文窗口 | Context Window | 一次请求里模型"能看到"的 token 上限（输入+输出）。类比：一次请求能携带的 payload 大小上限——超了就得裁剪或压缩。详见[基础篇 · 大语言模型基础](../01-基础篇/02-大语言模型基础.md)、[核心能力篇 · 记忆与上下文管理](../02-核心能力篇/07-记忆与上下文管理.md)。 |
| Temperature | Temperature | 控制输出"随机性/发散度"的旋钮，0 最确定、越高越天马行空。类比：`Math.random()` 的权重系数。注意：部分新模型（如 Opus 4.x）已不接受该参数，靠提示词来控制风格。 |
| 系统提示词 | System Prompt | 给模型设定身份、规则、口吻的"全局配置"，优先级高于普通对话。类比：应用的全局 config / 中间件，先于业务逻辑生效。详见[基础篇 · 提示工程](../01-基础篇/03-提示工程.md)。 |
| 少样本示例 | Few-shot | 在提示里塞几个"输入→期望输出"的例子，让模型照葫芦画瓢。类比：给函数几组单元测试用例当"行为说明书"。详见[基础篇 · 提示工程](../01-基础篇/03-提示工程.md)。 |
| 零样本 | Zero-shot | 不给例子，直接下指令让模型干活。与 Few-shot 相对。 |
| 思维链 | CoT, Chain-of-Thought | 让模型"先写出推理步骤再给答案"，复杂题目正确率显著提高。类比：要求函数打印中间日志而不只返回最终结果。详见[基础篇 · 提示工程](../01-基础篇/03-提示工程.md)。 |
| 思考 | Thinking / Extended Thinking | 模型在给正式答案前的内部推理过程，部分模型可配置展示摘要。新一代 Claude 用 `thinking: {type: "adaptive"}`（自适应，由模型自己决定想多深），**不要再用旧的 `budget_tokens`**。类比：组件渲染前的一段同步计算，结果你可选择是否打印出来。 |
| 努力程度 | Effort | 控制模型"想多深、花多少 token"的档位（`low`/`medium`/`high`/`xhigh`/`max`），放在 `output_config.effort` 里。类比：构建时的优化级别 `-O0`～`-O3`，越高越慢越费但通常更好。详见[工程篇 · 成本与性能优化](../03-工程篇/15-成本与性能优化.md)。 |
| 幻觉 | Hallucination | 模型一本正经地编造不存在的事实/接口/引用。类比：拼写检查器"自信地"把对的词改错。对付它的主要手段是 RAG（给它真实资料）和让它引用来源。详见[基础篇 · 大语言模型基础](../01-基础篇/02-大语言模型基础.md)。 |
| 多模态 | Multimodal | 模型不只能读文字，还能读图片（部分还能读 PDF、音频）。类比：`<input>` 从只收 text 变成也收 file/image。 |
| 提示工程 | Prompt Engineering | 系统性地设计提示词以稳定拿到想要的输出的手艺。详见[基础篇 · 提示工程](../01-基础篇/03-提示工程.md)。 |

## 二、Agent 核心

| 术语 | 英文 | 一句话解释（带前端类比） |
|---|---|---|
| 智能体 | Agent | 能**自己决定调用哪些工具、循环推进直到完成任务**的 LLM 应用，而不只是一问一答。类比：从"纯函数"升级成"带状态、会自己发请求、会重试的事件循环"。详见[核心能力篇 · Agent 核心循环与推理范式](../02-核心能力篇/05-agent核心循环与推理范式.md)。 |
| Agent 循环 | Agent Loop | "模型决策 → 执行工具 → 把结果喂回模型 → 再决策"的反复过程，直到任务完成。类比：浏览器的事件循环（event loop）——一轮轮处理任务直到队列清空。详见[核心能力篇 · Agent 核心循环与推理范式](../02-核心能力篇/05-agent核心循环与推理范式.md)。 |
| ReAct | ReAct (Reason + Act) | 一种 Agent 范式：交替进行"推理（Reason）"和"行动（Act，调工具）"。类比：`while` 循环里先 `think()` 再 `doAction()` 再看结果。出自经典论文，是最常见的 Agent 套路。详见[核心能力篇 · Agent 核心循环与推理范式](../02-核心能力篇/05-agent核心循环与推理范式.md)。 |
| 规划-执行 | Plan-and-Execute | 先让模型一次性列出完整计划，再逐步执行。与 ReAct（边想边做）相对，适合步骤清晰的长任务。详见[核心能力篇 · Agent 核心循环与推理范式](../02-核心能力篇/05-agent核心循环与推理范式.md)。 |
| 反思 | Reflection | 让模型回看自己的输出、找问题、再改进的一轮自我批评。类比：提交前自己 code review 一遍。详见[核心能力篇 · Agent 核心循环与推理范式](../02-核心能力篇/05-agent核心循环与推理范式.md)。 |
| 记忆 | Memory | Agent 在对话内/跨会话保存与取用信息的机制。**短期记忆**=本次对话历史（放在上下文窗口里）；**长期记忆**=写到外部存储（文件/数据库/向量库），下次再读回来。类比：短期=组件内 `state`，长期=`localStorage`/后端数据库。详见[核心能力篇 · 记忆与上下文管理](../02-核心能力篇/07-记忆与上下文管理.md)。 |
| 上下文工程 | Context Engineering | 精心决定"每一轮请求往上下文窗口里放什么"的工程实践（裁剪、排序、缓存、压缩）。类比：精打细算每个请求的 payload，既要带够信息又不能超限。详见[核心能力篇 · 记忆与上下文管理](../02-核心能力篇/07-记忆与上下文管理.md)。 |
| 压缩 | Compaction | 上下文快满时，把早期历史**自动总结**成一段摘要，腾出空间继续对话。类比：日志 rotation——旧日志压成摘要归档。注意它与"上下文裁剪（Context Editing，直接删旧内容）"不同，一个是总结、一个是删除。详见[核心能力篇 · 记忆与上下文管理](../02-核心能力篇/07-记忆与上下文管理.md)。 |
| 上下文裁剪 | Context Editing | 把陈旧的工具结果/思考块**直接清除**（不是总结），让对话保持精简。与 Compaction 相对。详见[核心能力篇 · 记忆与上下文管理](../02-核心能力篇/07-记忆与上下文管理.md)。 |
| 子 Agent | Subagent | 主 Agent 派生出来干某个子任务的独立 Agent，有自己的上下文。类比：主线程 `postMessage` 派一个 Web Worker 去干活。详见[核心能力篇 · 多 Agent 协作系统](../02-核心能力篇/09-多agent协作系统.md)。 |
| 多 Agent | Multi-Agent | 多个各有专长的 Agent 协作完成一个大任务。类比：微服务架构——一个服务专做一件事，互相调用。详见[核心能力篇 · 多 Agent 协作系统](../02-核心能力篇/09-多agent协作系统.md)。 |
| 编排者 | Orchestrator / Coordinator | 多 Agent 系统里负责"分派任务、汇总结果"的调度 Agent。类比：API 网关 / 主控线程。详见[核心能力篇 · 多 Agent 协作系统](../02-核心能力篇/09-多agent协作系统.md)。 |
| 交接 | Handoff | 一个 Agent 把任务（连同上下文）转交给另一个更合适的 Agent。类比：客服把工单转派给对应专家，连同上下文一起带过去。详见[核心能力篇 · 多 Agent 协作系统](../02-核心能力篇/09-多agent协作系统.md)。 |

## 三、工具与函数调用

| 术语 | 英文 | 一句话解释（带前端类比） |
|---|---|---|
| 工具 | Tool | 你"注册"给模型、让它在需要时调用的函数（带名字、描述、参数 schema）。类比：给模型注册的一组回调函数 / 可调 API。详见[核心能力篇 · 工具系统设计](../02-核心能力篇/06-工具系统设计.md)。 |
| 函数调用 / 工具使用 | Function Calling / Tool Use | 模型不直接执行函数，而是**输出"我想调用某函数、参数是这些"**，由你的代码执行后把结果喂回去。类比：模型只发"事件"，真正的 handler 在你这边跑。Claude 里响应 `stop_reason: "tool_use"` 时就该执行工具。详见[基础篇 · 结构化输出与函数调用](../01-基础篇/04-结构化输出与函数调用.md)、[核心能力篇 · 工具系统设计](../02-核心能力篇/06-工具系统设计.md)。 |
| 工具结果 | Tool Result | 工具执行完后回传给模型的结果，作为 `tool_result` 放进下一轮 `user` 消息。类比：回调的返回值再交回给状态机。 |
| 结构化输出 | Structured Output | 强制模型按指定的 JSON Schema 吐出结果，保证可被代码直接解析。类比：给接口加了 TypeScript 类型 + 运行时校验，不再是"裸 any"。Claude 用 `output_config: {format: {type: "json_schema", ...}}`。详见[基础篇 · 结构化输出与函数调用](../01-基础篇/04-结构化输出与函数调用.md)。 |
| JSON Schema | JSON Schema | 描述一个 JSON 数据"长什么样"的标准（字段、类型、必填项）。类比：用对象描述的运行时类型定义，前端的 zod/yup 也是干这个的。详见[基础篇 · 结构化输出与函数调用](../01-基础篇/04-结构化输出与函数调用.md)。 |
| 并行工具调用 | Parallel Tool Use | 模型在一轮里同时要求调多个工具，你并发执行后把所有结果一次性回传。类比：`Promise.all([...])`。**注意：所有结果要放在同一条 user 消息里回传。** 详见[核心能力篇 · 工具系统设计](../02-核心能力篇/06-工具系统设计.md)。 |
| 服务端工具 | Server-side Tool | 由模型厂商在自己服务器上执行的工具（如联网搜索、代码执行），你只需声明、无需本地实现。类比：调用一个托管的云函数，你只管声明用它。 |
| 客户端工具 | Client-side Tool | 你自己实现并执行的工具（如读写本地文件、查你的数据库）。类比：你自己写的业务 handler。 |
| MCP | MCP, Model Context Protocol | 一套"让 Agent 接入外部工具/数据源"的开放协议，相当于工具界的统一插头。类比：像 LSP（Language Server Protocol）统一了编辑器与语言服务，MCP 统一了 Agent 与工具服务。详见[工程篇 · MCP 与工具生态](../03-工程篇/11-mcp与工具生态.md)。 |
| MCP Server | MCP Server | 按 MCP 协议对外暴露工具/资源的服务进程，Agent 作为客户端连接它。类比：一个标准化的"工具后端"。详见[工程篇 · MCP 与工具生态](../03-工程篇/11-mcp与工具生态.md)。 |
| 工具搜索 | Tool Search | 工具特别多时，让模型先"搜索"出当前相关的少数工具再加载其 schema，省上下文。类比：路由懒加载——用到哪个再 import 哪个。详见[工程篇 · MCP 与工具生态](../03-工程篇/11-mcp与工具生态.md)。 |

## 四、RAG 与检索

| 术语 | 英文 | 一句话解释（带前端类比） |
|---|---|---|
| 检索增强生成 | RAG, Retrieval-Augmented Generation | 回答前先去知识库"查资料"，把查到的内容塞进提示再让模型作答，从而减少幻觉、用上私有/最新数据。类比：渲染前先 `fetch` 数据库再 render，而不是凭记忆瞎写。详见[核心能力篇 · RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)。 |
| 嵌入 / 向量化 | Embedding | 把一段文本转成一串数字（向量），语义相近的文本向量也相近。类比：给每段文本算一个"语义指纹/哈希"，但相似内容指纹也相似。详见[核心能力篇 · RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)。 |
| 向量 | Vector | Embedding 产出的那一串浮点数（如 1536 维），代表文本在"语义空间"里的坐标。 |
| 向量库 / 向量数据库 | Vector Store / Vector Database | 专门存向量并支持"找最相似的 K 个"的数据库（pgvector、Qdrant、Chroma 等）。类比：一个能按"语义相似度"而非"精确匹配"来查询的数据库。详见[核心能力篇 · RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)。 |
| 余弦相似度 | Cosine Similarity | 衡量两个向量"方向有多接近"的常用指标，越接近 1 越相似。类比：判断两段文本语义像不像的"距离函数"。详见[核心能力篇 · RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)。 |
| 切块 | Chunking | 把长文档切成一段段小块再做 embedding 与检索（太大装不进上下文、太小丢语义）。类比：把大列表做分页/虚拟滚动，按块处理。详见[核心能力篇 · RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)。 |
| Top-K 检索 | Top-K Retrieval | 检索时取相似度最高的前 K 条结果喂给模型。类比：`results.slice(0, k)`。详见[核心能力篇 · RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)。 |
| 重排 | Rerank / Reranking | 对初步检索出的候选用更精细的模型重新打分排序，把最相关的提到前面。类比：先用快但糙的索引粗筛，再用精细规则二次排序。详见[核心能力篇 · RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)。 |
| 混合检索 | Hybrid Search | 同时用"关键词检索（BM25）"和"向量语义检索"，再合并结果，兼顾精确匹配与语义匹配。类比：既支持精确 `WHERE name = ?`，又支持模糊语义搜索，两者结果取并集。详见[核心能力篇 · RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)。 |
| Agentic RAG | Agentic RAG | 让 Agent 自己决定"要不要查、查什么、查几次、要不要换关键词再查"的 RAG，而非固定一次检索。类比：从"渲染前固定查一次"升级成"组件按需多次 fetch、还会根据结果决定下一步查什么"。详见[核心能力篇 · RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)。 |
| 召回率 / 精确率 | Recall / Precision | 召回率=该找到的有没有都找到；精确率=找到的里面有多少是对的。类比：搜索结果的"全不全"与"准不准"。详见[核心能力篇 · RAG 检索增强生成](../02-核心能力篇/08-rag检索增强生成.md)。 |

## 五、工程与部署

| 术语 | 英文 | 一句话解释（带前端类比） |
|---|---|---|
| 流式输出 | Streaming | 模型边生成边返回，前端边收边渲染，不必等全部生成完。类比：`ReadableStream` / 边下边播，而不是等整个响应到齐。详见[工程篇 · 流式输出与前端集成](../03-工程篇/12-流式输出与前端集成.md)。 |
| SSE | SSE, Server-Sent Events | 服务端单向持续推消息给浏览器的标准协议，是流式输出最常用的传输方式。类比：你早就熟悉的 `EventSource`。详见[工程篇 · 流式输出与前端集成](../03-工程篇/12-流式输出与前端集成.md)。 |
| 首 Token 时延 | TTFT, Time To First Token | 从发请求到收到第一个 token 的等待时间，决定"页面多久开始有反应"。类比：首屏时间 / TTFB。详见[工程篇 · 成本与性能优化](../03-工程篇/15-成本与性能优化.md)。 |
| 提示缓存 | Prompt Caching | 把请求里"不变的前缀"（如长系统提示、文档）缓存住，后续命中只按极低价计费、也更快。**前缀匹配**：前缀任何一个字节变了，缓存就失效。类比：HTTP 缓存的 ETag / CDN 边缘缓存。详见[工程篇 · 成本与性能优化](../03-工程篇/15-成本与性能优化.md)。 |
| 框架 | Framework | 替你封装好 Agent 循环、工具调用、记忆等的库（LangGraph、Vercel AI SDK、Mastra 等）。类比：从手写 DOM 操作升级到用 React/Vue。**先懂原理再用框架。** 详见[工程篇 · Agent 框架选型](../03-工程篇/10-agent框架选型.md)、[资源与工具清单](./03-资源与工具清单.md)。 |
| 薄抽象层 | Thin Abstraction | 在你的代码和各家模型 API 之间封一层统一的 `chat()`/`generate()` 函数，方便换厂商。类比：自己封装的 `request()` 把 axios/fetch 差异藏起来。详见[基础篇 · 大语言模型基础](../01-基础篇/02-大语言模型基础.md)。 |
| 可观测性 | Observability | 能看清 Agent"内部到底发生了什么"的监控能力（看每一步的输入输出、耗时、花费）。类比：前端的 Sentry + 性能埋点，但对象是 Agent 的每一步推理与工具调用。详见[工程篇 · 可观测性与调试](../03-工程篇/14-可观测性与调试.md)。 |
| 追踪 | Trace | 一次完整 Agent 任务从头到尾的记录（含所有步骤、子调用）。类比：一条分布式链路追踪。详见[工程篇 · 可观测性与调试](../03-工程篇/14-可观测性与调试.md)。 |
| 跨度 | Span | 一次 Trace 里的某一个步骤/子操作（一次模型调用、一次工具调用）。类比：链路追踪里的一段 span。详见[工程篇 · 可观测性与调试](../03-工程篇/14-可观测性与调试.md)。 |
| 轨迹 | Trajectory | Agent 为完成任务实际走过的"决策+行动序列"。类比：用户在 App 里的操作回放路径，用来评测它走得对不对。详见[工程篇 · 评测与测试](../03-工程篇/13-评测与测试.md)。 |
| 批处理 | Batch API | 把大量不急的请求打包异步处理，价格通常对半砍。类比：把零散请求合并成定时任务批量跑。详见[工程篇 · 成本与性能优化](../03-工程篇/15-成本与性能优化.md)。 |
| 重试与退避 | Retry / Exponential Backoff | 请求失败（限流/5xx）时自动重试，并逐次拉长等待间隔。类比：fetch 失败后的指数退避重试，官方 SDK 一般已内置。详见[工程篇 · 部署与生产化](../03-工程篇/17-部署与生产化.md)。 |
| 部署 | Deployment | 把 Agent 应用上线到生产环境（密钥管理、限流、监控、扩缩容）。详见[工程篇 · 部署与生产化](../03-工程篇/17-部署与生产化.md)。 |

## 六、安全与评测

| 术语 | 英文 | 一句话解释（带前端类比） |
|---|---|---|
| 评测 | Evaluation / Eval | 用一套数据集和打分标准，量化衡量 Agent 表现好不好、改动有没有变差。类比：自动化测试 + 回归测试套件，只不过输出是"自然语言/行为"而非精确值。详见[工程篇 · 评测与测试](../03-工程篇/13-评测与测试.md)。 |
| LLM 充当裁判 | LLM-as-Judge | 用另一个（通常更强的）模型来给被测模型的输出打分，解决"开放式输出无法精确比对"的问题。类比：人工评审太贵，就训练一个自动评审员来代劳。详见[工程篇 · 评测与测试](../03-工程篇/13-评测与测试.md)。 |
| 提示注入 | Prompt Injection | 攻击者把恶意指令藏在用户输入或外部内容（网页、文档）里，诱导 Agent 偏离原意或泄露信息。类比：跟 XSS/SQL 注入同源的思路——把"数据"伪装成"指令"。**核心防线：区分可信指令通道与不可信数据。** 详见[工程篇 · 安全与防护](../03-工程篇/16-安全与防护.md)。 |
| 越狱 | Jailbreak | 用各种话术绕过模型的安全限制，让它做本不该做的事。详见[工程篇 · 安全与防护](../03-工程篇/16-安全与防护.md)。 |
| 护栏 | Guardrails | 在输入/输出处加的校验与拦截机制（过滤敏感内容、约束格式、阻断危险动作）。类比：表单校验 + 后端鉴权 + 危险操作二次确认，三层兜底。详见[工程篇 · 安全与防护](../03-工程篇/16-安全与防护.md)。 |
| 人在回路 | Human-in-the-loop | 在执行高风险或不可逆动作（发邮件、删数据、付款）前，先让人确认。类比：危险操作弹个"确认对话框"再放行。详见[工程篇 · 安全与防护](../03-工程篇/16-安全与防护.md)、[核心能力篇 · 工具系统设计](../02-核心能力篇/06-工具系统设计.md)。 |
| 最小权限 | Least Privilege | 只给 Agent 完成任务所必需的最小权限/工具，缩小出事时的爆炸半径。类比：API key 只授予必要 scope。详见[工程篇 · 安全与防护](../03-工程篇/16-安全与防护.md)。 |
| 限流 | Rate Limiting | 厂商对你调用频率/吞吐的上限，常见两个维度：RPM 和 TPM。详见[工程篇 · 成本与性能优化](../03-工程篇/15-成本与性能优化.md)。 |
| RPM | RPM, Requests Per Minute | 每分钟最多发多少个请求。类比：接口的 QPS 限制。 |
| TPM | TPM, Tokens Per Minute | 每分钟最多处理多少 token（输入+输出合计）。类比：带宽上限——不是看请求数，而是看"流量"。 |
| 拒绝 | Refusal | 模型出于安全判断拒绝回答，部分模型会返回 `stop_reason: "refusal"` 并带类别说明。处理建议：先判断 `stop_reason` 再读 `content`，必要时走"回退到另一模型"的策略。详见[工程篇 · 安全与防护](../03-工程篇/16-安全与防护.md)。 |
| 零数据保留 | ZDR, Zero Data Retention | 厂商不保留你请求数据的合规配置；注意部分模型要求一定保留期，ZDR 下会被拒。详见[工程篇 · 安全与防护](../03-工程篇/16-安全与防护.md)。 |

## 速查：最容易混的几对概念

- **思维链（CoT） vs 思考（Thinking）**：CoT 是一种提示技巧（你让它把推理写出来）；Thinking 是模型内置的推理能力/参数（用 `adaptive` 开启）。
- **压缩（Compaction） vs 上下文裁剪（Context Editing）**：前者把旧历史**总结**成摘要，后者把旧内容**直接删**。
- **函数调用 vs 结构化输出**：前者是"模型决定调哪个工具、给什么参数"；后者是"让模型最终答案本身长成指定 JSON"。两者底层都靠 schema 约束，常一起用。
- **短期记忆 vs 长期记忆**：短期=本轮对话历史（在上下文窗口里）；长期=写到外部存储跨会话复用。
- **RAG vs Agentic RAG**：前者固定检索一次；后者由 Agent 自己决定查不查、查几次、怎么查。
- **Trace vs Span vs Trajectory**：Trace 是一整次任务记录，Span 是其中一步，Trajectory 是"决策+行动"的序列（评测时关注）。

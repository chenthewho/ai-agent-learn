# 第 6 章 工具系统设计

> 如果说大模型是 Agent 的"大脑"，那么工具就是它的"手脚"。没有工具，模型只能在脑子里想；有了工具，它才能查数据、改文件、发请求、操作真实世界。第 4 章你学会了一次工具调用的机制，第 5 章把它放进了循环——本章回答一个更工程的问题：**怎么把一整套工具系统设计得好用、好管、好扩展？**

> **学习目标**
> - 掌握"好工具"的设计原则：清晰命名、在描述里写清**何时该调用**、精简的参数 schema。
> - 学会工具粒度的取舍：通用工具（bash / 代码执行）vs 专用工具，以及何时把一个动作"升级"为专用工具。
> - 学会把工具错误**作为 `tool_result` 返回给模型**（带 `is_error`）让它自我修正，而不是抛异常崩掉循环。
> - 理解并行工具调用，记住那个大坑：**所有 `tool_result` 必须放在同一条 user 消息里返回**。
> - 会实现一个可复用的**工具注册表（registry）+ 执行器（dispatcher）**，支持注册、生成 schema、校验、执行、错误捕获。
> - 了解人类确认（human-in-the-loop）、内置/服务端工具、工具太多时的检索方案。

**前置知识**：第 4 章[结构化输出与函数调用](../01-基础篇/04-结构化输出与函数调用.md)（工具调用机制）、第 5 章[Agent 核心循环与推理范式](./05-agent核心循环与推理范式.md)（工具在循环里如何被执行、`is_error` 如何回填）。

## 6.1 回顾：工具是怎么工作的

快速回顾，把第 4、5 章的机制接上。在 Claude 里，一次工具调用涉及三方：

1. **你**在 `tools` 里给模型一份"工具说明书"：每个工具是 `{ name, description, input_schema }`。
2. **模型**决定要用某个工具时，返回一个 `tool_use` 块，里面是工具名和它填好的参数（`input`），响应的 `stop_reason` 是 `tool_use`。
3. **你**的代码执行这个工具，把结果作为 `tool_result` 块（带上对应的 `tool_use_id`）放进下一轮 `user` 消息发回去。

> **前端类比**：工具系统 ≈ 一个**带 schema 校验的 RPC / 插件系统**。`input_schema` 就是 RPC 的参数契约；模型是"调用方"，你的执行器是"服务端"；而工具的 `description` ≈ 你给"插件"写的文档——写得好不好，直接决定模型用不用得对。

机制你已经会了。本章的重点是：**这套"插件系统"怎么设计才好用**——从单个工具的定义，到一整套工具的组织与治理。

## 6.2 好工具的设计原则

模型用不用你的工具、用得对不对，**几乎全靠你写的 `name` 和 `description`**。这一节是本章性价比最高的部分。

### 6.2.1 清晰命名

- **用动词 + 名词，具体到一眼能懂**：`get_weather`、`search_orders`、`send_email`，而不是 `weather`、`query`、`do`。
- **名字本身就传达意图**：`cancel_order` 比 `update_order_status` 更清楚地说明"这是个取消操作"。
- 同一套工具里命名风格统一（都用 `snake_case`，都用动词开头）。

### 6.2.2 描述里写清"何时该调用"——不只是"它能干嘛"

这是最容易被忽视、却最关键的一条。很多人把 `description` 写成功能说明书（"查询天气"），但**现代模型是靠描述来判断"现在这个场景该不该调它"**。所以描述里要包含**触发条件**：

```
✗ 偏弱：  "查询天气。"
✓ 更好：  "查询某个城市的当前天气。当用户问及天气、气温、是否下雨，
          或需要根据天气给出建议（如穿衣、出行）时调用此工具。"
```

差别在于：后者明确告诉模型**什么情况下该伸手**。对那些"用工具偏保守"的新模型，在描述里写明触发条件能显著提升它在该调用时的调用率。

实用清单：

- 写明 **何时该用**（触发场景）和 **何时不该用**（边界），必要时举个反例。
- 说清返回什么、单位是什么（"返回摄氏度温度"）。
- 如果工具有副作用或不可逆（发邮件、删数据），在描述里点明，提醒模型谨慎。

### 6.2.3 参数 schema 要精简

`input_schema` 是标准的 JSON Schema。原则是**够用就好，别堆**：

- 只暴露模型真正需要填的参数。内部用的、能从上下文推出来的，别让模型填。
- 每个参数都写 `description`，固定取值用 `enum`，真正必填的才放进 `required`。
- 别设计成"一个万能工具吃 20 个可选参数"——模型容易填错。宁可拆成几个职责单一的小工具。

```typescript
// ✓ 精简、每个字段有描述、用 enum 约束、只标真正必填的
input_schema: {
  type: "object",
  properties: {
    city: { type: "string", description: "城市名，如 '杭州'" },
    unit: {
      type: "string",
      enum: ["celsius", "fahrenheit"],
      description: "温度单位，默认 celsius",
    },
  },
  required: ["city"],
}
```

### 6.2.4 给个示例（可选但有用）

对参数结构复杂的工具，在 `description` 里附一两个调用示例（"例如：`{ \"city\": \"上海\", \"unit\": \"celsius\" }`"），能进一步降低模型填错参数的概率。这和你给同事写 API 文档时附 example 是一个道理。

## 6.3 工具粒度的取舍：通用 vs 专用

设计工具系统时，一个绕不开的决策是：**给模型一把"万能瑞士军刀"，还是一堆"专用螺丝刀"？**

### 两个极端

- **通用工具**：典型代表是 `bash`（执行 shell 命令）或代码执行工具。一个工具几乎能干任何事——读文件、调 API、跑脚本……**强大无比，但难管控**。你的代码收到的只是一个不透明的命令字符串，没法针对具体动作做校验、审批或并行调度。

- **专用工具**：`read_file`、`send_email`、`create_order` 这种一个工具干一件事。**可校验、可审批、可并行**——因为你的代码拿到的是带类型的具体参数，能精准地拦截、渲染、审计。

### 为什么粒度重要：模型发指令，你的"壳"来执行

关键在于这个分工：**模型只负责发出工具调用，真正执行的是你的代码（harness）**。模型不知道你的安全边界、审批策略、UI 长什么样——这些全靠你在执行那一侧把控。而**工具调用的"形状"决定了你的壳能做什么**：

- `bash` 给你的是同一个形状的命令字符串。一个删文件的 `rm` 和一个只读的 `ls`，在你眼里都是"一段 bash"，你**没法只拦截危险的那个**。
- 把"发邮件"做成专用的 `send_email(to, subject, body)`，你就能在执行前轻松弹个确认框；而 `bash -c "curl -X POST ..."` 你根本拦不住。

### 什么时候把一个动作"升级"为专用工具

经验法则：**用 `bash` / 代码执行打底求广度，遇到下面这些信号就把对应动作升级成专用工具**：

| 信号 | 为什么需要专用工具 |
| --- | --- |
| **安全边界**：危险、不可逆的操作（删除、付款、发邮件、改线上配置） | 专用工具才能在执行前做审批/确认；不可逆性是判据——越难撤销越该专用化 |
| **需要校验**：写文件前要检查"文件自上次读取后有没有被改过" | 专用 `edit` 工具能强制这个不变量，`bash` 没法保证 |
| **需要 UI 渲染**：某动作要弹窗、展示选项、阻塞等用户输入 | 把"向用户提问"做成专用工具，才能在前端渲染成模态框（Claude Code 就是这么做的） |
| **需要并行**：只读操作（搜索、读文件）想安全地并发执行 | 专用工具可标记为"并行安全"，调度器才能放心并发；`bash` 里分不清 `grep`（可并行）和 `git push`（不可并行），只能串行 |

一句话：**先用通用工具铺广度，按"需不需要拦截/渲染/审计/并行"逐步把高价值动作专用化。** 这是设计工具系统时反复要做的判断。

## 6.4 输入校验与错误处理：把错误还给模型

这是工具系统里**最能体现"为 Agent 而设计"思维**的一点。

普通函数出错了，你会 `throw`。但在 Agent 循环里，如果工具执行一抛异常、把循环崩掉，那就太可惜了——**模型其实有能力根据错误信息自我修正**：参数填错了？换个参数重试。城市名拼错了？改个拼写。所以正确的做法是：

> **把工具错误捕获下来，作为一个带 `is_error: true` 的 `tool_result` 返回给模型，让它看到错误、自己想办法补救——而不是抛异常崩掉整个循环。**

这相当于把"异常"变成了模型对话里的一条"反馈"。模型看到"错误：城市 'Hanzhou' 未找到，请提供有效城市名"，下一轮就可能改成 "杭州" 重试。

### TypeScript

```typescript
import Anthropic from "@anthropic-ai/sdk";

// 在循环里执行工具，把异常转成带 is_error 的 tool_result
function safeExecute(
  toolUseId: string,
  name: string,
  input: any,
): Anthropic.ToolResultBlockParam {
  try {
    const result = executeTool(name, input); // 你的实际执行逻辑
    return { type: "tool_result", tool_use_id: toolUseId, content: result };
  } catch (err) {
    // 关键：不 rethrow，而是把错误信息作为 tool_result 回给模型
    const message = err instanceof Error ? err.message : String(err);
    return {
      type: "tool_result",
      tool_use_id: toolUseId,
      content: `错误：${message}。请检查参数后重试，或换一种方式。`,
      is_error: true, // ← 告诉模型这是一次失败的调用
    };
  }
}
```

### Python

```python
def safe_execute(tool_use_id: str, name: str, tool_input: dict) -> dict:
    try:
        result = execute_tool(name, tool_input)  # 你的实际执行逻辑
        return {"type": "tool_result", "tool_use_id": tool_use_id, "content": result}
    except Exception as err:
        # 关键：不 re-raise，把错误信息作为 tool_result 回给模型
        return {
            "type": "tool_result",
            "tool_use_id": tool_use_id,
            "content": f"错误：{err}。请检查参数后重试，或换一种方式。",
            "is_error": True,  # ← 告诉模型这是一次失败的调用
        }
```

> **错误信息要写给模型看**：别只回 `"Error"` 或一串堆栈。写清**哪里错了、可以怎么改**（"城市 'xxx' 不存在，请提供中国大陆城市名"），模型才补救得动。
>
> **但不是所有错误都该还给模型**：鉴权失败、配置缺失这类"模型改参数也没用"的系统性错误，该抛就抛、该告警就告警，别让模型在那里徒劳重试。区分"模型能修的"和"模型修不了的"。

## 6.5 并行工具调用：那个一定要记住的坑

模型一次回复里**可以返回多个 `tool_use` 块**——比如它判断"查北京和上海的天气互不依赖"，就会在同一轮一口气发出两个 `get_weather` 调用，让你并行执行。这能省往返、降延迟，是个好东西。

并行执行本身很直观：把多个 `tool_use` 并发跑掉（`Promise.all` / `asyncio.gather`）。但有一个**极易踩、且后果隐蔽的坑**：

> **⚠️ 所有 `tool_result` 必须放在「同一条」`user` 消息里返回。**

也就是说：模型一轮发了 N 个 `tool_use`，你执行完后，要把这 N 个 `tool_result` 块**全部塞进一条** `user` 消息的 `content` 数组里发回去——**绝不能拆成 N 条 user 消息分开发**。

**为什么这是坑？** 因为拆开发通常**不会立刻报错**，循环照跑——但模型会"学到"一个教训：并行调用会导致结果被拆散、对不上。于是**它以后就不敢再并行调用了**，退化成一次只调一个工具，你的延迟优化全没了。这种"慢性病"很难定位，所以必须从一开始就写对。

### TypeScript

```typescript
// 模型这一轮返回了多个 tool_use，并发执行后，结果合并进【一条】 user 消息
const toolUseBlocks = response.content.filter((b) => b.type === "tool_use");

// 并发执行（只读、互不依赖的工具适合并行）
const toolResults = await Promise.all(
  toolUseBlocks.map(async (block) => {
    if (block.type !== "tool_use") return null;
    const result = await executeToolAsync(block.name, block.input);
    return {
      type: "tool_result" as const,
      tool_use_id: block.id,
      content: result,
    };
  }),
);

// ✓ 正确：所有结果放进同一条 user 消息
messages.push({
  role: "user",
  content: toolResults.filter((r) => r !== null),
});

// ✗ 错误示范（别这么干）：拆成多条 user 消息会让模型以后不敢并行
// for (const r of toolResults) messages.push({ role: "user", content: [r] });
```

### Python

```python
import asyncio

# 模型这一轮返回了多个 tool_use，并发执行后，结果合并进【一条】 user 消息
tool_use_blocks = [b for b in response.content if b.type == "tool_use"]


async def run_one(block):
    result = await execute_tool_async(block.name, block.input)
    return {"type": "tool_result", "tool_use_id": block.id, "content": result}


# 并发执行
tool_results = await asyncio.gather(*[run_one(b) for b in tool_use_blocks])

# ✓ 正确：所有结果放进同一条 user 消息
messages.append({"role": "user", "content": list(tool_results)})

# ✗ 错误示范（别这么干）：拆成多条 user 消息
# for r in tool_results:
#     messages.append({"role": "user", "content": [r]})
```

> 如果某个并行工具失败了，**也要为它返回一个 `tool_result`（带 `is_error: true`），不能直接丢掉**——少一个 `tool_result`，模型就会发现某个 `tool_use_id` 没有对应结果，从而困惑或报错。"每个 `tool_use` 都要有一个 `tool_result` 对应"是铁律。
>
> 想强制模型一次只调一个工具？在 `tool_choice` 里加 `disable_parallel_tool_use: true` 即可。但默认开着并行通常是好事。

## 6.6 人类确认（Human-in-the-Loop）

有些操作错了就回不来：删库、付款、给客户群发邮件、改线上配置。对这类**危险或不可逆**的工具，执行前应该**插入一道人类审批门**——模型说要做，但真正执行前停下来，等人点"批准"或"拒绝"。

实现思路（呼应第 5 章的循环和 6.3 的"专用工具"）：在执行工具**之前**，判断这个工具是否需要确认；需要的话就暂停循环、把"待确认的操作 + 参数"抛给上层（前端弹窗、CLI 提示、审批系统），拿到结果再决定执行还是把"用户拒绝了"作为 `tool_result`（带 `is_error`）回给模型。

### TypeScript

```typescript
// 标记哪些工具需要人类确认（危险/不可逆操作）
const NEEDS_APPROVAL = new Set(["send_email", "delete_order", "make_payment"]);

async function executeWithApproval(
  block: Anthropic.ToolUseBlock,
  // 由上层注入：弹窗/CLI/审批系统，返回用户是否批准
  askUser: (name: string, input: any) => Promise<boolean>,
): Promise<Anthropic.ToolResultBlockParam> {
  if (NEEDS_APPROVAL.has(block.name)) {
    const approved = await askUser(block.name, block.input); // ← 阻塞等待人类决定
    if (!approved) {
      // 用户拒绝：作为 tool_result 回给模型，让它换方案，而不是硬执行
      return {
        type: "tool_result",
        tool_use_id: block.id,
        content: `用户拒绝了操作 ${block.name}。请询问用户或改用其他方式。`,
        is_error: true,
      };
    }
  }
  const result = await executeToolAsync(block.name, block.input);
  return { type: "tool_result", tool_use_id: block.id, content: result };
}
```

### Python

```python
NEEDS_APPROVAL = {"send_email", "delete_order", "make_payment"}


async def execute_with_approval(block, ask_user) -> dict:
    # ask_user(name, input) 由上层注入，返回 bool 表示是否批准
    if block.name in NEEDS_APPROVAL:
        approved = await ask_user(block.name, block.input)  # 阻塞等待人类决定
        if not approved:
            return {
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": f"用户拒绝了操作 {block.name}。请询问用户或改用其他方式。",
                "is_error": True,
            }
    result = await execute_tool_async(block.name, block.input)
    return {"type": "tool_result", "tool_use_id": block.id, "content": result}
```

> 把"拒绝"作为 `tool_result` 回给模型（而不是直接抛错），模型就能优雅地换个方式或向用户追问——这又是 6.4"把反馈还给模型"思想的应用。前端工程师在这里有天然优势：把审批门做成一个漂亮的确认弹窗、带上"模型想发给谁、内容是什么"的预览，体验远胜后端的一行命令行确认。

## 6.7 内置 / 服务端工具简述

除了你自己定义的工具（**客户端工具**，由你的代码执行），模型厂商还提供一些**内置工具**。它们最重要的区分是**谁来执行**：

- **客户端工具（client-side）**：工具的"说明书"可能是厂商定义的（比如 `bash`、文本编辑器有固定的 schema），但**执行在你这边**——模型返回 `tool_use`，你的代码去跑、把 `tool_result` 回填。本章前面写的全是这类。
- **服务端工具（server-side）**：**整个在厂商的基础设施上执行**——你只要在 `tools` 里声明它，模型自己调用、自己拿结果，无需你写执行循环。

Claude 提供的一些工具（具体可用性与版本以官方文档为准）：

| 工具 | 执行方 | 用途 |
| --- | --- | --- |
| **代码执行（code execution）** | 服务端 | 在沙箱里跑 Python，做计算、数据分析、生成图表/文件 |
| **网页搜索 / 抓取（web search/fetch）** | 服务端 | 让模型查训练截止之后的信息，带引用返回 |
| **文件编辑（text editor）** | 客户端 | 模型发出查看/创建/替换文件的指令，你的代码执行 |
| **bash** | 客户端 | 模型发出 shell 命令，你的代码在受控环境执行 |
| **记忆（memory）** | 客户端 | 模型读写一个"记忆目录"，跨会话保存信息（你实现存储后端） |

> 关键区别再强调一遍：**服务端工具你只声明、不执行**（如代码执行、网页搜索）；**客户端工具你既声明又执行**（如 bash、文件编辑、记忆）。服务端工具因为在厂商那边跑一个内部循环，可能返回 `pause_turn` 需要你续跑——这正是第 5 章 5.4.4 讲的内容。记忆工具与第 7 章[记忆与上下文管理](./07-记忆与上下文管理.md)直接相关。

## 6.8 双语代码：工具注册表 + 执行器

前面零散的概念，现在收束成一个**可复用的基础设施**：把"工具的定义、schema 生成、参数校验、执行、错误捕获"统一管起来。这个 **注册表（Registry）+ 执行器（Dispatcher）** 会被本书后续的实战项目反复复用——所以值得写扎实一点。

设计目标：

- **注册**：用一个 `register()` 把"工具元信息 + 实现函数"登记进去。
- **生成 schema**：一键导出所有工具的 `{ name, description, input_schema }` 列表，直接喂给模型的 `tools` 参数。
- **校验**：执行前用 JSON Schema 校验模型传来的参数（少了必填项、类型不对，提前拦下）。
- **执行 + 错误捕获**：调用对应实现；任何异常（含校验失败）都转成带 `is_error` 的 `tool_result`，绝不崩循环。
- **元信息**：每个工具可带 `needsApproval`（是否要人类确认）、`readOnly`（是否并行安全）等标记，供上层调度。

### TypeScript

用 [Zod](https://zod.dev/) 定义参数 schema（前端工程师大多熟悉），再转成 JSON Schema 给模型。

```typescript
import { z } from "zod";
import { zodToJsonSchema } from "zod-to-json-schema";
import type Anthropic from "@anthropic-ai/sdk";

// 单个工具的定义：元信息 + Zod schema + 实现函数
interface ToolDef<T extends z.ZodTypeAny = z.ZodTypeAny> {
  name: string;
  description: string;
  schema: T; // Zod schema，既用于生成 JSON Schema，也用于运行时校验
  execute: (input: z.infer<T>) => Promise<string> | string;
  needsApproval?: boolean; // 危险操作，执行前需人类确认
  readOnly?: boolean; // 只读 → 可安全并行
}

class ToolRegistry {
  private tools = new Map<string, ToolDef>();

  // 注册一个工具
  register<T extends z.ZodTypeAny>(def: ToolDef<T>): this {
    this.tools.set(def.name, def as unknown as ToolDef);
    return this; // 支持链式注册
  }

  // 导出给模型的 tools 列表（Zod → JSON Schema）
  toAnthropicTools(): Anthropic.Tool[] {
    return [...this.tools.values()].map((t) => ({
      name: t.name,
      description: t.description,
      input_schema: zodToJsonSchema(t.schema, {
        target: "openApi3",
      }) as Anthropic.Tool["input_schema"],
    }));
  }

  // 某个工具是否需要人类确认 / 是否只读 —— 供上层调度
  needsApproval(name: string): boolean {
    return this.tools.get(name)?.needsApproval ?? false;
  }
  isReadOnly(name: string): boolean {
    return this.tools.get(name)?.readOnly ?? false;
  }

  // 执行一个工具调用：校验 → 执行 → 任何错误都转成 is_error 的 tool_result
  async dispatch(
    toolUseId: string,
    name: string,
    rawInput: unknown,
  ): Promise<Anthropic.ToolResultBlockParam> {
    const tool = this.tools.get(name);
    if (!tool) {
      return this.errorResult(toolUseId, `未知工具：${name}`);
    }
    // 1. 校验参数
    const parsed = tool.schema.safeParse(rawInput);
    if (!parsed.success) {
      return this.errorResult(
        toolUseId,
        `参数校验失败：${parsed.error.issues.map((i) => i.message).join("; ")}。请修正后重试。`,
      );
    }
    // 2. 执行 + 捕获异常
    try {
      const result = await tool.execute(parsed.data);
      return { type: "tool_result", tool_use_id: toolUseId, content: result };
    } catch (err) {
      const msg = err instanceof Error ? err.message : String(err);
      return this.errorResult(toolUseId, `执行出错：${msg}。请检查参数或换种方式。`);
    }
  }

  private errorResult(
    toolUseId: string,
    content: string,
  ): Anthropic.ToolResultBlockParam {
    return { type: "tool_result", tool_use_id: toolUseId, content, is_error: true };
  }
}

// ——— 用法：注册几个工具 ———
const registry = new ToolRegistry()
  .register({
    name: "get_weather",
    description:
      "查询某城市当前天气。当用户问及天气、气温、是否下雨，或需要据此给出建议时调用。",
    schema: z.object({ city: z.string().describe("城市名，如 '杭州'") }),
    readOnly: true, // 只读，可并行
    execute: ({ city }) => JSON.stringify({ city, temp_c: 12, condition: "小雨" }),
  })
  .register({
    name: "send_email",
    description: "给指定收件人发送邮件。这是不可逆操作，发送前需用户确认。",
    schema: z.object({
      to: z.string().describe("收件人邮箱"),
      subject: z.string(),
      body: z.string(),
    }),
    needsApproval: true, // 危险操作
    execute: async ({ to, subject }) => {
      // 真实场景：调邮件服务
      return JSON.stringify({ sent: true, to, subject });
    },
  });

// 喂给模型的 tools 参数
const tools = registry.toAnthropicTools();
// 在第 5 章的循环里：const result = await registry.dispatch(block.id, block.name, block.input);
```

### Python

用 [Pydantic](https://docs.pydantic.dev/) 定义参数模型（Python 生态主流），它能直接导出 JSON Schema，也能做运行时校验。

```python
import json
from typing import Callable, Any
from pydantic import BaseModel, ValidationError


class ToolDef(BaseModel):
    name: str
    description: str
    params_model: type[BaseModel]            # Pydantic 模型：生成 schema + 运行时校验
    execute: Callable[[BaseModel], str]      # 实现函数，入参是校验后的模型实例
    needs_approval: bool = False             # 危险操作，需人类确认
    read_only: bool = False                  # 只读 → 可并行

    model_config = {"arbitrary_types_allowed": True}


class ToolRegistry:
    def __init__(self) -> None:
        self._tools: dict[str, ToolDef] = {}

    def register(self, tool: ToolDef) -> "ToolRegistry":
        self._tools[tool.name] = tool
        return self  # 支持链式

    def to_anthropic_tools(self) -> list[dict]:
        """导出给模型的 tools 列表（Pydantic → JSON Schema）。"""
        return [
            {
                "name": t.name,
                "description": t.description,
                "input_schema": t.params_model.model_json_schema(),
            }
            for t in self._tools.values()
        ]

    def needs_approval(self, name: str) -> bool:
        tool = self._tools.get(name)
        return tool.needs_approval if tool else False

    def is_read_only(self, name: str) -> bool:
        tool = self._tools.get(name)
        return tool.read_only if tool else False

    def dispatch(self, tool_use_id: str, name: str, raw_input: dict) -> dict:
        """执行一个工具调用：校验 → 执行 → 任何错误都转成 is_error 的 tool_result。"""
        tool = self._tools.get(name)
        if tool is None:
            return self._error(tool_use_id, f"未知工具：{name}")

        # 1. 校验参数
        try:
            params = tool.params_model.model_validate(raw_input)
        except ValidationError as e:
            msgs = "; ".join(err["msg"] for err in e.errors())
            return self._error(tool_use_id, f"参数校验失败：{msgs}。请修正后重试。")

        # 2. 执行 + 捕获异常
        try:
            result = tool.execute(params)
            return {"type": "tool_result", "tool_use_id": tool_use_id, "content": result}
        except Exception as err:
            return self._error(tool_use_id, f"执行出错：{err}。请检查参数或换种方式。")

    @staticmethod
    def _error(tool_use_id: str, content: str) -> dict:
        return {
            "type": "tool_result",
            "tool_use_id": tool_use_id,
            "content": content,
            "is_error": True,
        }


# ——— 用法：定义参数模型并注册工具 ———
from pydantic import Field


class WeatherParams(BaseModel):
    city: str = Field(description="城市名，如 '杭州'")


class EmailParams(BaseModel):
    to: str = Field(description="收件人邮箱")
    subject: str
    body: str


registry = (
    ToolRegistry()
    .register(ToolDef(
        name="get_weather",
        description="查询某城市当前天气。当用户问及天气、气温、是否下雨，或需要据此给出建议时调用。",
        params_model=WeatherParams,
        read_only=True,  # 只读，可并行
        execute=lambda p: json.dumps({"city": p.city, "temp_c": 12, "condition": "小雨"}),
    ))
    .register(ToolDef(
        name="send_email",
        description="给指定收件人发送邮件。这是不可逆操作，发送前需用户确认。",
        params_model=EmailParams,
        needs_approval=True,  # 危险操作
        execute=lambda p: json.dumps({"sent": True, "to": p.to, "subject": p.subject}),
    ))
)

tools = registry.to_anthropic_tools()
# 在第 5 章的循环里：result = registry.dispatch(block.id, block.name, block.input)
```

### 把注册表接进第 5 章的循环

有了注册表，第 5 章那个手写循环里"执行工具"的部分就被收敛成一行 `registry.dispatch(...)`，并且天然支持校验、错误捕获、并行调度和审批判断：

```python
# 第 5 章循环的「行动」环节，现在长这样（Python 示意）
if response.stop_reason == "tool_use":
    tool_results = []
    for block in response.content:
        if block.type == "tool_use":
            # 需要确认的工具，先走审批（6.6）
            if registry.needs_approval(block.name) and not ask_user(block.name, block.input):
                tool_results.append(registry._error(block.id, f"用户拒绝了 {block.name}"))
                continue
            # 校验 + 执行 + 错误捕获，全在 dispatch 里搞定
            tool_results.append(registry.dispatch(block.id, block.name, block.input))
    # 所有结果放进「一条」 user 消息（6.5 的铁律）
    messages.append({"role": "user", "content": tool_results})
    continue
```

这就是工具系统设计的价值：**循环骨架保持干净，所有"脏活"（校验、容错、审批、并行）被收进可复用的注册表里。** 后续每个实战项目，你都可以直接拎这个 `ToolRegistry` 去用，只需 `register()` 新工具。

## 6.9 工具太多怎么办：工具检索与按需加载

随着 Agent 能力变强，工具会越来越多。但工具不是越多越好——**几十上百个工具的 schema 全塞进每次请求，有两个坏处**：

1. **占上下文、烧钱**：每个工具的 `description` + `input_schema` 都是 token，工具一多，光"工具说明书"就吃掉大量上下文。
2. **降低模型判断力**：候选太多，模型更容易选错工具。

应对思路是**别把所有工具一次性全给模型，而是按需加载**：

- **工具检索（tool search）**：先只给模型"工具目录"，让它根据当前任务**检索**出最相关的少数几个工具，再把这几个的完整 schema 加载进来。Claude 提供了内置的工具检索能力（声明一个 `tool_search` 类型的工具，并把其余工具标记为 `defer_loading: true` 延迟加载），模型会自己搜出需要的工具——而且是**追加** schema 而非替换，对提示缓存友好。
- **分组 / 分阶段**：按任务阶段只挂载当前需要的工具子集（比如"调研阶段"只给搜索类工具，"写作阶段"只给文件类工具）。

> 当工具来自外部、需要跨应用/跨团队复用时，还有一套标准协议来解决"工具怎么被发现、被接入"——那就是 **MCP（Model Context Protocol，模型上下文协议）**，它把工具/数据源标准化成可插拔的"服务端"，是第 11 章[MCP 与工具生态](../03-工程篇/11-mcp与工具生态.md)的主题。本章你只要知道：**工具多到一定程度，就该从"手写一堆工具"转向"检索 + 标准化接入"**。

## 6.10 前端视角：工具系统就是一个插件系统

把本章串起来，用前端经验落地：

- **工具系统 ≈ 带 schema 校验的 RPC / 插件系统。** `input_schema`（JSON Schema）就是接口契约——和你用 TypeScript 类型、Zod、或 OpenAPI 定义后端接口契约是同一回事。模型是调用方，你的 `dispatch` 是路由 + 执行。

- **工具描述 ≈ 给"插件"写的文档。** 一个文档写得烂的 npm 包没人会用对；一个 `description` 写得烂的工具，模型也用不对。"写清何时该调用"就像在 README 里写清"什么场景下该装这个包"。

- **注册表 ≈ 插件注册中心 / 路由表。** `registry.register()` 像往插件系统里注册一个插件；`dispatch()` 像一个带中间件（校验、错误捕获、审批）的路由。你在前端搭过的那套"注册 + 分发 + 中间件"心智，直接迁移过来。

- **`is_error` 回填 ≈ 优雅降级。** 前端不会因为一个组件渲染失败就白屏，而是显示个 fallback。Agent 也一样——一个工具失败了，把错误"渲染"给模型（`is_error`）让它补救，而不是让整个循环崩掉。

- **human-in-the-loop ≈ 二次确认弹窗。** "确定要删除吗？"这种确认交互你做过无数次，把它接到 Agent 的危险工具上即可——而且你能做得比命令行确认漂亮得多。

## 常见坑 / 最佳实践

- **`description` 只写功能、不写"何时调用"。** 这是工具不被正确调用的头号原因。务必写明触发场景和边界。
- **参数 schema 过度膨胀。** 一个工具吃十几个可选参数，模型必然填错。拆成职责单一的小工具。
- **工具出错就抛异常崩循环。** 正确做法是捕获后作为带 `is_error: true` 的 `tool_result` 回给模型，让它自我修正——但鉴权/配置类"改参数也没用"的错误除外。
- **并行工具结果拆成多条 user 消息发。** 本章最重要的坑：必须放在**同一条** user 消息里。拆开不会立刻报错，但会让模型以后不敢并行，悄悄拖慢你的 Agent。
- **并行时丢掉失败工具的 `tool_result`。** 每个 `tool_use` 都必须有对应的 `tool_result`（失败的也要回，带 `is_error`），少一个模型就困惑。
- **危险操作不设审批门。** 删除、付款、发邮件这类不可逆动作，执行前必须能人类确认，且把"拒绝"作为 `tool_result` 回给模型。
- **分不清客户端工具和服务端工具。** 服务端工具（代码执行、网页搜索）你只声明不执行，且可能返回 `pause_turn` 要续跑；客户端工具你既声明又执行。混淆会导致循环逻辑写错。
- **工具堆到几十上百个还全量塞给模型。** 占上下文、烧钱、降判断力。用工具检索 / 按需加载，或上 MCP。
- **每个项目都重写一遍工具执行逻辑。** 把校验、容错、审批收进一个可复用的 `ToolRegistry`，后续项目直接复用。

## 本章小结

1. **工具是 Agent 的手脚**，本章讲的是怎么把这套"带 schema 校验的 RPC / 插件系统"设计好。
2. **好工具靠描述**：清晰命名、在 `description` 里写清"何时该调用"、参数 schema 精简、必要时给示例——这是性价比最高的投入。
3. **工具粒度要权衡**：用通用工具（bash / 代码执行）求广度，遇到安全边界、需校验、需 UI、需并行的动作就升级为专用工具。
4. **把错误还给模型**：工具失败时返回带 `is_error: true` 的 `tool_result`，让模型自我修正，而不是抛异常崩循环。
5. **并行工具调用的铁律**：所有 `tool_result` 放在同一条 user 消息里返回，且每个 `tool_use` 都要有对应结果——否则模型会退化为不敢并行。
6. **危险操作要有人类确认**；区分客户端工具与服务端工具；工具太多时用工具检索 / 按需加载，并预告 MCP。
7. **可复用的 `ToolRegistry` + `dispatch`** 把注册、schema 生成、校验、执行、容错、审批统一收口，让循环骨架保持干净，后续项目直接复用。

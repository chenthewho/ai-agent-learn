## 2.1 Token 与分词:模型眼里的"字"不是你眼里的字

我们说 LLM "预测下一个 token"。那 **token 到底是个啥?**

token 是模型处理文本的**最小单位**。它**不等于一个字符,也不等于一个单词**,而是介于两者之间的"词块"。模型在训练前,会用一个叫**分词器(tokenizer)**的东西,把所有文本按一套统计规则切成 token,每个 token 对应词表里的一个编号。模型实际看到的、吐出来的,全是这些编号。

### 直观感受一下

英文里,常见词往往就是一个 token,生僻词或长词会被切成几块:

```
"hello"        → 1 个 token
"tokenization" → 可能切成 "token" + "ization",2 个 token
"ChatGPT"      → 可能切成 "Chat" + "G" + "PT",3 个 token
```

一个粗略的经验:**英文大约 4 个字符 ≈ 1 个 token**,或者说 **100 个英文单词 ≈ 130~150 个 token**。

### 中英文 token 差异:中文更"贵"

这里有个对中文开发者非常重要的点。**同样信息量的中文,通常比英文消耗更多 token。**

```
英文: "Hello, how are you?"        → 大约 6 个 token
中文: "你好,你最近怎么样?"          → 大约 10~15 个 token(取决于分词器)
```

为什么?因为主流模型的分词器主要在英文语料上训练,对中文的切分往往更碎——很多时候一个汉字甚至要占 1~2 个 token,而一个英文单词可能就 1 个 token。**结果就是:处理同样一段意思,中文往往更费 token,也就更费钱、更占上下文。** 这一点在你做中文应用时要心里有数。

> 不同厂商、不同模型的分词器都不一样,所以**同一段文字在不同模型上的 token 数会不同**。别想着背一个固定换算比例。

### 为什么按 token 计费?

因为 token 数直接反映了模型的**计算量**——模型每生成一个 token 都要做一次完整的前向计算,处理你的输入也要逐 token 编码。所以厂商按 token 收费天经地义。而且:

> **输入 token 和输出 token 通常分开计价,且输出更贵。** 以本书默认参考的 Claude Opus 4.8 为例:输入约 \$5 / 百万 token,输出约 \$25 / 百万 token(以官方文档为准)。输出比输入贵好几倍,是因为生成(逐 token 自回归)比读入更耗算力。

这条对 Agent 影响巨大:Agent 反复把历史塞回去当输入,输入 token 会滚雪球(下一节和 2.3 节细说)。

### 怎么估算 token 数?

需要**精确**算 token(比如做成本预估、判断会不会超上下文)时,有两条原则:

1. **粗估**:中文按"字数 × 1.5 ~ 2"、英文按"字符数 ÷ 4"心算个量级就行。
2. **精确**:**用每家模型自己的计数方式,别拿一家的工具去估另一家。**

> ⚠️ **一个常见错误:别用 `tiktoken` 去估 Claude 的 token。** `tiktoken` 是 OpenAI 的分词器,拿它算 Claude 会偏差很大(对中文、对代码尤其离谱)。要精确数 Claude 的 token,用 Anthropic 官方的计数接口(`POST /v1/messages/count_tokens`,SDK 里是 `client.messages.count_tokens(...)` / `client.messages.countTokens(...)`);要数 OpenAI 的就用 OpenAI 的工具。**各家用各家的。**

```ts
#### TypeScript
// 用 Anthropic 官方 SDK 精确数 Claude 的 token(别用 tiktoken!)
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic(); // 自动读环境变量 ANTHROPIC_API_KEY

const { input_tokens } = await client.messages.countTokens({
  model: "claude-opus-4-8",
  messages: [{ role: "user", content: "你好,帮我数一下这段话有多少 token" }],
});
console.log(`这段输入大约 ${input_tokens} 个 token`);
```

```python
#### Python
# 用 Anthropic 官方 SDK 精确数 Claude 的 token(别用 tiktoken!)
import anthropic

client = anthropic.Anthropic()  # 自动读环境变量 ANTHROPIC_API_KEY

resp = client.messages.count_tokens(
    model="claude-opus-4-8",
    messages=[{"role": "user", "content": "你好,帮我数一下这段话有多少 token"}],
)
print(f"这段输入大约 {resp.input_tokens} 个 token")
```

## 2.2 上下文窗口:一次请求能带的"payload 上限"

模型不是有无限记性的。**它一次能"看到"的 token 总量是有上限的,这个上限就叫上下文窗口(context window)。**

这个窗口装的是**这一次请求里的所有东西**:系统提示 + 全部对话历史 + 你这轮的输入 + 它即将生成的输出,全都算在里面。

### 前端类比:它像一次请求的 payload 上限

你调后端接口,body 太大会被拒(`413 Request Too Large`)。上下文窗口就是 LLM 这次请求能携带的"payload 上限",只不过单位是 token,而且这个上限里**还要预留出给模型回答用的空间**。

```
┌──────────────── 上下文窗口(比如 200K / 1M tokens)─────────────────┐
│                                                                    │
│  [系统提示]  [对话历史:user/assistant/user/assistant...]  [本轮输入] │  ← 输入
│                                                                    │
│  ......................[预留给模型生成的输出].......................  │  ← 输出
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
        输入 + 输出 的 token 总和,不能超过这个窗口
```

### 当前主流有多大?

近两年窗口暴涨。当前主流模型大致在 **200K 到 1M token** 这个区间:

- 大窗口模型(如 Claude Opus 4.8 / Sonnet 4.6)可达 **1M(一百万)token** 的上下文——大约能塞下一整本书还有富余。
- 较小的快速模型(如 Claude Haiku 4.5)通常在 **200K** 左右。
- (具体数字以各家官方文档为准,变化很快。)

> 1M token 听起来很大,但 Agent 跑长任务时,几十轮工具调用的历史累积起来,照样能把它撑爆。**"窗口大"不等于"可以乱塞"**——塞得越满越慢越贵。上下文管理是门学问,[核心能力篇有专章](../02-核心能力篇/)。

### 超出窗口会发生什么?

两种情况:

1. **请求时就超了**(输入已经太长):API 直接报错拒绝(类似 `400` / `request too large`)。这时你得**主动处理**——裁剪历史、做摘要、或分块,而**不要悄悄把内容截断**(截断会丢信息、让模型答得莫名其妙)。
2. **生成中逼近窗口**:有的模型会返回一个"上下文超限"的停止原因,告诉你它是因为撑到头了才停的,而不是正常答完。

这就引出了下一个、也是对前端最关键的概念。

## 2.3 消息结构与角色,以及最重要的一件事:API 是无状态的

### 消息结构与三种角色

现代对话式模型的输入,是一个**消息数组**,每条消息有个**角色(role)**:

| 角色 | 谁说的 | 作用 |
|---|---|---|
| `system` | 开发者(你) | 设定模型的身份、行为准则、规则。整段对话的"基调"。 |
| `user` | 用户 | 用户的输入、提问、指令。 |
| `assistant` | 模型 | 模型之前的回答。多轮对话里,把它过去的回复也放进来,它才记得说过啥。 |

一个三轮对话的消息数组长这样:

```ts
const messages = [
  { role: "system",    content: "你是一个简洁专业的编程助手。" },
  { role: "user",      content: "什么是闭包?" },
  { role: "assistant", content: "闭包是函数与其词法作用域的组合……" }, // 模型上一轮的回答
  { role: "user",      content: "用一个例子说明" },                    // 用户这一轮的新问题
];
```

### ⚠️ 最重要的事实:LLM API 是无状态的(stateless)

**这是前端工程师转 Agent 开发时,最容易栽跟头、也最必须记牢的一点。**

> **模型的 `/messages` 接口是无状态的。服务器不替你记任何对话历史。每一轮请求,你都必须把"从头到现在的完整对话历史"重新发过去。**

这意味着:模型并不会"记得"你们上一轮聊了什么。它之所以"记得",纯粹是因为**你把历史又发了一遍**。你不发,它就一问三不知。

#### 前端类比:你早就干过这事了

这个模式你其实无比熟悉。想想前端里你怎么管状态:

- 你在前端维护一个 `state`(比如一个 `messages` 数组);
- 每次要请求时,你把需要的 state 拼进 request body,发给后端;
- 后端(无状态地)处理完返回;
- 你再把结果更新回 `state`。

LLM API 一模一样:

```
你维护 messages 数组(state)
   │
   ▼ 每轮把完整 messages 发过去(拼 payload)
LLM API(无状态地处理,不记任何东西)
   │
   ▼ 返回这一轮的回答
你把回答 push 回 messages 数组(更新 state)
   │
   └──────── 下一轮,再发一次完整的 messages ────────┐
                                                    ▼
```

所以上一章那个 Agent 伪代码里,我们为什么要不停地 `messages.push(...)`?**因为我们在前端(客户端)亲手维护着这份"记忆",每轮重新发给那个失忆的服务器。** 把这条想透,你就理解了 Agent "记忆"的本质:它不神秘,就是你在管理 state。

#### 这条事实的两个直接推论

1. **对话越长,每轮请求越贵、越慢。** 因为你每轮都把越来越长的历史当输入重新发一遍——输入 token 线性增长。这正是 Agent 烧钱的根源,也是 2.7 节"提示缓存能省钱"和上下文管理要解决的问题。
2. **历史是你的责任。** 要不要裁剪、要不要摘要、要不要持久化到数据库以便下次会话接着聊——全是你客户端的事,服务器不管。

> 小提醒:`user` / `assistant` 通常要交替出现,且第一条一般是 `user`(`system` 不算在内,单独放最前)。具体规则各家略有差异,**以官方文档为准**。

## 2.4 采样与生成控制:让模型答得更确定还是更发散

模型每一步预测的,其实是"下一个 token 的**概率分布**"——比如 "今天天气" 后面,"很"占 30%、"不错"占 20%、"晴朗"占 15%……。到底挑哪个?这就是**采样(sampling)**。

经典的两个控制参数:

- **`temperature`(温度)**:控制随机性。
  - 低(接近 0):每次都挑概率最高的,输出**更确定、更稳定、更保守**。适合抽取、分类、代码生成等要"靠谱"的任务。
  - 高(接近 1 或更高):允许挑那些概率没那么高的 token,输出**更发散、更有创意、更随机**。适合头脑风暴、创意写作。
- **`top_p`(核采样)**:另一种控制方式,只在"累积概率达到 p 的那一小撮候选"里采样。和 `temperature` 作用类似。

经验上:**一般只调其中一个,别两个一起调**(很多模型同时传两个会报错或行为不可预期)。要稳定就压低 `temperature`,要发散就调高。

### ⚠️ 重要变化:部分新模型已经移除了采样参数

这是一个**正在发生的、容易让你踩坑的变化**,务必记住:

> **部分新一代模型(例如 Claude 4.6 及以上,包括本书默认用的 Claude Opus 4.8)已经移除了 `temperature` / `top_p` / `top_k` 这些采样参数。** 如果你给这些模型传采样参数,**请求会直接报错(400)**。这些模型改用"思考 / 努力程度(effort)"等机制来控制行为,并推荐用**提示(prompt)**来引导风格,而不是靠调温度。

为什么会这样?因为这些模型内置了强大的"思考"能力,行为控制的重心从"调采样随机性"转移到了"控制它思考多深、努力多大"(下一节讲)。

> **这正是为什么本书反复强调"以官方文档为准、别凭记忆编参数"**:模型 API 变化非常快,某个你以为天经地义的参数,在新模型上可能已经不存在了。写代码前,**确认你用的那个具体模型支持哪些参数**。本章涉及的 Claude 细节(模型 ID、思考、努力程度、采样参数移除等)均依据 Anthropic 官方写作约定核对;拿不准的一律查官方文档,**不要编造不存在的参数或确切数字**。

实践建议:

- 用**老模型**(或 OpenAI 的 `gpt-4o` 等仍支持采样的模型):需要稳定就 `temperature: 0`(注意:即便温度为 0 也不保证两次输出字字相同)。
- 用**新模型**(如 Claude 4.6+):**别传采样参数**,改用 effort 控制深度 + 用提示词约束风格(下一节)。

## 2.5 思考 / 推理模式简介:让模型"想清楚再答"

第 1 章说过,让模型"想出声"能提高正确率。现代模型把这个能力做成了可控的**思考(thinking)模式**:模型在给出最终答案前,先在内部进行一段推理,推理越充分,复杂任务上越准。

以 Claude(4.6+,含本书默认的 Opus 4.8)为例,有两个相关旋钮:

### 1. 自适应思考(adaptive thinking)

开启后,模型**自己决定**要不要思考、思考多少——简单问题少想甚至不想,难题多想。开法是传 `thinking: {type: "adaptive"}`。

> ⚠️ **注意一个版本差异(典型的"别凭记忆编参数"案例)**:老模型用的是 `thinking: {type: "enabled", budget_tokens: N}`(手动给一个固定的思考 token 预算)。但在 Claude 4.6+ 这种新模型上,`budget_tokens` 已被弃用甚至会报 400——**新模型用 `{type: "adaptive"}`,不要再传 `budget_tokens`**。控制思考"多深"改用下面的 effort 参数。这正是 2.4 节那条原则的又一个体现。

### 2. 努力程度(effort)

控制模型整体投入多少"力气"(思考深度 + 花的 token 量),取值通常是 `low` / `medium` / `high` / `xhigh` / `max`(具体支持哪些档以官方文档为准):

| effort | 何时用 | 代价 |
|---|---|---|
| `low` | 简单、对延迟敏感、不太需要"动脑"的任务;子任务 | 最快最省,但复杂任务可能想得不够 |
| `medium` | 多数日常任务,性价比平衡点 | 中等 |
| `high` | 大多数对智能敏感的任务的推荐下限 | 较贵但质量高 |
| `xhigh` | 编码、Agent 等高难度任务的推荐档 | 更贵 |
| `max` | 极难、且"对就行、贵无所谓"的任务 | 最贵,可能过度思考 |

放在 `output_config` 里:

```ts
output_config: { effort: "high" } // 注意是嵌在 output_config 里,不是顶层
```

**什么时候用思考模式?**

- **该用**:多步推理、数学、复杂代码、需要规划的 Agent 任务——思考能显著提升正确率。
- **没必要**:简单分类、翻译、信息抽取——开了纯属浪费 token 和时间。
- **代价**:思考会**消耗额外 token、增加延迟**(模型多想几秒甚至更久)。不是越多越好,要按任务难度配。

这里只是建立概念。思考模式在 Agent 里怎么用得恰到好处,是[核心能力篇](../02-核心能力篇/)的内容。

## 2.6 调用 API:封装一层统一的 `chat()` 抽象

理论讲够了,上代码。我们要做一件本书会反复用到的事:**封装一个统一的 `chat(messages, options)` 函数,屏蔽不同厂商的差异,通过环境变量切换后端(Claude / OpenAI)。**

为什么要这层薄抽象?第 1 章说过本书"框架无关、先讲原理"的原则。你的业务代码不该直接和某一家的 SDK 死死耦合——今天用 Claude,明天想换 OpenAI 或本地开源模型(经 Ollama/vLLM 跑 Qwen/Llama),只要换这层抽象的实现,上层一行不用动。**这是写可维护 Agent 的基本功。**

设计目标:

- 入参统一:`chat(messages, options)`,`messages` 就是 2.3 节那个标准消息数组。
- 通过环境变量 `LLM_BACKEND` 选 `claude` 还是 `openai`。
- 返回值统一:直接给回文本(以及可选的 usage 信息),上层不用关心各家响应结构的差异。
- 密钥一律走环境变量,**绝不硬编码**。

```ts
#### TypeScript
import Anthropic from "@anthropic-ai/sdk";
import OpenAI from "openai";

// 统一的入参类型
interface ChatOptions {
  system?: string;       // 系统提示
  maxTokens?: number;    // 生成上限(见 2.7 的取值建议)
  effort?: "low" | "medium" | "high" | "xhigh" | "max"; // Claude 思考努力程度
}
type Message = { role: "user" | "assistant"; content: string };

// ---- 后端实现:Claude ----
async function chatWithClaude(messages: Message[], opts: ChatOptions): Promise<string> {
  const client = new Anthropic(); // 读 process.env.ANTHROPIC_API_KEY
  const resp = await client.messages.create({
    model: "claude-opus-4-8",            // 本书默认推荐模型
    max_tokens: opts.maxTokens ?? 4096,  // Claude 必填 max_tokens
    system: opts.system,                 // system 是顶层独立字段,不放进 messages
    thinking: { type: "adaptive" },      // 4.6+ 用自适应思考,不要用 budget_tokens
    output_config: { effort: opts.effort ?? "medium" }, // effort 嵌在 output_config 里
    messages,                            // 注意:不传 temperature/top_p,4.6+ 已移除
  });
  // 响应 content 是一个块数组,可能含 thinking / text 等;取 text 块拼起来
  return resp.content
    .filter((b) => b.type === "text")
    .map((b) => (b as { text: string }).text)
    .join("");
}

// ---- 后端实现:OpenAI ----
async function chatWithOpenAI(messages: Message[], opts: ChatOptions): Promise<string> {
  const client = new OpenAI(); // 读 process.env.OPENAI_API_KEY
  // OpenAI 把 system 当成 messages 里的一条,放最前面
  const full = opts.system
    ? [{ role: "system" as const, content: opts.system }, ...messages]
    : messages;
  const resp = await client.chat.completions.create({
    model: "gpt-4o", // 以官方为准;OpenAI 这类模型仍支持采样参数
    messages: full,
    max_tokens: opts.maxTokens ?? 4096,
    // temperature: 0, // OpenAI 模型可调采样;需要稳定时打开
  });
  return resp.choices[0].message.content ?? "";
}

// ---- 统一入口:用环境变量切后端 ----
export async function chat(messages: Message[], opts: ChatOptions = {}): Promise<string> {
  const backend = process.env.LLM_BACKEND ?? "claude"; // claude | openai
  switch (backend) {
    case "claude": return chatWithClaude(messages, opts);
    case "openai": return chatWithOpenAI(messages, opts);
    default: throw new Error(`未知后端: ${backend}`);
  }
}

// ---- 用起来:上层代码完全不关心底下是哪家 ----
const answer = await chat(
  [{ role: "user", content: "用一句话解释什么是 AI Agent" }],
  { system: "你是一个简洁的技术讲解助手。", effort: "low" },
);
console.log(answer);

import os
import anthropic
from openai import OpenAI

# ---- 后端实现:Claude ----
def chat_with_claude(messages: list[dict], *, system: str | None = None,
                     max_tokens: int = 4096, effort: str = "medium") -> str:
    client = anthropic.Anthropic()  # 读 os.environ["ANTHROPIC_API_KEY"]
    resp = client.messages.create(
        model="claude-opus-4-8",              # 本书默认推荐模型
        max_tokens=max_tokens,                # Claude 必填
        system=system,                        # system 是顶层独立字段
        thinking={"type": "adaptive"},        # 4.6+ 用自适应思考,别用 budget_tokens
        output_config={"effort": effort},     # effort 嵌在 output_config 里
        messages=messages,                    # 不传 temperature/top_p,4.6+ 已移除
    )
    # content 是块数组,取 text 块拼起来
    return "".join(b.text for b in resp.content if b.type == "text")

# ---- 后端实现:OpenAI ----
def chat_with_openai(messages: list[dict], *, system: str | None = None,
                     max_tokens: int = 4096, **_) -> str:
    client = OpenAI()  # 读 os.environ["OPENAI_API_KEY"]
    full = ([{"role": "system", "content": system}] + messages) if system else messages
    resp = client.chat.completions.create(
        model="gpt-4o",          # 以官方为准;OpenAI 这类模型仍支持采样参数
        messages=full,
        max_tokens=max_tokens,
        # temperature=0,         # 需要稳定时打开
    )
    return resp.choices[0].message.content or ""

# ---- 统一入口:环境变量切后端 ----
def chat(messages: list[dict], *, system: str | None = None,
         max_tokens: int = 4096, effort: str = "medium") -> str:
    backend = os.environ.get("LLM_BACKEND", "claude")  # claude | openai
    if backend == "claude":
        return chat_with_claude(messages, system=system, max_tokens=max_tokens, effort=effort)
    if backend == "openai":
        return chat_with_openai(messages, system=system, max_tokens=max_tokens)
    raise ValueError(f"未知后端: {backend}")

# ---- 用起来 ----
answer = chat(
    [{"role": "user", "content": "用一句话解释什么是 AI Agent"}],
    system="你是一个简洁的技术讲解助手。",
    effort="low",
)
print(answer)
```

代码里有几个值得你注意的"差异点",正是这层抽象帮你抹平的:

- **`system` 放哪**:Claude 把 `system` 作为**顶层独立字段**;OpenAI 把它当成 `messages` 里的第一条。抽象帮你统一成一个 `system` 参数。
- **`max_tokens`**:Claude 必填(限制输出上限);默认给个合理值。
- **采样参数**:Claude 4.6+ **不传** `temperature`/`top_p`(会 400);OpenAI 的 `gpt-4o` 仍可传。抽象里默认不传,需要时按后端区分。
- **思考 / effort**:Claude 用 `thinking: {type: "adaptive"}` + `output_config.effort`;OpenAI 那条线这里没用到。
- **响应取文本**:Claude 返回的是**内容块数组**(可能含 thinking、text 等),要 filter 出 `text` 块;OpenAI 直接给 `choices[0].message.content`。抽象统一返回纯文本字符串。

> 想接**本地开源模型**(Qwen/Llama)也很简单:Ollama / vLLM 提供 OpenAI 兼容端点,把 OpenAI 客户端的 `baseURL` 指到 `http://localhost:11434/v1`(Ollama)即可复用 `chatWithOpenAI` 那条分支。这进一步说明这层抽象的价值。

## 2.7 流式输出、成本意识与前端视角

### 流式输出:先点到为止

你用过 ChatGPT 那种"文字一个字一个字蹦出来"的效果吧?那就是**流式输出(streaming)**——模型不等整段答案生成完才返回,而是边生成边把 token 推给你。

对前端工程师来说这太熟悉了:本质就是 **SSE(Server-Sent Events)/ 流式响应**,和你处理 `EventSource`、`ReadableStream` 是一回事。它能极大改善体验——用户不用干等几十秒,立刻看到进度;对跑得慢的 Agent 尤其重要(还记得第 1 章说 Agent 延迟高吗?流式是主要的体验解药)。

这里先建立概念。具体怎么在前端把流接起来、怎么渲染、怎么处理中断,是[工程篇·流式输出与前端集成](../03-工程篇/12-流式输出与前端集成.md)的重头戏。

> 小贴士:输出可能很长时(大 `max_tokens`),**建议用流式**——否则非流式请求容易因耗时太久撞上 HTTP 超时。`max_tokens` 怎么定也有讲究:非流式默认给个 `~4096`(本章示例用的值,稳妥)到 `16000` 量级;明确要长输出就走流式并调大。别给太小,否则答案会被截断在半句话。

### 成本意识:输入输出分开计费,缓存能省钱

把 2.1 和 2.3 的两条事实合起来,你就懂了 Agent 成本的来源:

1. **输入、输出 token 分开计费,输出更贵**(2.1)。
2. **API 无状态,每轮重发完整历史**(2.3)→ 对话越长,输入 token 越多。

两条叠加:一个跑十几轮的 Agent,光是反复重发的历史,输入 token 就能堆出一大笔钱。

**好消息:提示缓存(prompt caching)能大幅省钱。** 如果你每轮请求的**前缀**(比如那段又长又固定的系统提示、工具定义、或前面不变的历史)是一样的,厂商可以把这部分缓存起来,后续请求命中缓存时,这部分按极低的价格计费(可省高达 ~90%)。Claude 里通过 `cache_control: {type: "ephemeral"}` 标记可缓存的前缀。

缓存有个关键脾气要记住:**它是前缀匹配——前缀里任何一个字节变了,后面全部失效。** 所以别在固定前缀里塞时间戳、随机 ID 这种每次都变的东西,否则永远命中不了缓存。具体怎么设计才省钱,是[工程篇·成本与缓存优化(第 15 章)](../03-工程篇/)的内容,这里先建立"缓存能省钱、要把稳定内容放前面"的意识。

### 前端视角:无状态 + 自己拼历史 ≈ 你在前端管理 state 再发请求

把本章串起来,落到你最熟悉的前端心智上:

> **调 LLM 这件事,和你在前端"维护 state、把 state 拼进 payload、发给无状态后端、再用响应更新 state"是同一套模式。**

- `messages` 数组 = 你维护的 `state`;
- 每轮把完整 `messages` 发过去 = 把 state 拼进 request body;
- LLM 服务器无状态处理 = 你的后端无状态地处理请求;
- 把回答 `push` 回数组 = 用响应更新 state;
- token 计费 / 上下文上限 = 你对 payload 大小、请求成本的那份在意;
- 流式输出 = 你早就会的 SSE / 流式响应。

你不是在学一套全新的东西。你是在把"管理前端状态、调无状态接口"这套你已经精通的肌肉记忆,搬到 LLM 上。**带着这个类比,后面的一切都会顺很多。**

## 2.8 常见坑 / 最佳实践

- **坑:以为模型记得上一轮。** API 无状态,你不发历史它就失忆。多轮对话出现"它怎么忘了刚说的",十有八九是你漏发了历史。
- **坑:用 `tiktoken` 估 Claude 的 token。** 那是 OpenAI 的分词器,算 Claude 会大幅偏差。**各家用各家的计数接口。**
- **坑:给新模型传采样参数。** Claude 4.6+ 传 `temperature`/`top_p`/`top_k` 会 400。新模型用 effort + 提示词控制行为。
- **坑:给新模型传 `budget_tokens`。** 4.6+ 已弃用,用 `thinking: {type: "adaptive"}` + `effort`。
- **坑:`max_tokens` 给太小。** 答案会被截断在半句。非流式给到 `~4096~16000` 量级,长输出走流式并调大。
- **坑:固定前缀里塞时间戳/随机 ID。** 会让提示缓存永远失效,白白浪费省钱机会。把稳定内容放前面、易变内容放后面。
- **坑:凭记忆编 API 参数和模型名。** 模型接口变化飞快。写之前确认你用的**具体模型**支持什么,拿不准就查官方文档,绝不编造确切数字或不存在的参数。
- **最佳实践:封一层 `chat()` 抽象。** 屏蔽厂商差异,换模型只改一处。这是可维护 Agent 的基本功。
- **最佳实践:对话历史是你的责任。** 主动管理裁剪/摘要/持久化,别等撑爆窗口才处理,更别悄悄截断。
- **最佳实践:按任务难度配思考/effort。** 简单任务用 `low` 甚至不开思考,难任务才上 `high`/`xhigh`。别一刀切开到最大,纯烧钱。

## 2.9 本章小结

1. **token 是模型处理文本的最小单位**(词块,非字非词)。中文通常比英文更费 token、更贵。按 token 计费,**输入输出分开计价、输出更贵**。要精确数 token 用**各家自己的接口**,别拿 `tiktoken` 估 Claude。
2. **上下文窗口是一次请求能带的 token 上限**(主流 200K~1M),装着系统提示+历史+输入+输出。前端类比:一次请求的 payload 上限。超出要主动裁剪/摘要,别悄悄截断。
3. **消息由 system/user/assistant 三种角色组成。最关键的一条:LLM API 是无状态的,每轮必须重发完整历史。** 这和前端"维护 state、拼 payload、发给无状态后端、再更新 state"是同一套模式——Agent 的"记忆"本质就是你在客户端管 state。
4. **采样参数(temperature/top_p)控制确定性 vs 发散性,一般只调一个。但部分新模型(Claude 4.6+)已移除采样参数**,改用 effort + 提示词控制——这是"以官方文档为准、别凭记忆编参数"的典型案例。
5. **思考模式让模型想清楚再答**:Claude 4.6+ 用 `thinking: {type: "adaptive"}`(别用旧的 `budget_tokens`)+ `output_config.effort`(`low`~`max`)。难任务用,简单任务别开,思考费 token 费时间。
6. **封一层统一 `chat(messages, options)` 抽象**,用环境变量切 Claude / OpenAI(及本地开源),屏蔽 `system` 放置、采样参数、响应结构等差异。
7. **成本意识**:输入输出分开计费 + 无状态重发历史 = Agent 易烧钱;**提示缓存(前缀匹配)能省钱**,把稳定内容放前面。流式输出改善体验、防超时。
